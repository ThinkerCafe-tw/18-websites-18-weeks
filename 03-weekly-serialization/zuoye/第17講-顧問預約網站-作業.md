# 第十七講　顧問預約網站　作業

> 格式參照前學年函授作業（逐講、選擇＋是非＋簡述＋實作；題目在前、解答在後）。內容對應 115 上《網頁設計》第十七講。

## 壹、作業題目

### 一、選擇題（單選）

1. （　）一位老闆讀完獨立顧問的網站後遲遲不預約，本講指出他內心真正卡住的第一個問題，最接近下列何者？
   (1) 這位顧問的收費是不是市場上最便宜的
   (2) 萬一合作失敗了，責任能找誰承擔
   (3) 網站的配色與動畫夠不夠精緻
   (4) 顧問有沒有屬於自己的辦公大樓

2. （　）依本講的訂單狀態流程，客人剛送出預約、尚未完成付款時，訂單最合理的狀態應是下列何者？
   (1) 已完成
   (2) 已排程
   (3) 待付款
   (4) 已退款

3. （　）關於金流串接，本講強調「真正用來更新訂單狀態」的依據是下列何者？
   (1) 使用者付款後看到的成功頁畫面
   (2) 前端 JavaScript 計算出的訂單金額
   (3) 後端收到並驗證簽章與金額的 Webhook 通知
   (4) 客人自己在表單上勾選的「我已付款」

4. （　）下列哪一項做法，最符合本講「教學原型」這一層能做、且不會被誤解的範圍？
   (1) 用前端狀態模擬待付款、成功、失敗、取消，並標明尚未真正收款
   (2) 直接收真實信用卡並開立發票
   (3) 把商店代號與 API 金鑰寫進前端方便測試
   (4) 共用老師或學校的金流帳號正式營運

### 二、是非題

5. （　）為了讓網站看起來更專業，就算顧問沒有具名推薦人，也可以先放幾則匿名好評，反正客人也分不出真假，效果其實差不多。

6. （　）同一筆 Webhook 通知可能因網路重送而被系統收到兩次，所以系統應依交易編號、訂單編號與處理紀錄判斷「這件事是否已做過」，避免重複扣款或重複交付。

### 三、簡述題

7. 本講把顧問金流閉環拆成方案、預約、訂單、付款、交付五層，其中付款層再拆成「交易」與「回呼（Callback）」兩張表。請說明為什麼回呼要獨立成一張表，而不是只記在交易表裡。

8. 本講提到「信任三支柱」——資歷（Ability）、善意（Benevolence）、誠信（Integrity）。請各用一句話說明三者的意思，並說明為什麼只把資歷寫得很滿、卻缺另外兩根，信任仍然不會出現。

### 四、實作題

9. 請用本講的提示詞讓 AI 工具產出一個顧問預約與付款網站 `index.html`，存檔後用手機寬度打開。請逐項檢查下列四件事，並記錄至少兩個需要修改的地方與你打算用的修改句：
   (一) 服務方案是否寫清楚價格、時長、交付物與「不包含項目」；
   (二) 是否同時有付款成功、付款失敗、使用者取消三種結果；
   (三) 頁面是否標明這是金流流程原型、尚未真正串接，且沒有把 API 金鑰寫進前端；
   (四) 手機版的付款流程是否清楚、客人不會不知道下一步。

## 貳、作業解答

### 一、選擇題

1. **（2）**
   解析：本講破冰案例的核心是老闆陳志鴻點進顧問頁面後，第一個念頭不是「這個人很厲害」，而是「萬一他搞砸了，我找誰負責？」。獨立顧問沒有大型管顧那種品牌、資產與可被起訴的法人格，老闆要承擔「我相信了一個人」的脆弱。所以顧問網站要回答的第一個問題是「責任歸屬」，不是價格(1)、不是視覺(3)，更不是有沒有辦公大樓(4)。

2. **（3）**
   解析：本講訂單狀態的流轉是「草稿 → 待付款 → 已付款 → 已排程 → 已完成」，並有「付款失敗」「已取消／已退款」兩條分支。客人送出預約、還沒完成付款，正好停在「待付款」這一格。(1)(2) 是付款完成後才會推進到的狀態；(4) 是付款後又取消才會發生的分支，順序都不對。

3. **（3）**
   解析：成功頁只是給使用者看的「體驗提示」，不是收款憑證——使用者甚至可能根本沒付成功就被導回。真正能拿來更新訂單狀態的，是後端收到金流商通知後、驗證過簽章與金額的 Webhook 回呼。(1) 把畫面當憑證很危險；(2) 前端金額可被任意竄改，不能作數；(4) 客人自己勾選更不可信。

4. **（1）**
   解析：本講把金流實操分三層——教學原型、測試環境 sandbox、正式環境 production。教學原型能做的是「用前端狀態模擬待付款、成功、失敗、取消與交付」，但不能說已完成真實收款，所以(1) 正確且有標明。(2) 收真實卡、開發票是正式環境的事；(3) 把代號與金鑰寫進前端是明確禁忌；(4) 共用老師或學校帳號正式營運，責任歸屬會出問題。

### 二、是非題

5. **（×）**
   解析：本講指出，推薦人的力量來自「可追溯」——具名、有公司、在 LinkedIn 找得到的同行背書，重量超過網站任何文字；匿名好評因為無法追溯，幾乎等於沒有。放假的或無從查證的好評不只沒幫助，一旦被識破還會連真實內容一起變得不可信，這正是侵蝕信任的手法，不能做。

6. **（○）**
   解析：這正是本講提到的「冪等性」（idempotency）概念。金流商的 Webhook 可能因網路重送被收到不只一次，若系統不做判斷就會重複扣款或重複交付。正確做法是靠交易編號、訂單編號與處理紀錄判斷「這件事是否已做過」，做過就略過。這也是 PaymentCallback 要獨立成一張表、記錄原始回傳與處理時間的原因之一。

### 三、簡述題

7. **答**：回呼（PaymentCallback）要獨立成一張表，主要有三個原因——
   (一) **更新狀態的真正依據在這裡**：金流商除了把使用者導回成功頁，還會用後端通知（Webhook／Callback）送回付款結果，真正用來把訂單從「待付款」推到「已付款」的，是這筆驗證過的後端通知，而不是成功頁。它的地位重要到值得單獨保存。
   (二) **要保留可稽核的原始證據**：回呼表記錄「原始回傳資料、驗證結果、處理時間」，當帳務或爭議發生時，可以回頭比對金流商當初到底送了什麼、系統驗過沒有，交易表只記交易結果並不足夠。
   (三) **支援冪等性判斷**：同一筆通知可能被重送，獨立的回呼紀錄能讓系統判斷「這筆是否已處理過」，避免重複扣款或重複交付。把「一筆交易」與「可能多次的回呼」分開，資料結構才乾淨、責任才清楚。

8. **答**：信任三支柱出自 Mayer、Davis 與 Schoorman（1995）的信任模型——
   - **資歷（Ability）**：對方有沒有能力把事情做好，例如方法論、案例、可驗證的成果。
   - **善意（Benevolence）**：對方是不是真的為你著想、和你站在同一邊，例如具名推薦人願意為他背書、成效分潤把顧問收入綁進你的成敗。
   - **誠信（Integrity）**：對方說的和做的是否一致、有沒有可追蹤的原則，例如清楚的收費方式、退款規則與「什麼情況不接」的邊界。
   為什麼只有資歷不夠：本講強調「三根必須同時得分，信任才出現」。只把資歷寫得很滿、卻沒有具名推薦人（缺善意）、也沒有可追蹤的方法論與規則（缺誠信），老闆會懷疑「這個人是不是只在推銷自己」，信任就立不起來。信任不是一道宣告，是三根支柱同時站起來的結果。

### 四、實作題

9. **答（參考評分重點）**：本題為實作，無單一標準答案，評分看三件事——
   (1) **是否完成產出與檢查流程**：用本講提示詞產出 `index.html`、存檔，並以手機寬度逐項檢查題目列出的四件事（方案資訊／三種付款結果／原型告示與金鑰位置／手機版付款流程）。
   (2) **記錄的問題是否具體**：例如「方案卡只寫了交付物，漏掉『不包含項目』，客人會誤以為什麼都做」「只有付款成功頁，沒有失敗與取消的畫面」「頁面沒有任何『尚未串接、僅為模擬』的告示」屬具體；「感覺怪怪的」「不夠專業」過於籠統。其中「有沒有把 API 金鑰或商店代號誤寫進前端」是本講特別要學生養成的檢查習慣，學生若主動查到並指出，應給予加分。
   (3) **修改句是否可操作**：例如「請在每張方案卡都補上『不包含項目』欄位，並在訂單摘要付款前再次顯示退款規則」「請補上付款失敗與使用者取消兩種狀態卡，讓三種結果都看得到」「請在頁面最上方加一條原型告示，標明本頁尚未串接藍新或綠界、所有付款皆為畫面模擬，並把任何金鑰移出前端」。並確認修改後其他區塊沒有被弄壞、手機版仍是單欄好點。
