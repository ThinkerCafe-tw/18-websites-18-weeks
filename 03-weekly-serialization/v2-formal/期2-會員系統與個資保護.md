## 壹、課業輔導

### 一、本期學習定位

本期涵蓋教材第四至第六講，主題為網站功能從「資訊展示」邁向「服務互動」的關鍵階段。此三講共通的技術核心為「使用者資料的蒐集、儲存與再利用」，共通的法律核心則為 2023 年修正施行之《個人資料保護法》所確立之安全維護義務。本期課業輔導依三個層次展開：預約流程的使用者經驗設計、會員註冊的最小必要蒐集原則、以及個資法修正對個人網站營運者的實質影響，並於文末提供一則美容院會員系統的實例分析。

### 二、重點整理與補充

#### （一）預約系統的使用者經驗三層結構

家醫診所、美容院、法律事務所此類時段型服務業，其線上預約系統需同時處理三個層次。第一層為「空檔可見性」，使用者開啟頁面後，應能於三秒內辨識當日或當週之可預約時段，避免以下拉選單逐一探索；第二層為「填答負擔最小化」，預約所需欄位以「姓名、手機、欲選時段」為原則，附註、特殊需求或備忘事項應一律設為選填；第三層為「確認回饋」，送出預約後系統應即時顯示具識別性之預約編號，並透過電子郵件或簡訊再次回傳，以降低使用者重複送單之疑慮。教材第四講之範例已涵蓋第一、第二層，第三層所涉之「非同步通知」設計屬教材未展開之補充範圍。實務建議：初學者可採用 EmailJS、Formspree 等第三方表單服務先行建置，待流量成長再遷移至自架 API 服務。

#### （二）會員註冊的最小必要蒐集原則

會員註冊為網站蒐集個人資料之關鍵節點。依《個人資料保護法》第 5 條所揭示之原則——「蒐集、處理及利用個人資料，應尊重當事人之權益，依誠實及信用方法為之，不得逾越特定目的之必要範圍」——註冊時應僅蒐集達成特定服務所必要之資料。以教材第五講之美容院會員系統為例，核心欄位為「姓名、手機、生日月份」三項：手機作為身分驗證媒介，姓名作為識別之用，生日月份用於當月優惠通知，不需蒐集完整西元年。身分證字號、戶籍地址、職業類別等欄位，除非與服務履行直接相關（例如須開立統一發票或寄送實體商品），否則不應列入註冊流程。建議學員於自行擴充欄位時，先就每一欄位回答「若此資料遺失，對使用者所生之風險，是否能被蒐集之必要性正當化」，以此作為欄位取捨之依據。

#### （三）2023 年《個人資料保護法》修正重點

《個人資料保護法》於 2023 年 5 月 16 日經立法院三讀通過修正，同月 31 日總統公布施行，其中對個人網站營運者具實質影響者為第 48 條罰則之大幅調升。修正前，非公務機關違反安全維護措施之罰鍰上限為新臺幣 50 萬元；修正後，若違反同法第 27 條第 1 項所定安全維護義務且情節重大者，罰鍰上限提高至新臺幣 1,500 萬元。此項修正不以「個資實際外洩」為要件，主管機關認定業者未採取適當之安全維護措施即得裁罰。學員於建置會員網站時，應至少落實三項基礎措施：第一，密碼以單向雜湊演算法（如 bcrypt）儲存，禁用明文或可逆加密；第二，全站強制 HTTPS 導向，禁用混合內容；第三，資料庫查詢採參數化語法，以防止 SQL Injection 攻擊。現行法另要求業者於個資外洩時應於合理時間內通知當事人，違反者亦屬情節重大之裁量事由。

### 三、實例分析：美容院會員系統數位化之四階段

觀察社區型美容院之會員管理，其數位化歷程可概分為四個階段。第一階段為「純紙本」，老闆娘以筆記本記錄熟客姓名與偏好，無個資外洩風險但亦無擴張可能。第二階段為「通訊軟體輔助」，店家透過 LINE 或 WhatsApp 與熟客直接對話，客戶資料散落於各聊天室；此階段便於回訪，但店家更換手機或員工離職後即有資料遺失與不當再利用之風險，且難以回應當事人之查閱、更正、刪除請求。第三階段為「Google Sheet 中介」，由店家建立表單供客戶自填，資料匯入試算表；此方式雖解決集中管理問題，但 Sheet 分享權限若誤設為「任何人可檢視」即違反安全維護義務。第四階段為「輕量會員系統」，即教材第五、第六講所建置者：以 Supabase 為後端，配合行動認證與欄位級存取權限，方能兼顧便利性與法遵要求。學員可依自身服務規模選擇適當階段，但無論採用何種形式，均應符合個資法最小必要蒐集與安全維護之原則。

## 貳、習題與解答

### 第一題【選擇題】

依《個人資料保護法》第 5 條所揭示之原則，下列敘述何者**正確**？

(A) 只要取得當事人同意，蒐集之個資範圍即無限制
(B) 個人網站之經營者非公務機關，不適用本法
(C) 個人資料之蒐集、處理及利用，不得逾越特定目的之必要範圍
(D) 僅蒐集姓名與電話即不適用本法

**答：(C)**

**解析**：本法第 5 條明定「蒐集、處理及利用個人資料，應尊重當事人之權益，依誠實及信用方法為之，不得逾越特定目的之必要範圍，並應與蒐集之目的具有正當合理之關聯」。(A) 縱取得同意，仍受特定目的必要範圍之限制，過度蒐集仍違法；(B) 非公務機關（含自然人、法人、個人工作室）原則上均受本法規範；(D) 姓名與電話屬本法第 2 條第 1 款定義之個人資料，適用本法。

### 第二題【選擇題】

關於 2023 年修正後之《個人資料保護法》第 48 條罰則，下列敘述何者**正確**？

(A) 罰鍰上限自新臺幣 50 萬元提高至 1,500 萬元
(B) 須個資實際外洩始得裁罰
(C) 罰則僅適用於資本額逾新臺幣一億元之企業
(D) 修正後罰鍰上限為新臺幣 500 萬元

**答：(A)**

**解析**：2023 年修正後第 48 條規定，非公務機關違反同法第 27 條第 1 項所定之安全維護義務，情節重大者，罰鍰上限自新臺幣 50 萬元提高至新臺幣 1,500 萬元。(B) 裁罰要件為「未採取適當之安全維護措施」，不以實際外洩為必要；(C) 條文未設資本額門檻，個人工作室亦適用；(D) 上限為 1,500 萬元，非 500 萬元。

### 第三題【選擇題】

下列何者**不屬於**預約系統使用者經驗之「第一層：空檔可見性」之良好設計？

(A) 以月曆視圖標示可預約與已額滿時段
(B) 以紅綠色塊區分時段狀態
(C) 使用者須逐層點開下拉選單始能看到可用時段
(D) 頁面開啟後三秒內能辨識當週是否有空檔

**答：(C)**

**解析**：「空檔可見性」之核心在於讓使用者於最短時間內掌握可預約狀態。(A)(B)(D) 皆為可見性良好之設計。(C) 以逐層點開之方式顯示時段，增加使用者認知負擔，屬第一層設計之反例。

### 第四題【是非題】

美容院會員系統註冊時，為利於日後行銷分眾，蒐集客戶之完整西元年生日、職業類別與家庭成員人數均為合理之蒐集。（　）

**答：（×）**

**解析**：依最小必要蒐集原則，應僅蒐集達成特定服務所必要之資料。當月優惠通知僅需「生日月份」，完整西元年與職業類別、家庭成員人數均與美容院核心服務（髮型、美容、會員回訪）無直接關聯，屬過度蒐集。若業者有進行分眾行銷之需求，應另行設計選填之「偏好調查」欄位，並於蒐集時告知用途與當事人權利。

### 第五題【是非題】

店家使用 Google Sheet 儲存會員資料，只要將分享連結設定為「任何人可檢視」而未公開張貼，即不構成違反《個人資料保護法》第 27 條安全維護義務。（　）

**答：（×）**

**解析**：「任何人可檢視」之連結屬於可公開存取之狀態，任何取得連結者（包括爬蟲、分享轉發）皆可閱覽全部個資。主管機關判斷安全維護措施是否適當，並不以「是否實際外洩」或「連結是否被主動張貼」為要件，而以「蒐集者是否採取合理且適當之防護措施」為斷。Sheet 之權限應設為「具權限者始可檢視」，並對特定帳號授權。

### 第六題【應用題】

試寫出以 Python 搭配 SQLite 建立會員資料表時，如何以參數化查詢防止 SQL Injection 攻擊。請以插入一筆會員資料（姓名、電話、雜湊後密碼）之語法說明。

**答**：

```python
import sqlite3
import bcrypt

conn = sqlite3.connect('members.db')
cursor = conn.cursor()

name = user_input_name
phone = user_input_phone
password_hash = bcrypt.hashpw(user_input_password.encode(), bcrypt.gensalt())

cursor.execute(
    "INSERT INTO members (name, phone, password_hash) VALUES (?, ?, ?)",
    (name, phone, password_hash)
)
conn.commit()
```

**解析**：關鍵為 `execute()` 之第二個引數以 tuple 傳入使用者輸入，並於 SQL 字串中以 `?` 作為佔位符。切勿以字串串接方式組合 SQL 語句（如 `f"INSERT INTO members VALUES ('{name}')"`），此作法將使攻擊者得以輸入 `'; DROP TABLE members; --` 等惡意字串執行資料庫指令。密碼部分採用 bcrypt 單向雜湊儲存，違反此原則即違反本法第 27 條之安全維護義務。

### 第七題【應用題】

會員網站儲存使用者密碼時，下列四種方式，何者符合《個人資料保護法》第 27 條之安全維護義務？請排序並說明理由。

(甲) 明文儲存
(乙) MD5 雜湊
(丙) bcrypt 單向雜湊
(丁) AES-256 對稱加密

**答**：符合程度由高至低為 丙 > 丁 > 乙 > 甲。

**解析**：
- **（丙）bcrypt**：為設計供密碼儲存之單向雜湊演算法，具計算成本可調與自動加鹽特性，即使資料庫外洩，攻擊者亦難以在合理時間內還原明文。此為目前業界建議作法。
- **（丁）AES-256 對稱加密**：雖為強加密演算法，但屬可逆式加密，只要取得金鑰即可還原明文；若金鑰與資料庫同機儲存，實質安全等同於明文。僅於特定場景（如需解密比對外部系統）始適用，且須搭配金鑰管理服務（KMS）使用。
- **（乙）MD5 雜湊**：雖為單向雜湊，但 MD5 已被證實可於合理時間內碰撞，且網路上已有大量彩虹表，可直接反查常見密碼。不符現行安全標準。
- **（甲）明文儲存**：任何資料庫外洩事件即等同所有使用者密碼外洩，且同一密碼常於多站重複使用，將擴大損害範圍。明顯違反安全維護義務。

### 第八題【情境題】

某美容院會員王小姐以電子郵件請求網站經營者提供其所有個人資料之副本，並要求刪除其歷史消費紀錄。經營者應如何回應？請依《個人資料保護法》說明處理流程。

**答**：

依《個人資料保護法》第 3 條，當事人就其個人資料享有查詢、請求閱覽、請求製給複製本、請求補充或更正、請求停止蒐集處理或利用、請求刪除等權利，且此等權利不得預先拋棄或以特約限制之。經營者之處理流程應為：

1. **身分核對**：先確認請求者確實為王小姐本人，可透過回覆註冊電子郵件、簡訊驗證或電話確認完成。
2. **回應時限**：依第 13 條規定，非公務機關應於接獲請求之 15 日內為准駁之決定；必要時得延長 15 日，並書面通知請求人。
3. **提供副本**：將王小姐之個人資料（含姓名、電話、消費紀錄、偏好紀錄）以可讀取之格式整理並提供。
4. **刪除評估**：消費紀錄若已作為會計、稅務或其他法定應保存資料（例如發票存根），依第 11 條第 3 項規定得不予刪除，但應停止再利用；若無法定保存義務，則應於合理時間內完成刪除並回報當事人。
5. **紀錄存查**：處理過程應製作紀錄，保存至少 5 年，作為日後主管機關稽核之依據。

### 第九題【情境題】

某學員架設之健身房會員網站，於上線三個月後發現資料庫曾遭異常登入，雖尚無證據顯示個資外洩，但登入來源 IP 非預期之地區。學員應如何處理？

**答**：

1. **立即止血**：立即強制所有會員重設密碼，並檢視近期資料庫操作紀錄，確認是否有異常查詢或匯出行為。
2. **影響評估**：於 24 至 72 小時內完成技術鑑識，評估資料是否實際被讀取、複製或修改。
3. **通知當事人**：依《個人資料保護法》第 12 條規定，非公務機關違反本法規定，致個人資料被竊取、洩漏、竄改或其他侵害者，應查明後以適當方式通知當事人。「適當方式」得為電子郵件、簡訊或掛號信件，內容應包含事件經過、已採取之因應措施、當事人可採取之保護行動（如更換密碼）與聯絡窗口。
4. **主管機關通報**：若情節重大，依各目的事業主管機關之要求辦理通報（如金融業通報金管會、醫療業通報衛福部等）。健身產業現行未有主管機關強制通報規定，但仍建議自主通報以展現善意。
5. **強化措施**：依事件原因強化安全措施，例如加入登入地區限制、二階段驗證（2FA）、登入失敗次數鎖定機制等，並將強化措施書面記錄存查。

### 第十題【問題與討論】

試就自身熟悉之產業（如餐飲、零售、補習、諮商等），設計一組會員註冊表單之「最小必要欄位」，並說明每個欄位之蒐集目的、是否為必填、以及資料保存期限。欄位數量不限，但須能說明其必要性。

**答（參考架構）**：

本題無標準答案，評分重點在於：

1. **是否能說明每個欄位之特定目的**（如：手機用於身分驗證、生日月份用於當月優惠）；
2. **是否區分必填與選填**（必填欄位應僅限達成服務核心功能所必要者）；
3. **是否設定保存期限**（例如會員終止後 6 個月內刪除歷史消費紀錄，但發票存根依稅法保存 5 年）；
4. **是否對敏感欄位（身分證字號、詳細地址、健康資料）有特別說明**，若蒐集須具備明確正當理由。

參考範例（餐飲業熟客系統）：
- 姓名（必填，識別用途，終止會員後 6 個月刪除）
- 手機末四碼（必填，用於現場辨識，終止會員後 6 個月刪除）
- 生日月份（選填，當月優惠通知，終止會員後 6 個月刪除）
- 飲食禁忌（選填，餐點安全，終止會員後 6 個月刪除）
- 每次消費紀錄（自動記錄，會計憑證，依稅法保存 5 年）

學員於撰寫時應展現對「最小必要」與「目的拘束」兩原則之理解，並能於欄位設計上具體落實。
