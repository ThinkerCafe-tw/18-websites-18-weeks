## 壹、課業輔導

### 一、本期學習定位

本期涵蓋教材第七至第九講，三者共同處理「時間」此一資源的有限性。法律事務所處理的是「專業時間的稀缺性」如何透過網站呈現；餐廳處理的是「同一時段座位有限」的即時衝突；瑜伽教室處理的是「重複性課程」的循環排程。本期課業輔導補充三項重點：專業服務網站的信任指標設計、訂位系統的並發衝突處理機制、以及週期性課表的資料結構與呈現方式。

### 二、重點整理與補充

#### （一）專業服務網站的信任指標

法律、會計、醫療、諮商等專業服務業，訪客於決定是否預約前，需累積足夠的信任。教材第七講提出「數位名片」概念，本期補充四項具體指標。第一為「執業證書可驗證性」：律師網站應列出完整姓名、律師證號與公會所屬，使訪客得自行至司法院網站查驗；此為業界最基本之誠信指標。第二為「具名案例而非抽象描述」：寫「協助中小企業處理合約爭議」信任度偏低，若能具名「協助某科技公司完成 A 輪投資合約」則更具可信度，唯須取得當事人同意。第三為「具體回應時限」：標示「來信三日內必覆」優於「我們會盡快回覆」。第四為「免費諮詢之明確範圍」：須說明免費諮詢涵蓋何種問題、時長限制與後續收費方式，避免使用者誤解後產生爭議。此四項指標不需炫目設計，但對轉換率有直接影響。

#### （二）訂位系統的並發衝突處理

餐廳訂位為典型之並發場景：兩位使用者同時看到「週六七點尚有一桌」，並同時送出預約。若無衝突處理機制，將導致超訂。教材第八講建立基本訂位流程，本期補充三種衝突處理策略。第一為「悲觀鎖」：使用者進入結帳頁後，該時段立即標記為「處理中」，鎖定 3 至 5 分鐘。優點為資料一致性最強，缺點為若使用者放棄結帳，將造成時段浪費。第二為「樂觀鎖」：不預先鎖定，待送出時才驗證時段是否仍可用，若已被他人預約則回傳錯誤並請使用者重選。優點為資源利用率高，缺點為使用者體驗較差。第三為「候補機制」：被預約滿時不直接拒絕，而是加入候補名單；原訂位取消時自動通知候補者。建議之綜合策略為：高峰時段採「樂觀鎖 + 候補機制」，平日採「悲觀鎖」以降低開發複雜度。資料庫層面需於訂位表建立 `(restaurant_id, time_slot, status)` 組合唯一索引，以從底層阻擋重複寫入。

#### （三）週期性課表的資料結構

瑜伽、健身、才藝等機構之課程通常具週期性，例如「每週一、三、五晚上七點的哈達瑜伽」。若為每一週次新增一筆資料庫紀錄，一學期 16 週將產生大量冗餘資料且難以統一修改。教材第九講採用「課程樣板 + 例外項目」之資料模型，此為業界通用設計。課程樣板表儲存週期性資訊（哪門課、星期幾、幾點、老師、教室），例外項目表儲存單一偶發事件（某日停課、某日調時、某日換老師）。系統顯示課表時，依樣板產生預計課程列表，再套用例外項目覆蓋之。此設計之優點為資料精簡、統一修改容易；學期中若調整常態課表，僅需修改樣板一筆；若僅某週停課，僅需新增例外項目一筆。部分系統採用 iCalendar RFC 5545 標準之 RRULE 語法表達週期規則，可直接匯出至 Google Calendar 供學員訂閱。

### 三、實例分析：瑜伽教室週期性課表三種實作比較

以一間週一至週日每日四堂課的瑜伽教室為例，一學期 16 週課表共有 448 堂課。三種實作方式比較如下。**第一種為「完全展開」**，每堂課建立一筆資料庫紀錄，共 448 筆。優點為查詢簡單，直接依日期範圍撈取即可；缺點為新增學期或調整時段時須手動產生大量資料，且修改常態課表需逐筆更新。**第二種為「樣板 + 例外」**，建立 28 筆樣板（7 天 × 4 時段）與若干筆例外紀錄。顯示時由程式根據樣板推算各週實際課程，再套用例外覆蓋。優點為資料精簡、維護成本低；缺點為程式邏輯較複雜，需處理時區、節日等特殊情況。**第三種為「RRULE 規則」**，採 iCalendar 標準語法表達循環規則，如 `FREQ=WEEKLY;BYDAY=MO,WE,FR;INTERVAL=1`。優點為符合標準、可匯出至外部行事曆；缺點為需引入 iCalendar 解析函式庫，且臺灣常見之農曆節日處理較繁瑣。建議依規模選擇：單店採第二種方案即可兼顧簡潔與彈性；連鎖分店或跨時區業務則建議採第三種，以利跨系統整合。

## 貳、習題與解答

### 第一題【選擇題】

關於專業服務網站之「信任指標」，下列何者**最有效**提升轉換率？

(A) 使用大量動畫與影片強化視覺體驗
(B) 標示律師證號、具名案例與具體回應時限
(C) 採用深色背景搭配金色字體以顯示高級感
(D) 於首頁播放老闆自述影片

**答：(B)**

**解析**：專業服務之信任基礎為「可驗證性」與「具體性」。(B) 所列三項皆可被訪客驗證或追蹤，最能建立信任。(A)(C) 為表層視覺，對專業信任影響有限。(D) 自述影片若無佐證仍屬單方陳述，轉換率提升有限。

### 第二題【選擇題】

兩位使用者同時看到餐廳某時段僅剩一桌並送出預約，為防止超訂，資料庫層面**最根本**之機制為何？

(A) 前端 JavaScript 驗證
(B) 於訂位表建立 `(restaurant_id, time_slot, status)` 組合唯一索引
(C) 送出後以 setTimeout 延遲 500ms 再寫入
(D) 增加伺服器 CPU 效能

**答：(B)**

**解析**：前端驗證與延遲處理皆無法防止並發寫入，只有資料庫層面之唯一索引能從底層阻擋重複資料。當第二筆寫入嘗試違反唯一約束時，資料庫將回傳錯誤，應用程式再據以提示使用者。(A) 前端驗證可被繞過，(C) 延遲僅能降低機率而非根絕，(D) CPU 效能與並發正確性無關。

### 第三題【選擇題】

瑜伽教室一學期 16 週共 448 堂課之排課，採「樣板 + 例外」資料模型之**主要優點**為何？

(A) 查詢速度比完全展開快
(B) 程式邏輯較簡單
(C) 資料精簡且統一修改容易
(D) 可離線運作

**答：(C)**

**解析**：「樣板 + 例外」之核心優勢在於資料規模精簡（28 筆樣板 + 少數例外 vs 448 筆完全展開）與維護便利性（修改常態課表僅需修改樣板一筆）。(A) 查詢速度反而可能略慢（需額外計算）。(B) 程式邏輯較完全展開複雜。(D) 與資料模型無關。

### 第四題【是非題】

律師網站之案例描述只要寫「協助中小企業處理合約爭議」即可，不需具名，因為具名會涉及當事人隱私。（　）

**答：（×）**

**解析**：「是否具名」與「是否侵犯隱私」為不同層次之問題。正確作法為「取得當事人同意後具名」，若當事人不同意具名，則應以「某產業某類型之某情境」具體化描述，而非採用「中小企業」「合約爭議」此類抽象敘述。完全抽象之描述缺乏信任建立效果，且業界普遍做法為以匿名方式具體化（例如「某電商平台於 A 輪募資時之合約爭議」），兼顧隱私與信任。

### 第五題【是非題】

若採用「悲觀鎖」處理餐廳訂位，使用者進入結帳頁後時段立即被鎖定，為免資源浪費，鎖定時間應設為 30 分鐘以上。（　）

**答：（×）**

**解析**：悲觀鎖之鎖定時間應反映「正常使用者完成結帳所需之合理時間」，通常為 3 至 5 分鐘。若設為 30 分鐘，將使未完成結帳之使用者長時間占用時段，嚴重影響其他使用者體驗並降低整體訂位率。實務上業界標準為 5 分鐘，並於即將逾時前以彈出視窗提醒使用者。

### 第六題【應用題】

試以 SQL 撰寫一段建表語句，建立餐廳訂位表 `reservations`，並於 `(restaurant_id, time_slot)` 組合建立唯一索引以防止超訂。

**答**：

```sql
CREATE TABLE reservations (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  restaurant_id INTEGER NOT NULL,
  time_slot DATETIME NOT NULL,
  guest_count INTEGER NOT NULL,
  customer_name TEXT NOT NULL,
  customer_phone TEXT NOT NULL,
  status TEXT NOT NULL DEFAULT 'confirmed',
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  UNIQUE (restaurant_id, time_slot, status)
);

CREATE INDEX idx_reservations_time
  ON reservations (restaurant_id, time_slot);
```

**解析**：`UNIQUE (restaurant_id, time_slot, status)` 確保同一餐廳、同一時段、狀態為 `confirmed` 者僅能存在一筆，從底層阻擋超訂。將 `status` 納入唯一索引之考量在於：若使用者取消預約（status 改為 `cancelled`），該時段應可再被預約，故僅 `confirmed` 狀態受唯一性限制。另建 `idx_reservations_time` 為一般索引以加速查詢。實務應用時需考量：同一時段若餐廳有多桌，唯一索引需納入桌次；跨時段長用餐（如訂 19:00 但實際使用至 21:00）需額外處理時段重疊檢查。

### 第七題【應用題】

瑜伽教室採「樣板 + 例外」資料模型，某學期某週因颱風停課一天，應如何於資料庫操作？

**答**：

假設樣板表為 `class_templates`，例外表為 `class_exceptions`，停課操作如下：

```sql
-- 新增一筆例外紀錄，type 標記為 'cancelled'
INSERT INTO class_exceptions (
  template_id, exception_date, type, reason
)
SELECT id, '2026-09-15', 'cancelled', '颱風停課'
FROM class_templates
WHERE day_of_week = 'MON';  -- 颱風當天為週一，僅需停該日所有週一課程
```

**解析**：此操作之優點在於：第一，不修改樣板表，保留常態排程之完整性；第二，例外紀錄可附帶原因欄位（`reason`），日後可供學員查閱停課歷史；第三，恢復課程僅需刪除或標記此例外紀錄，無需重建樣板。若採「完全展開」模型，則需逐筆更新當日所有課程之 status，操作複雜且易漏。

### 第八題【情境題】

某律師事務所網站上線後，訪客數量穩定但預約諮詢之轉換率偏低（每月訪客 1,200 人，僅 3 人實際預約）。請診斷可能原因並提出三項改善建議。

**答**：

**可能原因診斷**：

1. **信任指標不足**：訪客無法判斷此律師是否真實存在、專業程度如何。若網站僅列姓名與聯絡方式，未附律師證號、執業年資、具名案例，訪客易將此網站視為「疑似詐騙」而放棄。
2. **諮詢門檻過高**：若網站要求訪客先填寫 20 項欄位始能送出諮詢，多數人會中途放棄。
3. **費用資訊不透明**：訪客最擔心之問題為「諮詢是否收費、費用為何」。若網站未明確說明，多數訪客會選擇直接撥打其他有明確費用揭示之事務所。

**三項改善建議**：

1. **增加可驗證之信任指標**：於首頁增加律師證號、執業年資、公會所屬，並附司法院查詢連結。若有代表性案例（取得當事人同意），具名列出 3 至 5 件。
2. **簡化預約流程**：首次諮詢表單僅收「姓名、手機、簡短描述」三項必填，進階資訊於確認諮詢後再補。同時明確標示「三個工作日內專人回覆」。
3. **明確費用揭示**：標示「首次 15 分鐘電話諮詢免費」或「現場諮詢每小時 NT$3,000」等具體費用，避免訪客因不確定而卻步。

### 第九題【情境題】

某餐廳採「樂觀鎖 + 候補機制」處理訂位，某週六晚 19:00 時段原訂位客人臨時取消。請說明系統應如何自動通知候補名單，以及通知順序之設計考量。

**答**：

**自動通知流程**：

1. 原訂位狀態由 `confirmed` 改為 `cancelled` 時，觸發資料庫 trigger 或應用層 event。
2. 查詢該時段之候補名單（依加入時間排序）。
3. 通知候補名單中**最前一位**使用者（非全部通知）。
4. 通知內容包含：時段、確認期限（例如 10 分鐘內回覆）、確認連結。
5. 若該使用者於期限內確認，則該時段狀態改為 `confirmed`，其餘候補維持不變。
6. 若該使用者於期限內未回覆，系統自動通知次一位候補者，並將未回覆者移至候補名單末端。

**通知順序之設計考量**：

- **先到先得**：最符合使用者期待，爭議最少，為業界通行做法。
- **全部同時通知**：會造成「搶位」體驗，且多數候補者會因「已被他人搶走」而降低對該餐廳之好感。
- **依消費金額排序**：僅適用於 VIP 等級明顯之場域（如高級餐廳），一般場合不宜採用，否則將損害品牌公平形象。
- **避免僅通知前一名而讓他人等待過久**：故須設定「確認期限」機制，使流程有明確進度。

### 第十題【問題與討論】

試比較「完全展開」「樣板 + 例外」「RRULE 規則」三種課表資料模型於下列情境之適用性：(甲) 單店瑜伽教室一學期 16 週課表、(乙) 連鎖健身房全臺 20 家分店之月課表、(丙) 跨國企業線上培訓之跨時區課程。

**答（參考架構）**：

**（甲）單店瑜伽教室**：建議採「樣板 + 例外」。資料量（28 筆樣板 + 少數例外）適中，維護成本低，程式複雜度可控。完全展開會產生過多冗餘（448 筆），RRULE 則過於複雜不必要。

**（乙）連鎖健身房 20 家分店**：建議採「樣板 + 例外」並擴充為「分店樣板」。每家分店之課表為獨立樣板，但共用通用例外處理邏輯。若採 RRULE，可利於統一匯出全臺課表供學員跨店使用，但會增加初期開發成本，建議視跨店使用頻率決定。

**（丙）跨國企業線上培訓**：建議採「RRULE 規則」。跨時區課程涉及夏令時間、各地節日、參與者使用不同行事曆系統訂閱等複雜需求，iCalendar 標準能統一處理此類問題。亦可直接匯出 .ics 檔供使用者匯入 Google Calendar、Outlook 等系統。

評分重點：是否能依規模與跨系統需求選擇適當方案，並能具體說明判斷依據。
