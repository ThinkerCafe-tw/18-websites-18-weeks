### 二、重點整理與補充

#### （一）隱私設計的雙面並進原則

諮商預約網站（第十六講）之特殊性在於「訪客於預約前已處於情緒低潮」，若網站設計讓訪客感到資料外洩之焦慮，即使實際資安措施完備，仍會流失訪客。隱私設計需雙面並進。**技術面**包含：全站 HTTPS、資料加密儲存、最小必要蒐集（詳第二期）、記錄存取日誌、定期資安檢測。**心理面**包含：提供「匿名預約」選項（僅填化名與手機）、色調採溫暖之米色或淺綠而非醫療之白藍、預約流程文案避免「個案」「患者」等標籤化用語、明確告知「僅諮商師本人可見您的資訊」。教材第十六講已處理心理面，本期補充技術面之具體要求：資料庫欄位層級權限（例如初診表之「個人病史」欄位僅諮商師帳號可讀，櫃檯帳號僅可讀姓名與時段）、備份資料加密、離職員工帳號立即停用。雙面之關鍵在於**同步傳達**：網站須同時讓訪客感受到「我被尊重」（心理）與「我的資料安全」（技術），缺一不可。諮商業尤為如此，其他敏感產業（醫療、法律、人資）亦適用此原則。

#### （二）自動化通知的時機與頻率設計

活動報名網站（第十七講）之自動化通知設計涉及三個面向：時機、頻率、管道。**時機面**，一場活動之標準通知節點為：報名成功即時（確認信）、活動前 7 天（提醒）、活動前 1 天（細節與地圖）、活動當天早上（最終提醒）、活動後 1 天（感謝與問卷）。五個節點非固定，應依活動性質調整：線上講座取消「活動前 1 天地圖」；實體工作坊增加「活動前 3 天器材清單」；收費活動於報名後 24 小時未付款應加入「付款提醒」。**頻率面**，同一名報名者於一場活動中之通知總數以 5 封為上限，超過則訪客易將後續通知標記為垃圾郵件，損害寄件網域之信譽（sender reputation）。**管道面**，Email 適合長篇資訊與附件，SMS 適合短促提醒與驗證碼，LINE 官方帳號適合雙向溝通。三管道並用時，同一則訊息不應全通路發送，應依內容性質擇一為主、另一為輔。教材第十七講之自動通知系統採 Email + LINE 雙軌，本期擴充補充 SMS 之使用時機為「確認時段」與「活動前 1 小時提醒」此類即時性要求高之情境。

#### （三）金流整合之技術選型與法遵要求

顧問預約網站（第十八講）之金流整合涉及第三方服務之選型。臺灣常見選項有綠界、藍新、街口、LINE Pay、PayPal 等，選型考量為費率、入帳速度、支援之付款方式三項。中小業者首次接入建議以綠界或藍新為主：支援信用卡、ATM、超商代碼、電子錢包等多元管道；API 文件中文完整；且支援定期定額扣款，利於訂閱制服務。費率方面，信用卡約 2.5–3%，ATM 每筆 NT$10–15，超商代碼每筆 NT$25–35。入帳速度方面，信用卡為隔月結算，ATM 與超商為隔日。法遵要求需注意三點：第一，**不得自行儲存完整信用卡號**，應採 Token 機制；第二，**發票合規開立**，個人戶年營業額未滿 8 萬元免開，超過則應開立電子發票；第三，**境外交易須符合跨境金流法規**，外幣支付須採 Stripe、PayPal 等支援跨境之服務。

### 三、實例分析：顧問業一人公司之金流系統四階段

觀察獨立顧問、律師、會計師等一人專業服務業，其金流系統演進可分四階段。**第一階段「現金與面交」**：現場收取現金或支票，零手續費但帳務不透明、稅務風險高。**第二階段「純銀行轉帳」**：顧問提供帳號，客戶自行轉帳後通知確認；手續費低但需手動對帳。**第三階段「第三方金流 + 手動開單」**：使用綠界或藍新，客戶可選信用卡、ATM、超商；金流管道多元，但仍需顧問手動於平台建立每筆交易單。**第四階段「網站整合 + 自動開單」**：網站直接串接金流 API，客戶預約後自動產生交易單並寄送付款連結；款項入帳後系統自動發送收據與時段確認。顧問完全脫離交易手續之介入，得以專注諮詢本身。教材第十八講所建置之系統即此階段實作。金流系統升級不是炫技，而是把顧問之時間從帳務雜務中釋放出來。此為整學期從「網站」走到「商業閉環」之收斂點。

## 貳、習題與解答

### 第一題【選擇題】

諮商預約網站之隱私設計原則，下列敘述何者**正確**？

(A) 技術面之 HTTPS 與加密儲存即已足夠，心理面為次要
(B) 心理面之溫暖文案即已足夠，技術面交由工程師處理即可
(C) 技術面與心理面需同步並進，缺一不可
(D) 使用者若不擔心隱私即無需設計隱私機制

**答：(C)**

**解析**：隱私設計之核心為「訪客同時感受到被尊重與資料安全」。技術面確保「實際上安全」，心理面確保「感受上安心」，兩者缺一訪客即會流失。(A) 忽視心理面將導致訪客於預約前即因焦慮放棄。(B) 忽視技術面將造成實質外洩風險。(D) 使用者是否擔心隱私與業者之法定義務無關，依個資法皆應採取適當措施。

### 第二題【選擇題】

一場為期一天之實體工作坊，下列通知時程何者**最合理**？

(A) 報名成功即時、活動前 14 天、活動前 7 天、活動前 3 天、活動前 1 天、活動當天早上、活動後 1 天（共 7 封）
(B) 報名成功即時、活動前 7 天、活動前 1 天、活動當天早上、活動後 1 天（共 5 封）
(C) 僅於報名成功與活動當天發送（共 2 封）
(D) 每日發送倒數計時訊息

**答：(B)**

**解析**：業界通行標準為 4–5 封通知。(A) 7 封已超過上限，報名者易標記為垃圾郵件。(B) 節點完整且不冗餘：報名確認、一週前提醒、一日前細節、當日最終提醒、事後感謝問卷。(C) 節點不足，活動前提醒缺失將造成出席率下降。(D) 頻率過高，屬騷擾行為。

### 第三題【選擇題】

顧問接受信用卡付款時，關於「儲存信用卡號」之敘述，下列何者**正確**？

(A) 只要資料庫加密即可自行儲存完整卡號
(B) 不得自行儲存完整卡號，應採第三方金流之 Token 機制
(C) 可儲存前 6 碼與後 4 碼，中間以 X 遮蔽
(D) 須取得客戶同意後始得儲存完整卡號

**答：(B)**

**解析**：依 PCI-DSS（Payment Card Industry Data Security Standard）規範與臺灣金管會之相關規定，非金融機構不得自行儲存完整信用卡號。正確作法為採第三方金流服務之 Token 機制：卡號僅傳送至金流服務商，本站僅儲存該次交易之 Token（一組代碼），日後定期扣款時以 Token 向金流服務商請求扣款即可。(A)(D) 違反規範。(C) 之遮蔽方式雖較安全，但仍屬自行儲存範疇，不符規定。

### 第四題【是非題】

諮商預約網站採溫暖米色與淺綠色調，並提供匿名預約選項，即可完全取代技術面之資安措施。（　）

**答：（×）**

**解析**：心理面之設計無法取代技術面之資安責任。即使訪客於預約時感到安心，若資料庫未加密、無存取日誌、離職員工仍可存取，一旦發生外洩事件，業者仍須承擔個資法之法律責任，且訪客對業者之信任將一次崩解。心理面與技術面之關係為「並進」而非「取代」。

### 第五題【是非題】

顧問業者每月收入不足 NT$8 萬元，即不需開立任何形式之發票或收據。（　）

**答：（×）**

**解析**：依《加值型及非加值型營業稅法》規定，個人戶每月營業額未達起徵點（銷售貨物 NT$8 萬元、銷售勞務 NT$4 萬元）者，免辦營業登記並免開統一發票，但仍應依客戶要求開立「收據」。此外，若客戶為公司行號需要發票以列報費用者，業者若無法開立，將失去此類客戶。實務建議：業者年收入若預計超過起徵點，應主動辦理營業登記並使用電子發票系統；若長期低於起徵點，亦應備妥「普通收據」供客戶索取。

### 第六題【應用題】

試設計一組資料庫欄位層級權限規則，適用於諮商預約系統之「預約表」，使櫃檯帳號僅能讀取基本資訊，諮商師帳號可讀取完整資訊。

**答**：

**欄位分類與權限**：

| 欄位 | 櫃檯帳號 | 諮商師帳號 | 系統管理員 |
|---|---|---|---|
| id | 讀 | 讀 | 讀/寫 |
| 預約時段 | 讀/寫 | 讀 | 讀/寫 |
| 姓名 / 化名 | 讀 | 讀 | 讀/寫 |
| 手機末四碼 | 讀 | 讀 | 讀/寫 |
| 完整手機 | ✗ | 讀 | 讀/寫 |
| 預約主題摘要 | ✗ | 讀 | 讀/寫 |
| 個人病史 | ✗ | 讀/寫 | ✗（僅備份） |
| 諮商紀錄 | ✗ | 讀/寫 | ✗（僅備份） |

**實作方式（以 PostgreSQL 為例）**：

```sql
-- 建立櫃檯角色
CREATE ROLE receptionist;
GRANT SELECT (id, time_slot, name_or_alias, phone_last4)
  ON appointments TO receptionist;
GRANT UPDATE (time_slot) ON appointments TO receptionist;

-- 建立諮商師角色
CREATE ROLE therapist;
GRANT SELECT, UPDATE ON appointments TO therapist;

-- 限制諮商紀錄僅該諮商師可見
CREATE POLICY therapist_own_records ON session_notes
  FOR ALL TO therapist
  USING (therapist_id = current_user_id());
```

**解析**：欄位層級權限（column-level access control）為諮商、醫療、法律等高隱私產業之基本要求。櫃檯帳號僅需看到「姓名與手機末四碼」以確認身分；諮商師始可讀取完整病史與諮商紀錄；系統管理員即使擁有資料庫最高權限，亦應透過加密備份機制避免直接讀取敏感欄位。此設計同時滿足個資法第 27 條之安全維護義務與業界對諮商隱私之倫理要求。

### 第七題【應用題】

試撰寫一段 JavaScript（或虛擬碼），計算一場實體工作坊之自動化通知應於何時發送。已知工作坊時間為 2026-05-20 14:00，報名者為 john@example.com，請產出完整之通知排程時間表。

**答**：

```javascript
function generateNotificationSchedule(eventDate, registrationTime, email) {
  const event = new Date(eventDate);
  const registration = new Date(registrationTime);

  const schedule = [
    {
      time: registration,
      type: 'confirmation',
      subject: '【報名成功】您已完成工作坊報名',
      recipient: email
    },
    {
      time: new Date(event.getTime() - 7 * 24 * 60 * 60 * 1000),
      type: 'reminder_week',
      subject: '工作坊一週後登場，請確認您的出席',
      recipient: email
    },
    {
      time: new Date(event.getTime() - 1 * 24 * 60 * 60 * 1000),
      type: 'reminder_day',
      subject: '明日工作坊，地圖與交通資訊',
      recipient: email
    },
    {
      time: new Date(event.getTime() - 4 * 60 * 60 * 1000),
      type: 'reminder_morning',
      subject: '工作坊今日下午登場',
      recipient: email
    },
    {
      time: new Date(event.getTime() + 24 * 60 * 60 * 1000),
      type: 'thanks',
      subject: '感謝您的參與，請協助填寫問卷',
      recipient: email
    }
  ];

  return schedule.filter(notif => notif.time > new Date());
}

// 使用範例
const schedule = generateNotificationSchedule(
  '2026-05-20T14:00:00+08:00',
  '2026-04-15T10:30:00+08:00',
  'john@example.com'
);

console.log(schedule);
```

**解析**：此函式之設計要點有三。第一，**以事件時間為基準向前推算各通知時間**，確保不論報名時間早晚皆能正確生成排程。第二，**過濾已過之通知**（`notif.time > new Date()`），避免系統對過去時點發送通知（例如報名時距活動僅 3 天，則「一週前提醒」不會被加入排程）。第三，**結構化之 schedule 物件**便於後續擴充（加入 LINE、SMS 管道），亦便於客製化單一活動之通知內容。實務應用時應以 cron job 或訊息佇列（Redis Queue）執行排程發送，避免依賴單一伺服器之 setTimeout。

### 第八題【情境題】

某諮商所欲上線預約網站，老闆擔心「客戶可能因為網站上有自己的資料而不敢預約」。請提出五項具體設計，同時解決技術面之安全性與心理面之安心感。

**答**：

1. **匿名預約機制**：預約表僅需「化名」與「手機末四碼」兩項必填欄位，完整資料於首次諮商時以紙本或口頭確認，避免訪客於未建立信任前即需留下完整資訊。
2. **色調與文案設計**：全站採溫暖之米色、淺綠或淡粉色，避免醫療之白藍；文案避免「個案」「患者」「病人」等標籤化用語，改以「預約者」「前來的朋友」等中性詞彙。
3. **明確之存取權限告知**：於網站顯著位置說明「您的預約資訊僅諮商師本人可讀，櫃檯人員僅可看到姓名與時段以確認身分」，並附「資料保存 3 年後自動刪除」之政策。
4. **技術面之具體措施**：全站 HTTPS 強制導向、資料庫欄位級權限（如第六題之設計）、離職員工帳號於離職日立即停用、每季進行一次資安檢測。
5. **緊急狀況之處理流程公開**：說明「若您預約後有任何顧慮或希望取消，可於 24 小時內直接來信，我們將立即刪除所有相關資料」，讓訪客知道有退出機制，反而提升預約勇氣。

此五項設計同時作用於技術與心理兩面，缺一不可。

### 第九題【情境題】

某獨立顧問欲將現有「純銀行轉帳」之金流模式升級為「網站整合 + 自動開單」，每月預估交易約 30 筆、平均每筆 NT$5,000。請分析升級之成本與效益，並建議是否升級。

**答**：

**升級成本分析**：

1. **金流服務商之費率成本**（以綠界信用卡 2.8% 為例）：30 筆 × 5,000 × 2.8% = NT$4,200 / 月
2. **開發成本**：若外包 API 串接約 NT$30,000–60,000 之一次性成本；若自行開發（本課程學員）約 40–80 小時之學習與實作時間
3. **維護成本**：每月約 2–4 小時之對帳與例外處理

**升級效益分析**：

1. **時間效益**：原「純銀行轉帳」模式每筆交易之手動對帳約 5 分鐘，30 筆 × 5 = 150 分鐘 / 月。升級後減為自動對帳 + 每月 2 小時例外處理，節省約 30 分鐘 / 月。
2. **轉換率提升**：提供信用卡付款可提升約 10–20% 之交易達成率（部分客戶厭惡 ATM 轉帳之麻煩）。若提升 15%，每月可多 4.5 筆交易 × 5,000 = NT$22,500
3. **品牌專業度**：擁有自動化金流之網站明顯較「請匯款至某帳號」之型態更具專業感，有利於服務費之提高

**建議**：

- 若顧問預期未來 6 個月內每月交易可穩定於 30 筆以上，應升級。費率成本 NT$4,200 遠低於轉換率提升所帶來之 NT$22,500 額外收入。
- 若交易量不穩定或有可能下降，建議先維持現狀，改以「LINE 官方帳號接收報名 + 人工發送綠界付款連結」之過渡方案。
- 自行開發為學員之學習機會，建議在能力允許之範圍內自行實作，不僅省下外包費用，亦為職涯累積實戰作品。

### 第十題【問題與討論】

綜整整學期 18 講之學習，試論「個人網站」對於一人公司或自由工作者之三項核心價值，並說明你個人最大的學習收穫。

**答（參考架構）**：

本題為整學期收束題，無標準答案。評分重點在於：

1. **是否能跨越單一講次之視角**，整合 18 講之共通脈絡；
2. **是否能以自身情境反思學習收穫**，而非僅複述教材；
3. **是否展現對「網站作為商業基礎建設」之理解**，而非僅視為技術練習。

**三項核心價值之參考面向**：

- **第一：自主性之具體化**。擺脫平台演算法與抽佣的影響，擁有直接面對客戶的管道。
- **第二：信任之可見化**。專業能力、過往案例、服務流程透過網站具體呈現，降低客戶之決策成本。
- **第三：時間之解放**。自動化之預約、通知、金流系統將業者從重複性雜務中釋放，專注於真正有價值之專業服務。

**學習收穫之反思面向**：

- 技術面：掌握 HTML/CSS/JavaScript 之實作能力，能夠自行建置網站而非依賴模板。
- 思維面：理解網站非裝飾品，而是商業模式之載體。
- 工具面：學會與 AI 協作開發，大幅降低個人網站之建置門檻。
- 應用面：能將本學期所學應用於自身或家人朋友之事業，產生實際效益。

本題為整學期之反思性總結，無論學員回答之內容為何，只要能展現「從 18 週的學習中提煉出個人之理解」，即為合格。評分不看結論，看思考深度與個人化程度。
