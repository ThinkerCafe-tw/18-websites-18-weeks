# 第 11 講：民宿訂房網站

民宿最尷尬的一刻，是旅客已經被首頁的氣氛打動、卻在訂房區看到「請先匯款再傳簡訊」，於是默默退回 Airbnb。本講就用 AI（Artificial Intelligence，人工智慧）協作做出一個民宿訂房網站原型，把房型、日期、價格與取消政策接成一條「從想住到送出訂房」的可靠流程。

本講完成後，你會得到四個成果：

- 一份民宿訂房網站需求摘要。
- 一個可展示的訂房網站原型。
- 一份房型、房況、訂單、付款與後台的資料模型草稿。
- 一份功能型網站驗收清單，能檢查是否有超賣、假付款、假評論或流程斷點。

## 1. 破冰案例

週五晚上八點半，一對夫妻把車停在休息站找下週末要住的地方。Booking.com、Airbnb 都很熟，但這次想找的不是標準化住宿，於是搜尋「宜蘭 礁溪 手作民宿」。

搜尋結果第三個不是大型平台，而是一家民宿自己的官網。首頁第一張照片不是空房間，而是早晨的光從竹百葉窗斜切進來，桌上一碗稀飯配幾碟醃漬小菜。副駕駛座上的她看了三秒，說：「這家感覺對。」

她往下滑，看見房間、早餐、散步路線，也看見主人寫的一小段話：為什麼搬到礁溪、為什麼只做三間房。信任開始建立。可是點到訂房區，她看到一句話：

```text
如欲訂房，請先匯款至以下帳號，再傳簡訊告知入住日期與人數，我們確認後回覆。
```

她停了一下，把手機放回腿上，又打開 Airbnb。

這家民宿做對了一件事：讓旅客感覺不是在買一張床，而是進入一種生活。做錯的是沒把這份信任接到現代訂房流程——旅客怕房間沒保留、匯款後沒確認、臨時取消不能退錢。

民宿網站的難度就在這裡：既要像一個有溫度的家，也要像一個可靠的系統。

## 2. 業態需求

民宿訂房網站要同時解決三層問題：旅客的不安、房東的收入、系統的準確性。

第一，旅客要確認這裡適不適合他。飯店有品牌標準，民宿靠獨特性；獨特性吸引人，也讓人不安——怕照片修太多、隔音差、出問題沒人處理。所以民宿網站要比形象網站提供更多證據：真實照片、設備與限制、入住與取消規則、夜間導航。

第二，房東要降低對 OTA（Online Travel Agency，線上旅遊訂房平台，例如 Booking.com、Airbnb、Agoda）的依賴。OTA 能帶流量，也會抽成；民宿若每張訂單都靠平台，會遇到表 11-1 的問題。

**表 11-1　過度依賴 OTA 平台的問題與影響**

| 問題 | 對民宿的影響 |
|---|---|
| 平台抽成 | 每張訂單約有一部分房費流向平台 |
| 評價綁架 | 評論與排名留在平台，民宿自己帶不走 |
| 定價壓力 | 平台促銷、排名與折扣機制會影響價格 |
| 客戶資料隔離 | 房東難以經營回頭客 |

官網直訂不是取代平台，而是讓平台負責拉新、官網負責留存。

第三，系統要避免訂房錯誤。咖啡廳網站做錯按鈕只少一通電話；民宿訂房網站做錯，可能讓同一間房同天被兩組旅客訂走。

所以本講正式進入「功能型網站」：它不只展示資訊，而要處理資料、狀態與流程——哪間房哪天可訂、付款前是否保留、付款與取消後房況如何同步。

因此本講的技術主題是「訂單狀態機與付款閉環」。第 4 節把房況與訂單拆成兩條對應的狀態流程，第 5 節讓 AI 補上資料模型與狀態，再用驗收清單檢查保留、付款、取消與退款能否接起來。

基本業態假設見表 11-2。

**表 11-2　本講的基本業態假設**

| 項目 | 本講設定 |
|---|---|
| 店型 | 小型獨立民宿 |
| 主要客群 | 週末短住旅客、夫妻、朋友小旅行、家庭客 |
| 網站型態 | 功能型訂房網站原型 |
| 核心行動 | 查詢房況、選擇日期、送出訂房、完成付款或付款模擬 |
| 技術目標 | 用 AI 產出可展示的訂房流程，並拆出資料模型與狀態流程 |

這一講不要求真的串接金流，第一版目標是「看得懂、跑得通、查得出問題」。

## 3. 網頁設計重點

### 3.1 網站風格

民宿網站的風格不能只追求漂亮，要讓旅客感覺「我知道我會住進什麼地方」。大型平台的照片追求標準化，民宿官網可以反過來把生活感放進去，例如早晨餐桌、窗外景色。但生活感不等於照騙——把三坪房間拍成六坪，旅客抵達後感到落差，信任就破掉。不同類型的民宿視覺語言不同，首頁重點也要跟著調整。

第一張照片尤其重要。研究指出，使用者對網頁的第一印象在極短時間內形成（Lindgaard 等人，2006，見 6.7 參考文獻）；首圖是整個訂房漏斗的開關，要在半秒內回答「這裡是什麼氣氛、想不想把週末放這裡」。

### 3.2 建立信任

信任模組要直接回應旅客的恐懼：旅客不只怕價格高，更怕到現場才發現和想像不同，如表 11-4。

**表 11-4　民宿訂房網站的信任模組**

| 信任模組 | 應該包含 | 解決什麼不安 |
|---|---|---|
| 房型照片 | 主圖、副圖、浴室、窗景、入口 | 避免照騙 |
| 設備與規則 | 床型、衛浴、停車、check-in、寵物 | 確認需求、降低溝通成本 |
| 取消政策 | 幾天前可退、退多少、如何申請 | 降低損失焦慮 |
| 真實評論 | 可含小缺點，不只全五星 | 提供第三方證據 |
| 主人介紹 | 為何經營、如何接待、聯絡方式 | 讓人知道背後有人 |
| 交通指引 | 導航連結、停車入口、夜間提醒 | 避免最後一哩焦慮 |

取消政策尤其重要。行為經濟學的「損失趨避」指出，人對損失比對等值獲得更敏感（Kahneman 與 Tversky，1991，見 6.7 參考文獻），太硬的取消政策會讓旅客在下單前停住。階梯式取消政策（表 11-5）用「越早取消退越多」平衡安全感與成本。

**表 11-5　階梯式取消政策範例**

| 距離入住日 | 建議退款比例 | 設計理由 |
|---|---|---|
| 30 天以上 | 100% | 行程還早，給旅客信心 |
| 14 到 30 天 | 75% | 房東已有部分機會成本 |
| 7 到 14 天 | 50% | 重新出租時間變短 |
| 3 天內 | 0% 或僅保留延期 | 房東很難再賣出房間 |

規則清楚，比看似彈性卻要私訊詢問更能建立信任。

### 3.3 引導行動

民宿網站的轉換路徑比形象網站長，旅客會經過一連串確認，基本路徑可設計成：

```text
看到首頁 → 看房型 → 查日期 → 看價格與規則 → 填資料 → 付款或送出訂房 → 收到確認
```

每一步要回答一個問題，如表 11-6。

**表 11-6　訂房路徑各步驟要回答的問題**

| 步驟 | 旅客心裡的問題 | 網站要提供的答案 |
|---|---|---|
| 首頁 | 這裡是不是我想住的地方？ | 主張、首圖、生活感 |
| 房型 | 這間房適合幾個人？ | 人數、床型、設備、照片 |
| 日期 | 我想住的日期有沒有房？ | 日期選擇器與房況顯示 |
| 價格 | 總共多少錢？有沒有額外費用？ | 房價、加人費、清潔費、折扣 |
| 規則 | 不合適或臨時取消怎麼辦？ | 取消政策、入住規則 |
| 付款 | 我付了錢後有沒有被確認？ | 付款狀態與訂房確認 |

直訂優惠可放在首頁與訂房區，但要精準，例如：

```text
官網直訂，比平台價格省 8%，並附入住前在地散步地圖。
```

這句比「官網最優惠」更好，因為點出了利益與價值。

## 4. 功能與資料模型

民宿訂房網站是本課程第一個完整的功能型網站範例。

### 4.1 為什麼不能只做表單

如果網站只有「我要訂房」表單，送出後房東還要人工確認日期、是否撞單、人數與加價，客人一多就容易超賣或漏回。表單只收資料、不保證房況，真正的系統至少要做到：

1. 日期與房況。
2. 訂單與付款。
3. 後台與管理。

### 4.2 頁面與功能模組

民宿訂房網站可拆成的模組見表 11-7。

**表 11-7　民宿訂房網站的頁面與功能模組**

| 模組 | 功能 | 必要內容 |
|---|---|---|
| Hero 首頁 | 建立第一印象 | 民宿名稱、生活主張、首圖、直訂優惠 |
| 空間故事 | 建立人感 | 主人介紹、早餐、在地連結、經營理念 |
| 房型列表 | 讓旅客比較 | 房型、價格區間、人數、照片、設施 |
| 房型詳情 | 讓旅客判斷 | 主圖、副圖、床型、衛浴、規則、可入住人數 |
| 日期選擇器 | 查詢房況 | 入住日、退房日、已訂日、價格 |
| 訂房表單 | 收集資料 | 姓名、電話、Email、人數、備註 |
| 付款流程 | 完成確認 | 訂單摘要、付款狀態、成功與失敗頁 |
| 在地指南 | 增加民宿價值 | 景點、餐廳、散步路線、雨天備案 |
| 交通指引 | 降低抵達焦慮 | 導航、停車、夜間路標 |
| 後台 | 讓房東管理 | 房況、訂單、價格、黑名單日期 |

### 4.3 資料模型

先把資料拆出來，AI 才不會把內容寫成漂亮但不能管理的頁面。資料模型見表 11-8。

**表 11-8　民宿訂房網站的最小資料模型**

| 資料物件 | 說明 | 重要欄位 |
|---|---|---|
| `RoomType` 房型 | 對外顯示的房型，例如雙人房、四人房 | 名稱、人數、坪數、設備、照片、平日價、週末價 |
| `Room` 房間 | 實際可出租的房間，例如 201、202 | 房間編號、所屬房型、是否上架 |
| `Availability` 房況 | 某間房在某一天的狀態 | 房間、日期、狀態、價格、保留到期時間 |
| `Order` 訂單 | 旅客的一次訂房 | 入住日、退房日、人數、旅客資料、總金額、訂單狀態 |
| `Payment` 付款 | 訂單的付款紀錄 | 金流商、交易號、金額、付款狀態、通知時間 |
| `Guest` 旅客 | 回頭客或會員資料，第一版可先嵌在訂單中 | 姓名、Email、電話、入住次數 |

最容易被初學者忽略的是 `RoomType` 和 `Room` 的差別：「檜木雙人房」是一種房型，若實際有三間，系統裡就必須是三個房間，否則賣出一間可能誤判整個房型沒了，或反過來超賣。

### 4.4 房況狀態

房況不是一句「有房 / 沒房」就能表示，某天某間房可能處在不同狀態，如圖 11-1（請排版將下列狀態轉換製成流程圖；此圖含分支，非線性橫式流程）。

**圖 11-1　某一間房在某一天的房況狀態流程**

```text
可訂 → 保留中 → 已預訂 → 入住中 → 清潔中 → 可訂
  │        │
  │        └→ 付款逾時 → 可訂
  └→ 封鎖 / 維修
```

各狀態意思如表 11-9。

**表 11-9　房況各狀態的意義與典型觸發**

| 狀態 | 白話說明 | 典型觸發 |
|---|---|---|
| 可訂 | 旅客可以選這一天 | 沒有訂單、沒有封鎖 |
| 保留中 | 旅客正在付款，先暫時鎖住 | 訂房表單送出 |
| 已預訂 | 付款成功或房東確認 | 金流通知成功 |
| 入住中 | 旅客正在住 | 入住日到達 |
| 清潔中 | 旅客退房後等待整理 | 退房日到達 |
| 封鎖 / 維修 | 房東手動不開放 | 自住、維修、包棟、節日調整 |

「保留中」是訂房系統關鍵的一步：旅客付款期間房間若還顯示可訂，另一人同時下單就會超賣。解法是送出訂房就把房間鎖一段時間（例如 15 分鐘），付款成功變已預訂，逾時釋放回可訂。

### 4.5 訂單與付款狀態

訂單狀態和房況狀態不同，但必須對應，如圖 11-2（請排版將下列狀態轉換製成流程圖；此圖含分支，非線性橫式流程）。

**圖 11-2　訂單狀態流程**

```text
待付款 → 已確認 → 已完成
   │         │
   │         └→ 已取消 → 已退款
   └→ 已取消
```

付款流程分兩種畫面，如表 11-10。

**表 11-10　付款返回頁與金流通知 webhook 的分工**

| 畫面 | 功能 | 注意事項 |
|---|---|---|
| 付款返回頁 | 旅客付款後被導回網站看到的畫面 | 只能當使用者體驗提示 |
| 金流通知 webhook | 金流公司主動通知網站付款結果 | 應作為系統確認付款的主信源 |

Webhook（外部服務事件通知）是金流公司主動通知網站付款結果的機制，必須以它為主信源，不能只信成功頁。第一版課堂原型不用真的串接金流，但流程要正確：做「付款模擬」按鈕與成功、失敗頁，不宣稱已串接、不收真實信用卡資料。

### 4.6 後台管理

民宿網站只有前台，房東很快卡住，因為房價、房況、維修日、退款都會變。後台至少四個模組，見表 11-11。

**表 11-11　民宿後台的四個基本模組**

| 後台模組 | 房東要做什麼 |
|---|---|
| 房況管理 | 在日曆上鎖日期、開日期、標記維修或私人使用 |
| 訂單管理 | 查看待付款、已確認、已取消訂單，匯出月結 |
| 價格管理 | 設定平日、週末、旺季、連住折扣 |
| 黑名單日期 | 春節、自住、設備維修等日期不可訂 |

動態定價第一版先分四層就夠：平日價、週末加價、旺季價、長住折扣。後台要記得：房東通常不是工程師，要像日曆而不是資料庫，能點選就別讓他寫公式。

### 4.7 從原型走向真實訂房系統

民宿訂房網站進入真實營運後，技術難點不是畫日曆，而是「同一間房在同一天只能賣一次」。這在前端只是一個灰色日期，在系統裡卻牽涉資料庫鎖定、訂單狀態、付款時限與金流通知，可拆成五個節點，見表 11-13。

**表 11-13　真實訂房流程的五個技術節點**

| 技術節點 | 系統要做什麼 | 如果沒處理會怎樣 |
|---|---|---|
| 查房況 | 依房間、日期、人數查詢可售狀態 | 已訂日期仍被顯示可訂 |
| 暫時保留 | 送出訂房後鎖住房間一段時間 | 兩個旅客同時付款造成超賣 |
| 建立訂單 | 產生訂單編號、總價、取消規則與付款期限 | 房東無法追蹤每筆交易 |
| 確認付款 | 由 webhook 更新付款與房況狀態 | 成功頁關掉或重整後狀態不一致 |
| 釋放或退款 | 付款逾時、取消、退款時更新房況 | 房間被卡住，明明可賣卻不能賣 |

正式訂房系統還需要「交易式處理」（訂單與房況保留一起成功或失敗），並有三個保護：房況逐房逐日進資料庫、付款由 webhook 驗證後才更新、保留與退款留紀錄。

## 5. AI 協作實作

這一講的 AI 協作目標不是做出正式可收款的系統，而是做出可展示、可討論、可驗收的原型，建議分四輪：

1. 先產出網站原型。
2. 再補資料模型。
3. 再補互動狀態。
4. 最後做驗收與除錯。

### 5.1 第一輪：產出訂房網站原型

可以使用這段提示詞：

```text
幫我建一個民宿訂房網站原型。

網站名稱：山邊晨光民宿
地點：宜蘭礁溪
主要客群：台北週末短住旅客、夫妻、小家庭
風格：溫暖、安靜、有手作感
色調：米白、木褐、深苔綠

請做成一個可以直接打開的單一 index.html（HyperText Markup Language，超文字標記語言），包含以下區塊：
1. 首頁主視覺：一句話說清楚民宿生活主張，並放兩個按鈕：「查看房型」「查詢日期」
2. 空間故事：主人為什麼在這裡開民宿，早餐食材從哪裡來
3. 房型列表：至少兩種房型，每種要有人數、床型、平日價、週末價、主要設備
4. 日期查詢：入住日、退房日、人數選擇器，先用前端模擬房況即可
5. 訂房摘要：顯示選擇的房型、晚數、總價、取消政策
6. 訂房表單：姓名、Email、電話、備註
7. 付款模擬：付款成功與付款失敗兩種狀態，不要收真實信用卡資料
8. 在地指南：三個推薦地點
9. 交通指引：開車、搭車、停車與夜間抵達提醒
10. 常見問答：入住、退房、早餐、寵物、取消政策

技術要求：
1. 請做成單一 index.html，CSS（Cascading Style Sheets，階層式樣式表）與少量 JavaScript（網頁互動程式語言）都寫在同一個檔案裡。
2. 手機版要好讀，按鈕高度至少 44px。
3. 日期不可選到過去。
4. 若選到已預訂日期，要顯示不可訂。
5. 價格要依晚數計算。
6. 所有圖片先用免費圖庫的示意圖並以註解標出替換檔名；上線前必須換成民宿實景，不能拿示意圖當真實照片用。
7. 不要產生假評論、假媒體報導或假證照。
8. 程式碼中請用註解標出哪些地方未來要接資料庫、金流 webhook 與後台。
```

產出的網站如果漂亮但流程跑不通，不算完成：民宿網站不是海報，要能讓人選日期、看價格、填資料、走完訂房。

### 5.3 第二輪：要求 AI 補資料模型

第一版畫面出來後，接著問 AI：

```text
請根據剛才的民宿訂房網站，幫我整理資料模型。

請用表格列出：
1. RoomType 房型
2. Room 實際房間
3. Availability 每房每日房況
4. Order 訂單
5. Payment 付款
6. Guest 旅客

每個資料物件請列出欄位、欄位說明、範例值。
請特別說明 RoomType 和 Room 的差別，以及 Availability 為什麼要用「每房每日一筆」來設計。
```

這一步讓學生理解：卡片、表單、日曆都對應到資料，沒拆清楚就難維護。

### 5.4 第三輪：要求 AI 補狀態流程

接著補上狀態流程：

```text
請幫我補上民宿訂房的狀態流程。

請包含：
1. 房況狀態：可訂、保留中、已預訂、入住中、清潔中、封鎖、維修
2. 訂單狀態：待付款、已確認、已取消、已退款、已完成
3. 付款狀態：未付款、付款中、付款成功、付款失敗、已退款

請用白話說明每一個狀態如何轉換。
也請說明為什麼付款成功頁不能取代 webhook。
```

### 5.5 第四輪：修改與除錯

AI 第一次產出的訂房網站常見五種問題，見表 11-14。

**表 11-14　AI 產出訂房網站的五種常見問題**

| 常見問題 | 為什麼危險 | 修改方式 |
|---|---|---|
| 只做聯絡表單 | 看起來能訂，其實不能查房況 | 要求加入日期查詢與房況狀態 |
| 所有日期都能選 | 會造成已訂日期仍可下單 | 加入已訂日與不可選邏輯 |
| 總價不會變 | 晚數、週末、加人費都沒算 | 要求顯示計價公式 |
| 付款按鈕假裝真串接 | 可能誤導使用者 | 改成「付款模擬」，標明未串接 |
| 放假評論或假證照 | 破壞信任，也可能違法 | 全部刪除，改成「待填入真實評論」 |

可用這段提示詞修正：

```text
請檢查剛才的民宿訂房網站是否有以下問題：
1. 是否只是表單，沒有房況查詢？
2. 是否所有日期都能選？
3. 是否沒有計算晚數與總價？
4. 是否假裝已串接真實金流？
5. 是否出現假評論、假媒體報導或假證照？

請直接在原本的 index.html 上改，保留原本視覺風格，但補上缺少的互動與提醒文字。
```

## 6. 上線與延伸

第一版民宿訂房網站可先用靜態網站上線，例如 Vercel（前端部署平台）或 Netlify，適合展示品牌、房型、在地指南與訂房流程；但正式營運前，要分清原型與正式系統。

### 6.1 第一版可以上線的內容

第一版可安全上線的內容：民宿介紹、房型介紹、真實照片、價格區間、入住規則、取消政策、交通指引、聯絡方式與表單詢問。

如果還沒有資料庫與金流，不要把網站寫成「即時訂房已完成」，可改成：

```text
送出後，我們會在 2 小時內確認房況並回覆。
```

這句不如即時訂房方便，但誠實。

### 6.2 正式營運需要的系統

要變成真正的訂房系統，後續需加入的見表 11-16。

**表 11-16　正式營運需要補齊的系統**

| 延伸功能 | 需要處理的事情 |
|---|---|
| 資料庫 | 儲存房型、房間、房況、訂單、付款 |
| 會員或訂單查詢 | 讓旅客查自己的訂單，不一定要完整會員系統 |
| 金流串接 | 綠界、藍新、Stripe 或其他付款服務 |
| Webhook | 以金流通知確認付款結果 |
| Email / LINE 通知 | 訂房成功、入住提醒、取消通知 |
| 房東後台 | 日曆鎖房、改價、查訂單、處理退款 |
| 資安與個資 | 電話、Email、付款資料不可隨便曝光 |

這些不必全塞進第一版，留待第二版。

### 6.3 真實部署前夜檢查

店家若問「能不能拿去營運？」要看表 11-17 條件是否補齊。

**表 11-17　真實部署前夜檢查清單**

| 檢查點 | 需要確認的內容 |
|---|---|
| 資料庫 | 房型、房間、每日房況、訂單與付款紀錄是否真的保存 |
| 後端 API | 查房況、建立訂單、取消訂單、更新付款是否由後端處理 |
| 併發保護 | 兩個人同時訂同一間房時，系統是否只允許一筆成功 |
| 金流環境 | 是否使用商家自己的金流帳號、測試環境與正式環境 |
| Webhook 驗證 | 是否驗證金流商簽章，而不是只相信前端傳回的結果 |
| 通知紀錄 | Email 或 LINE 通知是否有發送紀錄與失敗重送機制 |
| 權限 | 房東後台是否有登入與操作紀錄，避免任何人都能改房況 |
| 個資 | 旅客電話、Email、付款紀錄是否有最小化收集與保護 |

一旦牽涉金錢、個資與庫存，責任就從「畫面好不好看」轉到「資料狀態能不能被正確維護」，這正是功能型網站的專業判斷。

### 6.4 可以延伸的商業功能

民宿收入不只來自房費，網站後續可加入：

- 在地導覽預約。
- 早餐加購。
- 接駁服務。
- 租車或活動推薦。
- 回頭客折扣碼。
- 入住後評價邀請。

但每個延伸功能都要回到同一原則：不讓旅客覺得被推銷，而是被照顧。
