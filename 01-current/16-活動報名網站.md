# 第 16 講：活動報名網站

一個人盯著報名連結遲遲不按，卡住她的往往不是票價，而是頁面講完了講師和名額，卻沒讓她相信「去了之後我會變成想成為的那種人」。本講就用 AI（Artificial Intelligence，人工智慧）協作做出一個活動報名網站原型，把活動資訊、票種、名額、付款與通知流程，組成一個先建立信任、再促成報名的數位入口。

本講完成後，你會得到四個成果：

- 一份活動報名網站需求摘要。
- 一個可展示的網站原型。
- 一份活動、場次、票種、名額、報名人、通知狀態、候補名單的資料模型草稿。
- 一份手機版、信任證據、流程斷點與上線前檢查清單。

本講學習重點：

◆ 理解報名工作坊的人，心裡真正買的是什麼
◆ 掌握活動主辦方的三種核心收益邏輯：贊助商、階梯票價、現場加購
◆ 用心理學解釋「承諾一致性」與「稀缺性原則」如何驅動報名決策
◆ 認識三種常見的活動行銷黑暗手法，並理解為何不能走這條路
◆ 學會把本講產業需求轉成頁面模組、資料模型、AI 提示詞與驗收清單

## 1. 破冰案例

**本講情境：週四晚上十一點，她盯著那個連結**

週四晚上十一點，林欣瑜的 LINE 亮起，同事丟來一個連結：「超強！我朋友上個月去，說整個視野都打開了。」她點進去 —— **「高階品牌行銷實務工作坊」**，五千元，早鳥截止明晚十二點；講師一排斜槓身份，課程大綱六個標題都像這個月的行銷趨勢關鍵字。

五千元是她這個月薪資的八分之一，但讓她停在頁面上的不是回本計算，是想找一個理由說服自己。她讀學員評論，搜尋一種感覺：**有沒有人跟她一樣、去了之後變得不一樣？** 她找到一則「終於搞懂品牌策略，推薦！」，盯著想了三十秒——那個人做出了什麼具體的東西？她沒答案，關掉了頁面。

---

她不是因為五千元放棄的，是讀完頁面還是不知道「去了之後我會是誰」。頁面講了講師、課程與名額，卻沒告訴她：一個跟她一樣在職場卡關的企劃，去了之後帶回來什麼、改變了什麼。

**活動報名的本質不是購票，是讓一個人相信，這段時間花在這裡，會讓她成為想成為的那種人。** 從這個理解出發，每個設計決策——哪張照片放第一屏、先寫議程還是先寫學員故事——才有判斷基準。

## 2. 業態需求

**產業聆聽：她在報名頁面上到底在找什麼**

**三個核心恐懼**：(1)**知識空心化** —— 怕講師只重念網路免費查得到的東西；(2)**社交孤立** —— 怕自己程度最底層，想知道「一起上課的人是誰」；(3)**無法落地** —— 課上完什麼都用不上，只剩「花了五千元卻沒改變」。

**三個猶豫點**：大綱夠不夠具體（寫「帶走一份可套用的工作表」猶豫少一半）、過往學員組成（「適合所有行銷人」是警訊）、早鳥截止與「不予退款」的矛盾（要她在資訊不完整下做不可逆決定）。

**三個信任觸發**：實作產出的展示（讓她問「我也能做到嗎」）、講師的失敗經驗（失敗裡才有判斷邏輯）、課後社群的具體承諾（「每週四丟案例、親自回覆」，買的是持續運作的系統）。

**人性真相：客人買的往往不是「知識」，而是「對平庸的抵抗」。** 她買的是「跟沒進步的人不一樣」的自我認同，儀式感（講義、證書、小聚）有時比內容更能支撐滿意度。

---

### 2.1 產業營運邏輯

**產業心法：老手是怎麼算活動這門生意的**

老手算的是三層收益，不是「賣了多少票」：

- **贊助商才是主要收入。** 精準的參加者樣本是 B2B 品牌付費廣告也找不到的受眾，老手**讓贊助商精準曝光、參加者卻不覺得被推銷**（用場地、餐飲、工具試用換曝光與名單），收入可佔整體六成以上。
- **階梯票價管現金流。** 早鳥票收的是啟動資金，**鎖定願意先付錢的承諾者，再用他們的錢付廣告與訂金**；三段定價對應早決策、標準、衝動三類人，各付可接受的最高價。
- **現場加購層層拆分。** 把體驗拆成可個別定價的模組（講義、課後小組、照片包），毛利率遠高於票本身，利用承諾效應：**已決定花五千元，再多花五百元不算大決策**。

三層加起來才是真正的帳，只看票價的主辦人最快撐不下去。

因此本講的技術主題是「票務庫存與付款逾時處理」。前面談的階梯票價、名額稀缺與退票規則，落到網站上就是票種庫存、付款期限與候補遞補要同步管理：第 4 節把這些拆成可檢查的資料模型與六個後端判斷點，第 5 節的提示詞則要求 AI 在原型中示範待付款、付款逾時與候補遞補的狀態切換。

## 3. 網頁設計重點

### 3.1 網站風格

風格要快速傳達活動類型、時間感與可信度。主視覺可以有能量，但日期、地點、票種、講者、議程與退費規則要比裝飾更清楚。不同活動節奏不同：講座重講者內容，市集重攤位動線，工作坊重成果材料。

### 3.2 建立信任

信任模組包含主辦單位、講者、過往照片、票種名額、退費規則、個資同意與通知方式。活動常變動，更新紀錄與通知機制很重要；名額倒數可用，但必須連到真實票種與庫存狀態。

### 3.3 引導行動

主要行動是選票種、報名、加入候補、查看交通、加入行事曆、接收通知。CTA（Call to Action，行動呼籲）要跟票種狀態同步，如「報名早鳥票」「加入候補」。

### 3.4 心理學錨點

**你為什麼在倒數計時結束前按下去**

羅伯特．席亞爾迪尼（Robert Cialdini）在二○○六年《影響力》（*Influence: The Psychology of Persuasion*，Harper Business）記錄三個穩定規律，正是活動報名背後的大腦迴路：

- **承諾一致性（Commitment and Consistency）** —— 做出小承諾後行為傾向與之一致。早鳥票讓參加者四個月前就宣告「我是會去這種活動的人」，放棄它失去的不只是錢，還有對外宣告的身份。
- **稀缺性（Scarcity）** —— 對即將失去的機會反應強度約為同等獲益的兩倍；「名額僅剩 12 個」讓人跳過理性比較直接行動，用在真實供需是資訊，用在假名額是謊言。
- **社會證明（Social Proof）** —— 不確定時人觀察「相似的人在做什麼」。**讓林欣瑜留下來的不是頭銜，是背景相似的人拿著她在那堂課完成的東西說話。**

這三個原則不是操控工具，是人的行為本來如此，要用在真實供需。

### 3.5 信任底線

**三種讓你三年後被退費電話轟炸的手法**

**底線一：假性稀缺與影子名額。** 「僅剩 3 個名額」但後台從沒動過，數字寫死或定時重置，在他看清退票規則前逼他付款；一旦發現隔週又是「最後名額」，失去的是對品牌的永久懷疑。

**底線二：虛假卡司與不可抗力更換。** 列名知名人士撐高票價，活動前三天改他人代打、不予退票——用名字賣票、收錢後用「不可抗力」卸責，屬消費者保護法廣告不實的灰色地帶，那批被騙的學員會成為最有力的反向宣傳者。

**底線三：個資轉賣與授權陷阱。** 報名條款用小字寫「同意將個資提供予合作夥伴行銷」、打勾才能報名，活動後參加者接到貸款、保險、投資的電話，違反個人資料保護法目的明確告知的要求，傷害整個付費活動生態圈的報名意願。

---

**這些手法我們選擇不用。想讓人下次繼續來，「下次」建立在他上次確實得到他以為會得到的東西；短期名單換不了長期信任。**

---

## 4. 功能與資料模型

**實作主題：把「活動管理與一對多通知」落成可檢查的網站。**

這一層把判斷轉成網站結構。AI 協作不是丟一句「做個漂亮網站」，而是把業態需求、頁面模組、資料欄位、流程規則與驗收標準說清楚，再讓它產出第一版。需求先整理如表 16-1。

**表 16-1　活動報名網站的需求模組**

| 模組 | 功能 | 必要內容 |
|---|---|---|
| Hero 首頁 | 建立第一印象 | 名稱、定位、主張、主要行動按鈕 |
| 信任證據 | 降低猶豫 | 真實照片、資歷、評論、規則或案例 |
| 核心流程 | 完成主要任務 | 讓參與者理解活動資訊、票種、名額、付款與通知流程 |
| 資料表單 | 收集必要資訊 | 姓名、聯絡方式、情境欄位、備註 |
| 規則說明 | 避免誤解 | 價格、限制、取消或退款規則 |
| 聯絡與後續 | 接住下一步 | 電話、LINE、Email、地圖或確認訊息 |

### 頁面與功能模組

把網站想成一組模組，每個回答一個使用者問題：我在哪裡？這適合我嗎？我相不相信？下一步做什麼？單頁網站也要有結構：導覽列跳到主要區塊，主要按鈕放第一屏與結尾。

先拆出表 16-2 的資料物件。第一版不一定接資料庫，但你必須知道背後需要哪些資料，AI 才不會只產出表面文案。

**表 16-2　活動報名網站的基礎資料物件**

| UserRequest | 姓名、聯絡方式、需求、備註 | 用於頁面內容、表單或狀態檢查 |
| ServiceItem | 名稱、內容、價格或條件、限制 | 用於頁面內容、表單或狀態檢查 |
| ScheduleSlot | 日期、時段、名額、狀態 | 用於頁面內容、表單或狀態檢查 |
| Policy | 取消、退款、隱私或服務邊界 | 用於頁面內容、表單或狀態檢查 |
| TrustEvidence | 照片、評論、資歷、案例或證明 | 用於頁面內容、表單或狀態檢查 |

流程可以用這條線檢查：

```text
看見首頁 → 判斷是否適合 → 查看信任證據 → 進入核心流程 → 填寫必要資料 → 看到確認或下一步
```

中間任何一步若需要人用 LINE 或電話補洞，就要在網站上說清楚，別讓它假裝有完整系統。

### 票種、名額與候補狀態

核心不是「我要報名」按鈕，而是票種和名額管理；票種規則不清楚，後面就會出現退票、補差額、候補通知與報到混亂。資料模型可拆得更精準，如表 16-3。

**表 16-3　活動報名網站的精準資料模型**

| 資料物件 | 重要欄位 | 用途 |
|---|---|---|
| `Event` | 活動名稱、日期、地點、主辦單位、活動狀態 | 對外活動主資料 |
| `EventSession` | 場次、開始時間、結束時間、容量 | 多場次或分場管理 |
| `TicketType` | 票種名稱、價格、開賣時間、截止時間、名額 | 早鳥、一般、學生、VIP |
| `Registration` | 報名人、票種、付款狀態、報到狀態 | 一筆報名 |
| `PaymentRecord` | 金額、付款方式、付款結果、交易時間 | 付款追蹤 |
| `WaitlistEntry` | 候補順位、通知時間、保留期限 | 額滿後候補 |
| `NotificationLog` | 對象、內容、通道、發送狀態 | 一對多通知紀錄 |
| `ConsentRecord` | 個資用途、同意時間、撤回方式 | 個資與行銷授權 |

票種狀態如圖 16-1（請排版將下列狀態轉換製成流程圖）。

**圖 16-1　票種狀態的轉換流程**

```text
尚未開賣 → 販售中 → 售完 → 候補中 → 關閉
        │
        └→ 截止時間到 → 關閉
```

報名狀態則是另一條線，如圖 16-2（請排版將下列狀態轉換製成流程圖）。

**圖 16-2　報名狀態的轉換流程**

```text
待付款 → 已付款 → 已確認 → 已報到 → 已完成
   │         │

```

最常見的錯誤是只剩一個總名額、沒拆到票種，導致「總名額還沒滿、早鳥卻已售完」這種無法對使用者解釋的狀態。

### 通知與個資同意

活動網站要把通知當成系統的一部分，而非最後人工補寄。基本通知至少六種，如表 16-4。

**表 16-4　活動報名網站的基本通知**

| 通知 | 觸發 | 內容 |
|---|---|---|
| 報名成功 | 付款完成 | 活動資訊、票種、報到方式 |
| 付款提醒 | 送出報名但未付款 | 付款期限、取消規則 |
| 活動前提醒 | 活動前 1 到 3 天 | 時間、地點、交通、攜帶物 |
| 候補通知 | 名額釋出 | 回覆期限、付款連結 |
| 異動通知 | 時間、地點、講師變更 | 變更原因、退票選項 |
| 課後資料 | 活動結束後 | 簡報、錄影、社群連結 |

個資同意不能藏在一行小字裡，報名表應把用途拆開：

- 活動報名與聯絡：必要。
- 活動前後通知：必要。
- 主辦方後續課程資訊：選填。
- 合作夥伴行銷：不得預設勾選，且要清楚列出對象或用途。

### 票務系統的後端判斷

真正難的是「名額、付款、候補與通知」四件事要同步，一件慢半拍主辦方就要在活動前一天人工補洞。交易邏輯可拆成六個判斷點，如表 16-5。

**表 16-5　票務系統的後端判斷點**

| 判斷點 | 系統要回答的問題 | 技術重點 |
|---|---|---|
| 票種庫存 | 這張票還能不能賣？ | 不只看總名額，要看票種名額、場次容量與保留名額 |
| 待付款保留 | 報名送出但未付款時，要保留多久？ | 建立付款期限，逾時自動取消並釋放名額 |
| 付款確認 | 誰能把報名狀態改成已付款？ | 正式系統應由金流通知或後端確認，不只靠前端按鈕 |
| 候補遞補 | 名額釋出後通知誰？ | 依候補順位建立保留期限，不可全部同時通知 |
| 通知紀錄 | 信有沒有發出去？ | 每次 Email、簡訊或 LINE 通知都要留下狀態 |
| 退款與取消 | 取消後票種、付款與通知如何回復？ | 報名狀態、付款狀態、票種庫存必須一起更新 |

名額管理不是前端把「剩餘名額」減一——正式系統要避免兩人同時買最後一張票，也要避免未付款的人占住名額。

## 5. AI 協作實作

本節依「提示詞 → 產出結果 → 檢查修改 → 保存部署」四個小節進行。

### 5.1 本講提示詞

可以用下面這份提示詞作為第一版：

```text
你是網頁設計老師與前端工程師。
幫我做成一個「活動報名網站」的單頁網站原型，建成一個可以直接打開的單一 index.html。

【網站目標】
活動資訊、票種、名額、報名表、通知與候補規則

【使用者情境】
使用者第一次進入網站時，需要在手機上快速判斷這個服務是否適合自己，並能完成主要行動：讓參與者理解活動資訊、票種、名額、付款與通知流程。

【必要區塊】
1. Hero 首頁：名稱、定位、主張、主要行動按鈕
2. 信任證據：真實照片、資歷、評論、規則或案例
3. 活動資訊：日期、地點、議程、講者、適合對象
4. 票種區塊：早鳥、一般、學生或 VIP 票種，顯示價格、名額、開賣與截止時間
5. 報名流程：待付款、已付款、已確認、付款逾時、候補中
6. 候補規則：名額釋出後保留多久，逾時如何往下一位遞補
7. 通知說明：報名成功、付款提醒、活動前提醒、異動通知
8. 資料表單：姓名、聯絡方式、票種、個資同意、備註
9. 規則說明：取消、退款、名額保留與隱私規則

【資料欄位】
請在頁面中示範下列資料結構：
1. Event：活動名稱、日期、地點、主辦單位、活動狀態
2. EventSession：場次、開始時間、結束時間、容量
3. TicketType：票種名稱、價格、開賣時間、截止時間、名額
4. Registration：報名人、票種、付款狀態、報到狀態
5. PaymentRecord：金額、付款方式、付款結果、交易時間
6. WaitlistEntry：候補順位、通知時間、保留期限
7. NotificationLog：對象、內容、通道、發送狀態
8. ConsentRecord：個資用途、同意時間、撤回方式

【互動示意】
請用少量 JavaScript 示範：選擇不同票種會改變價格與剩餘名額；送出報名後先進入待付款；可以切換付款成功、付款逾時、取消與候補遞補狀態；每次狀態變更都要顯示對應通知。

【設計要求】
1. 用清楚的 HTML（HyperText Markup Language，超文字標記語言）結構與 CSS（Cascading Style Sheets，階層式樣式表），建成一個可以直接打開的單一 index.html，CSS 寫在同一個檔案裡。
2. 請使用 RWD（Responsive Web Design，響應式網頁設計），手機、平板、桌機都要可讀。
3. 請避免假評價、假資歷、假價格或過度承諾。
4. 表單與按鈕要有清楚的互動狀態，可以用靜態示意，不需要真的送出到伺服器。
5. 畫面要清楚標示這是活動報名流程原型，尚未接正式金流、通知服務或後台資料庫。
6. 
```

這一版產出的內容可以整理成一份清單：

- 一個 `index.html`，HTML、CSS 與少量 JavaScript 寫在同一個檔案內，`<title>` 與 meta description 放入了活動名稱、票種與報名資訊，適合搜尋引擎理解。
- `header`：活動名稱、主辦單位與導覽。
- `main` 內依序是主視覺、信任證據、活動資訊、票種與名額、報名流程（含狀態示範面板與通知紀錄）、候補規則、通知說明、報名表單、規則說明九個 `section`。
- `footer`：主辦單位、Email、活動地區與原型聲明。

整體結構就是表 16-1 的需求模組落到程式碼後的樣子，骨架示意如下（九個 `section` 依序排在 `<main>` 內，此處只列首尾，`…` 代表略去內容）：

```html
<header>…活動名稱、主辦單位、導覽…</header>
<main>
  <section class="hero">…名稱、主張、查看票種、立即報名…</section>
  …（trust / info / tickets / flow / waitlist / notify / signup / rules）…
  <section id="rules">…取消、退款、名額保留與隱私…</section>
</main>
<footer>…主辦單位、Email、原型聲明…</footer>
```

其中表 16-1 的「核心流程」拆成活動資訊、票種、報名流程、候補與通知五個 `section`；「聯絡與後續」沒有獨立區塊，Email 放頁尾、確認訊息放狀態列，屬可接受的合併。

提示詞的【互動示意】也有落地：票種卡片與表單選單由同一組 `TicketType` 渲染，每送出一筆報名名額減一並進入「待付款」，售完改候補；示範面板可切換付款成功、逾時、取消與候補遞補，每次變更都在「通知紀錄（NotificationLog）」留一筆。付款只有「模擬付款成功」一顆按鈕，正式系統的付款確認要由金流通知或後端判斷（表 16-5 的「付款確認」），原型只示意狀態。

手機版的關鍵是 CSS 的 media query：平板（`max-width: 1024px`）改兩欄，手機（`max-width: 640px`）全部改單欄、`header` 直向排列、按鈕放大成整排：

```css
@media (max-width: 640px) {
  .ticket-grid, .trust-grid, .form-grid { grid-template-columns: 1fr; }
  .btn, .sim-btn { display: block; width: 100%; text-align: center; }
}
```

### 5.3 結果檢查與修改

AI 產出後不要只看第一眼漂不漂亮，請依序檢查：

- 首頁第一屏說清楚服務適合誰、下一步是什麼。
- 資料欄位足以支撐核心流程，而非只留姓名電話。
- 票種有價格、名額、開賣與截止時間。
- 售完後進入候補，而非繼續顯示可報名。
- 付款逾時、取消、退款與報到狀態分得清楚。
- 通知含報名成功、付款提醒、活動前提醒與異動通知。
- 個資同意把必要通知與行銷授權分開。
- 價格、限制、取消或隱私規則清楚。
- 手機版能完成主要行動，且文字與按鈕不重疊。

### 6.1 真實部署前夜檢查

活動網站要從課堂原型走到正式售票，至少要補上表 16-6 列出的能力。

**表 16-6　真實部署前夜的檢查點**

| 檢查點 | 需要確認的內容 |
|---|---|
| 票種庫存 | 每個票種、場次、保留席次是否獨立計算 |
| 付款期限 | 待付款報名是否有到期時間與自動取消規則 |
| 付款確認 | 付款狀態是否由後端或金流通知確認，而不是只靠前端 |
| 候補遞補 | 名額釋出時是否依順位通知，並給每位候補者獨立期限 |
| 通知服務 | Email、簡訊、LINE 是否有發送紀錄與失敗重送 |
| 報到資料 | 現場報到名單是否能匯出、核銷與避免重複報到 |
| 個資同意 | 必要通知、活動後資料、行銷授權是否分開取得 |
| 異動處理 | 改期、取消、講者更換、退款是否有一致流程 |

面授老師可用這張表提醒學生：活動報名不是表單作業，而是容量管理；只要收錢或保留名額，就進入交易型網站的責任範圍。

### 6.2 重要名詞

**階梯定價**（Tiered Pricing）：依決策時點設計不同價格層級（早鳥、一般、倒數），讓每位買家在意願最高時以對應價格成交。

**承諾一致性 / 稀缺性 / 社會證明**：席亞爾迪尼《影響力》三原則（見 3.4）。

**一對多通知**：主辦方同時向多名報名者發送狀態更新，Google 試算表搭配表單自動化即可做出基礎版。

**不可抗力條款誤用**：以「不可抗力」（Force Majeure）掩蓋自行變更內容並拒退款，通常不符消費者保護法（見 3.5）。

**候補保留期限**：候補名額釋出後保留給特定候補者確認的時間，期限過後未回覆才往下一位釋出。

**通知紀錄**：記錄何時、用什麼通道、寄給誰、是否送達，是活動異動時能否說明責任的依據。

### 6.3 自我評量

**第一題（是非）**：報名工作坊的人主要動機是「獲取知識」，因此活動頁最重要的是詳細課程大綱。

**第二題（是非）**：早鳥票核心功能之一，是讓主辦方四個月前預收啟動資金、支付廣告與廠商訂金。

**第三題（選擇）**：根據承諾一致性原則，早鳥票除了現金流還能達到什麼效果？
（甲）更早確認活動規模　（乙）讓參加者活動前就建立「我是會去這種活動的人」的身份認同　（丙）減少退票法律風險　（丁）提高贊助商談判籌碼

**第四題（情境）**：林欣瑜看到「終於搞懂品牌策略，推薦！」就離開了。根據社會證明，這則評論缺少了什麼？怎樣的評論能讓她留下來？

**第五題（選擇）**：下列哪種退票規則最能降低觀望者的報名顧慮？
（甲）不予退款　（乙）三日前可退 50%　（丙）七日前全額退、三至七日退 70%、三日內不退　（丁）特殊情況申請退款

**第六題（是非）**：一場設計完整的付費活動，主要收入通常是門票，贊助商只佔小部分。

**第七題（情境）**：廠商建議在報名表單加入「同意個資提供給合作夥伴」勾選項以換三個贊助商。根據信任底線，說明法律風險與品牌傷害，並提一個替代方案。
