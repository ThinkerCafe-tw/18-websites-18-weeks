# 第 18 講：總複習

會做出一頁網站，跟接得了一個案子，中間隔著「說清楚需求、報價、交付、驗收」這四道沒人教的關。本講就用 AI（Artificial Intelligence，人工智慧）協作把前十七講的網站類型與功能模組整理成原型，讓你從「做出一頁網站」推進到一套帶得走的接案流程。

本講完成後，你會得到四個成果：

- 一份十七種網站類型的功能地圖。
- 一份需求訪談與報價邊界清單。
- 一份交付文件、驗收項目與維護規則。
- 一份單頁、功能型、平台型網站差異的總表。

本講學習重點：

◆ 回顧前十七講揭露的共通人性結構，找出橫跨所有行業的核心模式
◆ 理解老手在任何行業都能賺錢的三種底層邏輯
◆ 掌握本書反覆出現的四個心理學原理，知道它們為什麼成立
◆ 辨識 AI（Artificial Intelligence，人工智慧）時代的護城河建立在哪裡，以及十七行業共通的五個暗門
◆ 依案子的規模與複雜度報出合理價格，並用正確的問法展開第一次客戶對話

## 1. 破冰案例

**本講情境：你現在和十八週前已經不是同一個人**

還記得第一講開頭那個場景嗎？週三下午兩點，信義區，一個背著後背包的人站在巷口滑手機。他看的不是咖啡好不好喝，他在找插座、找評論裡「帶電腦坐了一下午」這幾個字。

那個場景裡有一個貫穿全書、卻從未明說的問題：

**為什麼那個人的選店決策，跟「咖啡」幾乎沒有關係？**

現在你知道了。他買的不是咖啡，是「一段暫時的戶籍」、是「在人群中保持孤獨的安全感」。他的大腦在滑照片的五十毫秒裡，已經評估完這個空間允不允許他以自己的方式存在四個小時。你在十七個行業裡反覆看到同一件事：客人買的從來不是功能本身。

---

你這十八週建立的不只是十七個網站的技能，而是一個看行業的方式：一個行業接著一個行業地問「客人真正在害怕什麼、真正在買什麼」，慢慢長在身體裡的直覺。它讓你走進任何陌生行業都能問出正確的問題——而問出正確問題的人，比熟記工具清單卻沒問過問題的人貴得多。這一講不教新東西，是把十七個行業看到的東西整理成你帶得走的那幾樣。

## 2. 業態需求

**人性真相總表：十七個行業，同一個底層**

把這十七個場景疊在一起，你看到的不是十七個行業，是同一種人性的十七個面貌：

- **安全與孤獨**：咖啡廳賣暫時戶籍，瑜伽賣無條件接納，諮商賣「你接得住我嗎」的被承接。
- **焦慮的轉移**：診所、律師、汽車保養收的不只是服務費，是接過客人「這件事我來處理」的整包焦慮與掌控感。
- **自我形象的投射**：健身房賣「我還沒放棄自己」，攝影師賣「我值得被記錄」，餐廳賣主揪者的面子，活動報名是「我不接受平庸」的宣告。
- **代為盡責與被款待**：美容院給情緒留白，寵物美容給「寶貝被當貴賓」，民宿給朋友般的人味，家教媒合賣盡責證書，才藝教室的社群在媽媽那層，教練與顧問靠依附與陪伴。

收束成一句話：

**沒有一個行業，客人買的是功能本身。他們買的，全部都是一個情緒狀態的移動——從焦慮移向安全，從孤單移向被接納，從失控移向掌握，從平庸移向某種值得被記住的感覺。**

這一句話，是本書的核心。

---

### 2.1 共通營運邏輯

**老手跨行業都在做的三件事**

老手不說「我懂這個行業」，他說「我懂這門生意的邏輯」。賺錢的方式可歸進三個大類。

#### 第一類：把不可儲存的時間與空間，變成可優化的產品

座位、時段、桌次的共通本質是「今天沒人用就消失、無法存到明天再賣」。**老手算的不是這杯咖啡的毛利，是整個空間每個時段能讓什麼產值出去**——調控停留、離峰定價、平衡翻桌與包廂。

#### 第二類：從一次性服務，躍升為可重複銷售的知識資產

一次性服務的天花板是體力與時間（攝影師一天只能拍幾場）。老手把專業判斷打包成可重複銷售的產品——預設集、線上課程、師資班、工具包，讓一個產出同時服務一百人。**起點是把專業看清楚到能轉換成別人可用的形式。**

#### 第三類：把未來的消費承諾，轉換成當下的現金流

儲值卡、年卡、月費顧問制、療程包，本質都是**讓客人在用完服務之前先把錢給你**：換來可預測的現金流，前提是信任強到客人願意把錢放你那裡。這帶我們去下一層——**信任是可積累的資產，貨幣不是錢，是每一次的對待。**

因此本講的技術主題是「網站等級與交付責任」。前面看懂的人性結構與營運邏輯，落到接案現場，就是判斷一個網站屬於哪個功能等級、又要負起哪些交付責任。第 4 節會用功能等級總表、交付清單與驗收總清單把這個判斷落成可檢查的結構，第 5 節再讓 AI 把它做成接案工作台原型。

## 3. 網頁設計重點

### 3.1 網站風格

總複習的設計重點是把前十七講整理成可比較的地圖：不追求單一產業氛圍，而讓頁面像接案工作台，用表格、清單與篩選把各類網站放在同一張圖上（細節見 §4）。

### 3.2 建立信任

這一講的信任模組不是客戶見證，而是交付邊界本身——需求訪談表、功能範圍表、資料模型、狀態流程、測試與上線檢查，讓學生說清能交付什麼、不能承諾什麼。

### 3.3 引導行動

CTA（Call to Action，行動呼籲）是「選擇網站類型」「生成需求表」「建立驗收清單」，把全書能力收斂成可執行的接案流程。

### 3.4 全書心理學錨點

**四個全書都在用的原理**

這四個原理在前面各講已分別教過，這裡只標出名稱與來源，方便走進新行業時對照辨識。

- **原理一：五十毫秒第一印象**（Lindgaard et al., 2006, *Behaviour & Information Technology*, 25(2)）。

- **原理四：社會證明**（Cialdini, 2006, *Influence*, Revised ed.）。

### 3.5 AI 時代的信任底線與護城河

#### 你的護城河在哪裡

AI 工具不是你的護城河——你有，對手也有。護城河是對行業的深度理解：那個讓你走進陌生行業就問出正確問題、第一次對話就說出客人在害怕什麼的直覺。

本書顧問鄒老師有一句話：「AI 工具比較像各工具之間的黏著劑，如同 Python 語言一樣。」工具是骨架，連接工具的是你對問題的理解——不理解行業，網站說服不了人。

**沒有任何行業會被 AI 取代。會被取代的，是不理解行業人性的從業者。**

#### 十七行業共通的五個暗門

把整本書的高風險做法整理成五個跨行業陷阱，每個都借助一個心理弱點。

**暗門一：預付現金流的黑洞**（儲值卡、年費）— 已付讓店家鬆懈，品質滑落時又解約困難。弱點：貪優惠與懶惰。

**暗門二：專業資訊的暗箱**（術語霸凌）— 專業語言的目的若是讓你放棄追問，就從溝通變成操縱。弱點：對專業的崇拜與對自身無知的恐懼。

**暗門三：虛假稀缺與倒數計時** — 「今天最後三個名額」若明天重設，就是操弄錯失恐懼（FOMO）。弱點：稀缺被用在不誠實的前提上。

**暗門四：契約迷宮與摩擦成本** — 讓解約繁瑣到客人算帳後選擇繼續忍受。弱點：對行政程序的厭惡與「算了吧」的惰性。

**暗門五：包裝溢價與資歷造假** — 假莊園豆、假認證、假案例數字。弱點：對社會地位的追求與在陌生領域易被外表說服。

---

認識這五個暗門不是學如何用它們，是讓你做網站與文案時知道哪裡是明道、哪裡是偏路。真正的心法是理解支撐長青企業的明道——客人十年後還願意帶朋友來，是誠實累積出來的。

## 4. 功能與資料模型

**實作主題：把「接案實戰與技術地圖」落成可檢查的網站。**

前面幾層談產業、人性與風險，這一層把判斷轉成網站結構：不是請 AI「做一個漂亮網站」，而是把業態需求、頁面模組、資料欄位、流程與驗收標準說清楚，再讓 AI 產出第一版。

### 本講網站需求

本講網站型態是「接案檢查與交付規格」，核心行動是整理網站類型、功能等級、報價邊界與交付清單，先用表 18-1 拆需求：

**表 18-1　總複習網站的模組需求**

| 模組 | 功能 | 必要內容 |
|---|---|---|
| 網站類型總表 | 整理全書範圍 | 單頁、預約、會員、平台、金流 |
| 需求訪談 | 避免範圍失控 | 業態、客群、核心行動、資料來源 |
| 報價拆分 | 讓價格有依據 | 頁面數、功能數、整合項目、維護 |
| 交付清單 | 降低爭議 | 原始檔、部署網址、帳號權限、文件 |
| 驗收流程 | 確認可用 | 手機版、表單、資料、付款、SEO（Search Engine Optimization，搜尋引擎最佳化） |
| 維護規則 | 讓網站能活下去 | 更新頻率、責任人、費用、回復流程 |

### 頁面與功能模組

把網站想成一組可檢查的模組，每個回答一個使用者問題：我在哪裡？這適合我嗎？我相不相信？下一步做什麼？最小可行版本可以是單頁，但單頁不代表沒有結構：導覽列能跳到主要區塊，行動按鈕在首頁第一屏與流程結尾，表單送出後有確認訊息。

至少要先拆出表 18-2 的資料物件。第一版不一定要真的接資料庫，但你必須知道網站背後需要哪些資料，AI 才不會只產出表面文案。

**表 18-2　總複習網站的資料物件**

| 資料物件 | 欄位 | 範例或用途 |

| QuoteLine | 工作項目、估時、價格、備註 | 用於頁面內容、表單或狀態檢查 |

### 全書網站類型與功能模組總表

走完前十七講，你要能把網站分成幾種功能等級，而非只用「幾頁」報價，如表 18-3。

**表 18-3　全書網站類型與功能等級總表**

| 等級 | 網站類型 | 代表講次 | 核心功能 | 主要風險 |
|---|---|---|---|---|
| 等級 1 | 單頁形象網站 | 1 咖啡廳、2 健身房、6 法律、9 寵物、10 教練 | 介紹、信任、CTA、聯絡 | 文案不真實、手機版難用 |
| 等級 2 | 作品與部署網站 | 3 攝影師 | 作品分類、GitHub（程式碼與專案託管平台）、Vercel（前端部署平台）、SEO、RWD | 圖片路徑錯、部署後壞掉 |
| 等級 3 | 預約與排程網站 | 4 診所、7 餐廳、13 汽車保養、15 諮商、17 顧問 | 時段、名額、取消、提醒 | 超收、隱私、付款狀態錯 |
| 等級 4 | 會員與課程網站 | 5 美容、8 瑜伽、14 才藝、16 活動 | 會員、點數、報名、候補、通知 | 點數帳錯、候補混亂、通知漏寄 |
| 等級 5 | 交易與平台網站 | 11 民宿、12 家教媒合、17 顧問 | 訂單、付款、媒合、Webhook（外部服務事件通知）、後台 | 金流錯誤、資料不一致、權限不足 |

同一個網站可以跨等級：顧問網站可以只是等級 3 預約頁，也可升級成等級 5 金流閉環。報價差異不是頁面變多，是責任變重。

### 等級 5 交易型網站的技術責任

交易型網站最容易被低估，畫面上可能只多一顆「付款」按鈕，技術責任卻多出一整組系統邏輯。可把等級 5 拆成七個責任面，如表 18-4。

**表 18-4　等級 5 交易型網站的七個責任面**

| 責任面 | 代表問題 | 對應講次 |
|---|---|---|
| 庫存或容量 | 房間、票種、時段是否還能賣？ | 第 11 講民宿、第 16 講活動 |
| 訂單狀態 | 待付款、已付款、取消、退款是否分清楚？ | 第 11、17 講 |
| 金流確認 | 成功頁、失敗頁、webhook 誰才是主信源？ | 第 11、17 講 |
| 後台權限 | 誰可以改價、鎖日期、看訂單、匯出名單？ | 第 11、18 講 |
| 通知紀錄 | 付款提醒、報名成功、候補通知是否有發送狀態？ | 第 16 講 |
| 交付責任 | 付款後交付什麼，何時交付，誰負責？ | 第 17 講 |
| 維護與追查 | 出錯時能不能查到是哪一筆資料、哪一次通知、哪個人操作？ | 第 18 講 |

這是給面授老師的技術抓手：不要只問「網站好不好看」，要問「狀態由誰維護？」——能問出這問題，學生就從 AI 生成頁面跨到網頁系統設計。

### 真實部署前夜清單

客戶說「我明天就要上線收錢」，不能只回「可以」，要先確認表 18-5：

**表 18-5　真實部署前夜清單**

| 類別 | 上線前要確認 |
|---|---|
| 前端 | 手機版、表單驗證、錯誤狀態、成功/失敗/取消頁是否完整 |
| 後端 | 是否有 API 建立訂單、查詢狀態、取消或退款 |
| 資料庫 | 訂單、付款、庫存、通知、操作紀錄是否能保存 |
| 金流 | 是否使用商家自己的帳號，sandbox 是否測過，正式環境是否分開 |
| Webhook | 是否驗證簽章、金額、交易編號，是否能處理重送 |
| 權限 | 後台登入、角色權限、操作紀錄是否存在 |
| 個資 | 收集欄位是否必要，匯出與保存是否有邊界 |
| 維運 | 誰負責錯誤排查、備份、回復、金流對帳與客訴 |

這張清單不是要學生都成為全端工程師，而是讓他們接案時知道自己站在哪個責任層級：最危險的不是做不出頁面，而是太快做出看似可上線、實際責任沒被理解的網站。可用表 18-6 做需求訪談：

**表 18-6　需求訪談問題與目的**

| 問題 | 為什麼要問 |
|---|---|
| 網站只要展示，還是要處理資料？ | 決定是否需要資料模型 |
| 使用者會不會預約、報名、付款或登入？ | 決定是否有狀態流程 |
| 是否有名額、容量、房況或時段限制？ | 決定是否有超收風險 |
| 是否會收個資、健康資料或付款資料？ | 決定隱私與合規邊界 |
| 是否要由客戶自己更新內容？ | 決定是否需要後台 |
| 出錯時誰負責修？多久內修？ | 決定維護合約與費用 |

### 報價邊界與交付清單

接案時最危險的不是少收錢，是沒說清「包含什麼、不包含什麼」。把交付拆成七類，如表 18-7。

**表 18-7　交付類別、內容與驗收方式**

| 類別 | 交付內容 | 驗收方式 |
|---|---|---|
| 內容 | 頁面文案、圖片、服務說明 | 客戶逐頁確認 |
| 前端 | HTML（HyperText Markup Language，超文字標記語言）/CSS（Cascading Style Sheets，階層式樣式表）/JS、RWD、互動狀態 | 手機、平板、桌機測試 |
| 部署 | GitHub repository、Vercel 網址、自訂網域 | 公開連結可開啟 |
| 資料 | 表單欄位、資料模型、匯出格式 | 測試送出與資料檢查 |
| 整合 | 金流、通知、第三方 API、webhook | 測試環境跑通並列出正式環境缺口 |
| 後台 | 登入、角色權限、訂單或名單管理 | 測試不同角色是否只能做被允許的事 |
| 維護 | 更新頻率、錯誤回報、備份與回復 | 維護條款與 SLA |

報價可以用三個句型保護自己：

```text
本報價包含：首頁、服務頁、表單、手機版、一次部署。
本報價不包含：會員登入、金流串接、後台管理、第三方 API（Application Programming Interface，應用程式介面）費用。
若後續新增功能，將依新增資料模型、流程狀態與測試範圍另行估價。
```

這些話不是防客戶，是讓合作一開始就有邊界。

### 全書驗收總清單

每個網站交出去之前，都應至少跑過表 18-8：

**表 18-8　全書驗收總清單**

| 類別 | 檢查項目 |
|---|---|
| 內容真實 | 沒有假評論、假證照、假案例、假媒體報導 |

## 5. AI 協作實作

本講的 AI 協作分兩層：先讓 AI 產出接案工作台原型，再檢查、必要時請 AI 補強。本節依「提示詞 → 產出 → 檢查 → 部署」進行。

### 5.1 本講提示詞

可以用下面這份提示詞當第一版：

```text
你是網頁設計老師與前端工程師。
幫我建一個「總複習」的單頁網站原型。

【網站目標】
全書技術地圖、接案報價、交付清單、驗收表與維護規則

【使用者情境】
使用者第一次進入網站時，需要在手機上快速判斷這個服務是否適合自己，並能完成主要行動：整理全書網站類型、功能等級、報價邊界與交付清單。

【必要區塊】
1. 網站類型總表：單頁、預約、會員、平台、金流
2. 需求訪談：業態、客群、核心行動、資料來源
3. 報價拆分：頁面數、功能數、整合項目、維護
4. 交付清單：原始檔、部署網址、帳號權限、文件
5. 驗收流程：手機版、表單、資料、付款、SEO
6. 維護規則：更新頻率、責任人、費用、回復流程

【資料欄位】
請在頁面中示範下列資料結構：
1. WebsiteType：名稱、功能等級、適合業態、風險
2. ScopeItem：項目、是否包含、驗收方式
3. QuoteLine：工作項目、估時、價格、備註
4. DeliveryAsset：檔案、帳號、網址、文件
5. AcceptanceCheck：檢查項目、結果、修正方式
6. MaintenanceRule：更新頻率、責任人、費用

【設計要求】
1. 用清楚的 HTML 結構與 CSS，建成一個可以直接打開的單一 index.html（CSS 寫在同一個檔案裡）。
2. 請使用 RWD（Responsive Web Design，響應式網頁設計），手機、平板、桌機都要可讀。
3. 請避免假評價、假資歷、假價格或過度承諾。
4. 表單與按鈕要有清楚的互動狀態，可以用靜態示意，不需要真的送出到伺服器。
5. 
```

手機版的關鍵是 CSS 的 media query，AI 產出的規則節錄如下：

```css
/* 平板：類型卡片與維護卡片改為兩欄 */
@media (max-width: 1024px) {
  .type-grid, .rule-grid { grid-template-columns: repeat(2, 1fr); }
  .hero-grid { grid-template-columns: 1fr; }
}

/* 手機：全部單欄，header 改為直向排列，按鈕放大成整排可點 */
@media (max-width: 640px) {
  .site-header { flex-direction: column; align-items: flex-start; }
  .site-nav { gap: 12px; }
  .hero h1 { font-size: 1.5rem; }
  .type-grid, .rule-grid, .form-grid, .delivery-grid { grid-template-columns: 1fr; }
  .field-full { grid-column: auto; }
  .btn { display: block; width: 100%; text-align: center; }
  .hero-actions { flex-direction: column; }
}
```

### 5.3 結果檢查與修改

AI 產出後，不要只看第一眼漂不漂亮，要依序檢查：能否分辨五型網站、能否用功能等級而非頁數說明報價差異、報價有沒有明確的包含與不包含、交付是否含原始檔與帳號權限、驗收能否找出手機版與付款風險、維護規則是否說清責任人與費用。

### 5.4 保存與部署

要上線可以先用 GitHub 保存檔案，再用 Vercel 或 GitHub Pages 部署。GitHub 讓每次修改都有紀錄；部署則讓你用真實手機連結檢查網站是否可用。這條流程正式主場在第 3 講，本講把它收束成交付清單與責任分工，連同表 18-7、18-8 一併交代給客戶。

本講最重要的練習，是把「產業洞察」翻譯成「可檢查的網站需求」。當你能說清資料、流程、信任與風險，AI 才會從模板生成器變成真正的協作助理。

## 6. 上線與延伸

### 6.1 重要名詞

**損失規避**（Loss Aversion）：見 §3.4 原理三。

**社會證明**（Social Proof）：見 §3.4 原理四。

**信任三支柱**：見 §3.4 原理二（資歷、善意、誠信）。

**服務業的不可儲存性**：時間與空間不能庫存，未用容量永久消失，是定價與排程的根本約束（見 §2.1）。

**接案報價框架**：以「解決什麼問題」而非「有幾頁」為定價起點，依功能與整合難度分級。

**功能等級**：用網站是否需要資料、狀態、權限、付款、後台分級，比頁面數更反映交付難度（見表 18-3）。

**交付清單**：專案結束交給客戶的文件與資產（原始檔、部署網址、帳號權限、操作說明、維護規則）。

**驗收標準**：雙方事前同意的通過條件；沒有它，專案容易變成無止境修改。

### 6.2 自我評量

**第一題（是非）**：本書的核心主張是：只要有夠好的 AI 工具，任何人都能做出有效的行業網站，不需要理解行業本身。

**第二題（選擇）**：根據 Tversky 與 Kahneman 的損失規避研究，以下哪種文案對猶豫中的客戶更有說服力？
（甲）「定期保養，讓你的車跑得更順」
（乙）「缺乏保養的車，平均在六萬公里後出現的故障修繕費，是正常保養費用的三到五倍」
（丙）「我們提供最專業的汽車保養服務」
（丁）「六百位客戶選擇信任我們」

**第三題（情境）**：剛開始接案的你，接到餐廳老闆的詢問：「我想要一個網站，大概要多少錢？」根據本講建議，你第一個應該問什麼？請寫出你會怎麼問，並說明為什麼。

**第四題（是非）**：合約中「包含什麼」的清單，比「不包含什麼」的清單，在防止接案糾紛上更重要。

**第五題（情境）**：才藝教室老闆說：「我想要一個很酷很特別的網站，你先做，之後有什麼再說。」根據本講建議，你應該如何回應？請寫出你的處理策略。

**第六題（選擇）**：以下哪種商業模式最直接體現了「把未來消費承諾轉換成當下現金流」的邏輯？
（甲）按件計費的攝影接案
（乙）每次結帳的咖啡廳消費
（丙）美容院預購年度療程方案
（丁）餐廳按人數收費的包廂預訂

**第七題（申論）**：回想本書十七個行業，選三個你印象最深的「人性真相金句」，說明為什麼，以及它們有沒有讓你在日常生活裡，用不同的眼光看待某一個消費場景。
