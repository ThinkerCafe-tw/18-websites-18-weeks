# 第 01 講：咖啡廳網站

客人滑到一家咖啡廳之前，最難回答的一關，是這家店漂不漂亮看得出來，適不適合自己卻看不出來。本講就用 AI（Artificial Intelligence，人工智慧）協作做出一個小型咖啡廳的單頁網站，讓客人在手機上一眼判斷「這家店適不適合我」，再順手查看菜單、確認位置、打電話或導航前往。

本講完成後，你會得到三個成果：

- 一份咖啡廳網站需求摘要。
- 一個可以在瀏覽器開啟的 `index.html`。
- 一份上線前的手機版與轉換路徑檢查清單。

**開始之前：什麼是 Vibe Coding**

本書教的是 Vibe Coding：你用自然語言把需求講清楚，讓 AI 工具直接幫你把網站做出來，而不是自己一行一行寫程式。你不需要先學會所有語法，而是先學會把「要做什麼」說清楚。書裡每一講都會給你一段可以直接使用的提示詞，你可以選任何一款 Vibe Coding 工具（例如 Codex、Claude Code、Gemini 等），把提示詞交給它，它會直接幫你建出網站檔案。書裡的提示詞在哪一款工具上都通用；本書不綁定特定工具，重點是「怎麼把需求說清楚」這件事，換哪一款都帶得走。

## 1. 破冰案例

週三下午兩點，一個自由工作者站在信義區巷口滑手機。

他的包裡有筆電、充電器和一份還沒寫完的提案。他不是想找全台最好喝的咖啡，也不是想拍照打卡。他只是想找一個地方坐三個小時，把事情做完。

他打開 Google Maps，搜尋「咖啡廳 信義區」。第一家照片很好看，甜點擺得很精緻，評論也很多，但留言裡都是「很適合拍照」「姐妹聚會很棒」。他滑了幾秒就退回搜尋結果。

第二家照片比較普通，可是第一張就拍到長桌、插座和窗邊座位。評論裡有人寫：「帶電腦坐了一下午，Wi-Fi 穩，店員沒有催位。」他看了營業時間，按下導航，走過去。

這個人做決定的過程很短，但裡面藏著咖啡廳網站最重要的任務。

他不是先問「咖啡豆產地是哪裡」。他先問的是：

- 我能不能在這裡安心坐下來？
- 這裡會不會太吵？
- 有沒有插座和 Wi-Fi？
- 我進去之後會不會覺得自己打擾別人？
- 價格、位置、營業時間是不是清楚？

咖啡廳網站的第一個任務，不是把店裝成最漂亮的樣子，而是替客人降低進門前的不確定感。

如果客人想工作，網站要讓他看見桌面、插座、安靜感和不限時規則。  
如果客人想約會，網站要讓他看見燈光、座位距離、甜點和氛圍。  
如果客人想買豆子，網站要讓他看見豆種、烘焙程度、風味描述和購買方式。

同一家咖啡廳，可以有很多種客人。但網站不能同時對所有人講一樣的話。這一講要練習的，就是先判斷咖啡廳真正想服務誰，再用 AI 把這個判斷變成一個可上線的網站。

## 2. 業態需求

咖啡廳網站要解決的不是「有沒有網頁」這件事，而是四個更具體的問題。

第一，讓客人找得到。

客人常常不是直接搜尋店名，而是搜尋「附近咖啡廳」「台中東區咖啡廳」「可以工作的咖啡廳」「不限時咖啡廳」。如果網站裡有清楚的地點、服務特色、菜單、營業時間與店家介紹，就比較容易被搜尋引擎理解，也比較容易和 Google 商家檔案互相補強。

第二，讓客人判斷得快。

手機上的耐心很短。客人點進網站後，前幾秒通常只想知道三件事：這家店賣什麼、在哪裡、適不適合我。首頁如果一開始就塞滿品牌故事，反而會讓人找不到決策資訊。

第三，讓客人信任。

小型咖啡廳常見的問題不是沒有特色，而是特色沒有被說清楚。自家烘焙、友善工作、寵物友善、甜點現做、晚上供應精釀，這些都需要變成具體證據。網站要提供照片、菜單、規則、聯絡方式、真實地址和一致的品牌語氣，讓客人相信這不是臨時拼出來的頁面。

第四，讓客人行動。

咖啡廳網站常見的行動不一定是「線上付款」。更常見的是：

- 查看菜單。
- 打電話詢問。
- 開啟 Google Maps 導航。
- 加入 LINE 或 Instagram。
- 預約座位或包場。
- 購買咖啡豆、掛耳包或課程。

所以咖啡廳網站不是純形象頁，它應該是「從搜尋到到店」的轉換橋樑。

因此本講的技術主題是「HTML 結構與第一次上線」。這個主題會在第 4 節先拆成頁面區塊與資料物件，在第 5 節由 AI 產出可直接開啟的單一 `index.html`，最後在第 6 節上傳 GitHub 並用 Vercel 部署，取得可公開分享的網址。

本講採用的基本業態假設整理如表 1-1。

**表 1-1　本講的基本業態假設**

| 項目 | 本講設定 |
|---|---|
| 店型 | 小型獨立咖啡廳 |
| 主要客群 | 附近上班族、自由工作者、週末休閒客 |
| 網站型態 | 單頁式形象網站 |
| 核心行動 | 查看菜單、確認空間、導航前往、聯絡店家 |
| 技術目標 | 用 AI 產出可直接開啟的 HTML（HyperText Markup Language，超文字標記語言）網站，並完成手機版檢查 |

這個設定故意不做太複雜。第一講的重點是跑完一次完整流程：需求整理、AI 產出、修改、檢查、上線。等這條流程熟了，後面的診所預約、餐廳訂位、會員系統和顧問預約，才有共同的工作基礎。

## 3. 網頁設計重點

### 3.1 網站風格

咖啡廳網站的風格，不是先問「要不要好看」，而是先問「要讓哪一種客人覺得這裡適合他」。

如果目標客人是想帶筆電工作的上班族，網站風格應該安靜、乾淨、資訊清楚。照片要優先展示座位深度、桌面大小、插座位置、自然光和人與人之間的距離。用色可以偏木質、米白、深灰或低飽和綠色，讓畫面有穩定感。

如果目標客人是喜歡甜點與拍照的年輕族群，網站可以明亮一點，強調甜點櫃、季節限定、店內角落、拍照光線和社群連結。但即使走打卡風，也要避免讓資訊變得難找。

如果目標客人是精品咖啡愛好者，網站要把吧台、手沖器具、豆子來源、烘焙曲線、風味描述放在更前面。這類客人不只看氣氛，也看專業證據。

可以用表 1-2 決定第一版視覺方向。

**表 1-2　目標客人與第一版視覺方向**

| 目標客人 | 視覺關鍵 | 首頁應優先出現 |
|---|---|---|
| 想安靜工作的人 | 沉靜、穩定、留白、桌面感 | 插座、長桌、Wi-Fi、不限時規則 |
| 想約會聊天的人 | 溫暖、放鬆、柔光、距離感 | 雙人座、甜點、燈光、晚間氛圍 |
| 精品咖啡客 | 專業、職人、透明、細節 | 豆子來源、手沖、烘豆、風味輪 |
| 社群打卡客 | 明亮、鮮明、可分享 | 甜點、店景、季節活動、IG 入口 |

這就是「視覺語言」：不是單純選顏色，而是用照片、排版、字體、留白和語氣，告訴客人「這家店是哪一種空間」。

### 3.2 建立信任

咖啡廳網站的信任模組要盡量具體。模糊的句子，例如「環境舒適」「咖啡好喝」「歡迎光臨」，幫助有限。客人真正需要的是可以驗證的資訊。

基本信任模組至少包含七個，整理如表 1-3。

**表 1-3　咖啡廳網站的基本信任模組**

| 模組 | 內容 | 為什麼重要 |
|---|---|---|
| 店家基本資訊 | 店名、地址、電話、營業時間、公休日 | 降低到店前的不確定 |
| 菜單與價格 | 品項、描述、價格區間 | 避免客人擔心消費落差 |
| 空間設備 | 插座、Wi-Fi、座位數、是否限時 | 幫助客人判斷是否適合停留 |
| 真實照片 | 門面、座位、吧台、餐點 | 比抽象形容詞更有說服力 |
| 品牌故事 | 為什麼開店、選豆或甜點特色 | 建立記憶點 |
| 評價或推薦 | 精選評論、媒體報導、顧客回饋 | 提供第三方證據 |
| 聯絡入口 | 電話、LINE、Instagram、Google Maps | 讓客人知道下一步怎麼做 |

信任模組有一個原則：不要編造。

如果沒有媒體報導，就不要寫「媒體推薦」。如果沒有冠軍咖啡師，就不要放職人頭銜。如果不是每天自家烘焙，就不要把「自家烘焙」放在首頁。AI 很容易幫你寫出漂亮但不真實的文案，這時候使用者要負責把它拉回現實。

網站可以包裝，但不能偽裝。這是本課程所有商業網站都共用的底線。

### 3.3 引導行動

咖啡廳網站的轉換路徑可以很簡單，如圖 1-1 所示（請排版將下列步驟製成橫式流程圖）。

**圖 1-1　咖啡廳網站從搜尋到到店的基本轉換路徑**

```text
看到首頁 → 判斷是否適合 → 查看菜單與位置 → 點擊導航或聯絡 → 到店
```

首頁第一屏最好回答四件事：

1. 這是什麼店？
2. 它適合誰？
3. 我現在可以做什麼？
4. 它在哪裡？

因此第一屏應該有一句清楚主張，例如：

```text
安靜工作、手沖咖啡、午後甜點，離捷運站 3 分鐘。
```

比起：

```text
在城市裡，遇見一杯生活的溫度。
```

前者比較像網站，後者比較像海報。教材中的網站要先能完成任務，再追求詩意。

咖啡廳網站常見的行動按鈕整理如表 1-4。

**表 1-4　咖啡廳網站常見的行動按鈕**

| 按鈕 | 使用情境 | 建議放置位置 |
|---|---|---|
| 查看菜單 | 客人先判斷價位與品項 | 首頁第一屏、菜單區 |
| Google Maps 導航 | 客人已接近到店決策 | 首頁第一屏、頁尾固定資訊 |
| 打電話 | 訂位、詢問座位、包場 | 手機版可點擊 |
| 加 LINE / Instagram | 持續互動、活動通知 | 聯絡區、頁尾 |
| 購買咖啡豆 | 零售延伸 | 菜單後或品牌故事後 |

手機版尤其重要。咖啡廳網站大多數訪客是在路上查資料，按鈕要夠大、地址要可點、電話要可撥、菜單不要縮到看不清楚。只要手機版難用，設計再漂亮都會失去轉換。

## 4. 功能與資料模型

第一講不做複雜系統，先做單頁式網站。但即使是單頁網站，也需要先把資料拆清楚。這會讓你跟 AI 溝通更穩定，也會讓後續修改不容易亂掉。

可以先把「單頁式網站」和「複雜網站系統」分開，如表 1-5 所示。

**表 1-5　單頁式網站與複雜網站系統的差異**

| 類型 | 這一講的理解方式 | 常見內容 |
|---|---|---|
| 單頁式網站 | 主要內容集中在同一個頁面，訪客一路往下滑就能完成判斷。它可以有導覽列、區塊、按鈕與表單，但不需要登入、資料庫、會員權限或後台狀態管理。 | 咖啡廳首頁、菜單、空間照片、位置、聯絡方式、Google Maps 導航 |
| 複雜網站系統 | 網站不只是展示資訊，還要記錄資料、判斷狀態、處理權限或串接外部服務。 | 線上訂位、購物車、會員點數、訂房房況、付款結果、店家後台 |

第一講先做單頁式網站，是因為它能讓學生在一個下午內跑完「需求整理 → AI 產出 → 修改 → 檢查 → 上線」的完整流程。後面的章節再把這條流程加深，進入訂位、會員、金流、後台與資料狀態管理。

本講建議的頁面區塊整理如表 1-6。

**表 1-6　咖啡廳網站的建議頁面區塊**

| 區塊 | 功能 | 必要內容 |
|---|---|---|
| Hero 首頁主視覺 | 讓客人快速理解店的定位 | 店名、主張、主要按鈕、代表照片 |
| 關於我們 | 讓店有記憶點 | 開店故事、咖啡或甜點特色 |
| 菜單 | 讓客人判斷價格與品項 | 分類、品名、描述、價格 |
| 空間與設備 | 讓客人判斷是否適合停留 | 插座、Wi-Fi、座位、限時規則 |
| 營業資訊 | 讓客人能到店 | 地址、電話、營業時間、公休日 |
| 聯絡與行動 | 讓客人完成下一步 | 導航、電話、LINE、Instagram |

可以把咖啡廳網站想成幾個資料物件，整理如表 1-7。

**表 1-7　咖啡廳網站的資料物件**

| 資料物件 | 欄位 | 範例 |
|---|---|---|
| `CafeProfile` | 店名、標語、地區、品牌描述 | 山風咖啡、台中東區、自家烘焙 |
| `MenuItem` | 分類、品名、描述、價格 | 手沖咖啡、阿里山水洗、180 |
| `SpaceFeature` | 設備名稱、說明、是否提供 | 插座、每個長桌座位皆有 |
| `BusinessHour` | 星期、開始時間、結束時間、備註 | 週一至週日 11:00-20:00、週二公休 |
| `ContactLink` | 類型、顯示文字、連結 | Google Maps、電話、Instagram |
| `TrustItem` | 類型、內容、來源 | 顧客評論、媒體報導、店家承諾 |

這些資料物件不一定要真的寫成資料庫。第一講只需要把它們整理成一份清單，貼給 AI。AI 收到的資料越結構化，產出的網站越不容易漏東漏西。

本講最小可行版本的功能需求如下：

- 首頁有店名、一句話定位與主要行動按鈕。
- 菜單至少分成咖啡、非咖啡、甜點或輕食三類。
- 每個菜單品項有名稱、簡短描述與價格。
- 空間設備清楚列出插座、Wi-Fi、座位與限時規則。
- 地址可以點擊開啟 Google Maps。
- 電話在手機上可以直接撥打。
- 頁面在手機寬度下不會文字重疊、按鈕過小或圖片壓縮變形。

若店家已經有線上訂位、購物車、會員點數或後台管理，先不要放進第一講。那些功能會在後面更適合的章節處理。第一講的核心是：一個下午內做出能被客人使用的網站。

## 5. AI 協作實作

AI 協作的重點不是一句「幫我做咖啡廳網站」。那樣通常會得到漂亮但空泛的模板。

### 5.1 本講提示詞

比較好的做法是先整理資料，再要求 AI 依照資料產出。你可以把下面這份提示詞當作第一版範本。

```text
你是協助小型店家上線的網頁設計顧問與前端工程師。
請幫我製作一個咖啡廳單頁式網站。

【店家資料】
店名：山風咖啡
地點：台中東區，離火車站步行約 8 分鐘
店型：小型獨立咖啡廳，自家烘焙，提供手沖、義式咖啡、甜點
主要客群：附近上班族、自由工作者、週末想安靜聊天的客人
特色：
1. 平日下午不限時
2. 長桌座位有插座
3. Wi-Fi 穩定
4. 使用阿里山與台南小農合作豆
5. 週末提供季節甜點

【網站目標】
讓第一次來的客人能在手機上快速知道：
1. 這家店適不適合安靜工作或聊天
2. 菜單與價格大約是多少
3. 地址、營業時間、電話與導航方式
4. 如何聯絡或追蹤最新消息

【必備區塊】
1. 首頁主視覺：店名、一句話定位、兩個按鈕「查看菜單」「導航前往」
2. 關於我們：三到五句品牌故事，不要太浮誇
3. 菜單：咖啡、非咖啡、甜點三類，每類三項，附價格
4. 空間設備：插座、Wi-Fi、座位、是否限時
5. 營業資訊：地址、電話、營業時間、公休日、Google Maps 連結
6. 聯絡：Instagram、LINE 或電話

【視覺風格】
關鍵字：溫暖、安靜、木質、清楚、有留白
主色：深咖啡色、米白色、淺木色
避免：過度鮮豔、網美打卡風、太多動畫、看不清楚的小字

【技術要求】
1. 請做成一個可以直接打開的單一 index.html 檔案。
2. HTML、CSS（Cascading Style Sheets，階層式樣式表）、少量 JavaScript（網頁互動程式語言）可以寫在同一個檔案內。
3. 請使用響應式設計，手機、平板、桌機都要可讀。
4. 地址請做成可以點擊的 Google Maps 連結。
5. 電話請做成手機可撥打的 tel 連結。
6. 不要使用付費外部素材。
7. 不要編造顧客評論、媒體報導、獎項或不存在的認證。
8. 圖片先用免費圖庫（如 Unsplash）的示意圖，並用註解標註建議拍攝內容與日後要替換的檔名。

```

### 5.2 AI 產出結果

把這份提示詞交給你選的 Vibe Coding 工具，它就會照著山風咖啡的店家資料、把整間店的主視覺、菜單與營業資訊建成一個 `index.html`。用瀏覽器打開，你會看到類似圖 1-2 的咖啡廳首頁。

**圖 1-2　AI 依本講提示詞產出的咖啡廳網站首頁（桌機版）**

（圖檔：`fig-01-02-home-desktop.png`，隨稿附件）

這一版產出的內容可以整理成一份清單：

- 一個 `index.html`，HTML 與 CSS 寫在同一個檔案內。
- `main` 內依序是主視覺、關於我們、菜單（咖啡、非咖啡、甜點三類各三項，附價格）、空間與設備、營業資訊、聯絡六個 `section`，正好對應表 1-6 的頁面區塊。
- `header`（店名與導覽）與 `footer`（地址、電話與版權）是表 1-6 沒列、由 AI 補上的常見補充。
- 照片先以免費圖庫示意圖代替，AI 用註解標出日後要替換的檔名，例如 `images/hero.jpg`。

用手機寬度打開同一個檔案，CSS 裡的 media query 會讓菜單改成單欄，三類品項由上往下排，如圖 1-3。

### 5.3 結果檢查與修改

AI 產出後，不要立刻當成成品。請依照三輪檢查方式處理。

第一輪看大方向，確認它是不是像一家「安靜、木質、可工作」的咖啡廳。如果畫面看起來像高級餐酒館、甜點網拍或企業官網，就請 AI 重新調整視覺語言。

第二輪看資訊是否完整。檢查店名、菜單、地址、營業時間、電話、空間設備、按鈕是否都有出現。缺什麼就補什麼，不要一次要求 AI 大改全部。

第三輪看手機版。把瀏覽器縮窄到手機寬度，檢查文字是否太小、菜單是否擠在一起、按鈕是否容易點、地址與電話是否真的能點。

可以使用這些修改句：

```text
首頁現在太像高級餐廳，請改成比較適合自由工作者的安靜咖啡廳風格。
```

```text
手機版菜單卡片太擠，請改成單欄排列，並加大品名與價格的字級。
```

```text
請把「導航前往」按鈕連到 Google Maps，並把電話改成 tel 連結。
```

```text
請刪掉所有虛構評論和不存在的媒體推薦，改成「歡迎到店分享你的回饋」。
```

AI 協作開發的核心不是技術，而是判斷力。你要做的是看著畫面問：

- 這像不像這家店？
- 客人三秒內看得懂嗎？
- 資訊有沒有漏？
- 手機上能不能完成行動？
- 有沒有 AI 自己編出來的內容？

只要你能回答這些問題，就已經在做網頁設計了。

## 6. 上線與延伸

### 6.1 上線檢查清單

完成網站程式碼後，先不要急著公開。請先做本機檢查。

第一步，把 AI 產出的程式碼存成 `index.html`。

第二步，用瀏覽器打開這個檔案，從上到下瀏覽一次，確認所有區塊都有出現。

第三步，用手機版尺寸檢查。可以在 Chrome 開啟開發者工具，再切換裝置模式，模擬 iPhone 或 Android 手機。重點檢查：

- 首頁第一屏是否看得到店名、定位與主要按鈕。
- 菜單價格是否清楚、按鈕是否容易點、圖片是否變形。
- 地址是否可以開啟導航、電話是否可以直接撥打。
- 頁尾是否有完整營業資訊。

第四步，請兩個人幫你看。不要問「好不好看」，要問更具體的問題：

```text
你三秒內知道這家店在哪裡嗎？
你看得出來這家店適不適合帶筆電工作嗎？
你找得到菜單和價格嗎？
如果你現在要去，知道要按哪裡嗎？
```

如果四個問題有任何一題答不出來，網站還沒準備好上線。

上線方式可以先用最簡單的流程：

1. 建立 GitHub（程式碼與專案託管平台）帳號。
2. 建立一個新的 repository。
3. 上傳 `index.html`。
4. 到 Vercel（前端部署平台）建立新專案並連結該 repository。
5. 部署後取得網址，例如 `shanfeng-cafe.vercel.app`。
6. 將網址放回 Google 商家檔案、Instagram 個人檔案與 LINE 歡迎訊息。

部署背後的版本控制與自動部署原理，會在第 3 講正式拆解。

第一講不要求買網域，也不要求串訂位系統。先讓網站能被打開、能在手機閱讀、能引導客人到店。

本講的驗收標準如表 1-8 所示。

**表 1-8　本講驗收標準**

| 檢查項目 | 通過標準 |
|---|---|
| 內容 | 店名、菜單、地址、電話、營業時間完整 |
| 信任 | 沒有編造評論、獎項、媒體報導或不存在的特色 |
| 視覺 | 風格符合目標客群，照片與文字不互相打架 |
| 行動 | 菜單、導航、電話或社群入口清楚可點 |
| 手機版 | 文字可讀、按鈕可點、沒有重疊或變形 |
| 上線 | 網址可公開打開，並能從 Google 商家或社群入口連回 |

### 6.2 延伸功能與實作作業

後續可以延伸的功能包括：

- 自訂網域，例如 `shanfengcafe.example`。
- Google 商家檔案同步，讓搜尋結果和網站資訊一致。
- 線上訂位或候位表單。
- 咖啡豆與掛耳包購買頁。
- 最新活動或季節甜點公告。
- LINE 官方帳號連結，讓客人訂位或詢問座位。
- 簡單流量分析，觀察客人從哪裡進站、最常點哪個按鈕。

給學生的實作作業：

1. 整理一份咖啡廳的店家資料清單，包含店名、客群、特色、菜單與營業資訊。
2. 用本講提示詞讓 AI 產出 `index.html`，存檔後用瀏覽器打開。
3. 用手機寬度檢查首頁、菜單、營業資訊三個畫面，記錄需要修改的地方。
4. 至少使用一個修改句請 AI 調整，並確認修改後其他區塊沒有被弄壞。
5. 上傳 GitHub 並用 Vercel 部署，取得公開網址。
