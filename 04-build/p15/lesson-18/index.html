網站類型總表 需求訪談 報價拆分 交付清單 驗收流程 維護規則

把十七個網站的經驗，收斂成一張你帶得走的接案地圖

這個單頁原型整理全書的網站類型與功能等級、需求訪談問題、報價邊界、交付清單、驗收流程與維護規則。先選網站類型，再展開需求訪談，最後用驗收清單把案子交出去。

網站類型總表

全書十七講可以收斂成五種功能等級。報價與責任的差異，來自功能等級，不是頁面數。

對應資料物件：WebsiteType｜名稱、功能等級、適合業態、風險

等級 1

單頁形象網站

適合業態
咖啡廳、健身房、法律、寵物美容、個人教練
核心功能
介紹、信任、行動呼籲、聯絡
主要風險
文案不真實、手機版難用
等級 2

作品與部署網站

適合業態
攝影師與各類作品集
核心功能
作品分類、版本控制、部署、SEO、RWD
主要風險
圖片路徑錯、部署後壞掉
等級 3

預約與排程網站

適合業態
診所、餐廳、諮商、顧問
核心功能
時段、名額、取消、提醒
主要風險
超收、隱私、付款狀態錯
等級 4

會員與課程網站

適合業態
美容院、瑜伽教室、才藝教室、活動報名
核心功能
會員、點數、報名、候補、通知
主要風險
點數帳錯、候補混亂、通知漏寄
等級 5

交易與平台網站

適合業態
民宿訂房、家教媒合、顧問金流閉環
核心功能
訂單、付款、媒合、webhook、後台
主要風險
金流錯誤、資料不一致、權限不足

需求訪談

接案的第一步不是報價，是把需求問清楚。先確認業態、客群、核心行動與資料來源，範圍才不會失控。

對應資料物件：ScopeItem｜項目、是否包含、驗收方式

本原型不會把資料送到後端；訪談結果用於決定功能等級與報價拆分。

報價拆分

報價要拆成可以驗收的工作項目：頁面數、功能數、整合項目與維護。下表是欄位示範，估時與價格在需求訪談後才能載明，不在原型裡放虛構數字。

對應資料物件：QuoteLine｜工作項目、估時、價格、備註

報價拆分欄位示範（非實際報價）
工作項目估時價格備註
頁面數：首頁與服務頁訪談後估定依範圍載明含文案整理與手機版
功能數：表單與資料欄位訪談後估定依範圍載明欄位數與驗證規則影響估時
整合項目：金流、通知、第三方 API訪談後估定依範圍載明需列出測試環境與正式環境缺口
維護：每月更新與錯誤排查訪談後估定依範圍載明寫進維護條款，不含新功能

本報價包含：首頁、服務頁、表單、手機版、一次部署。

本報價不包含：會員登入、金流串接、後台管理、第三方 API 費用。

若後續新增功能：將依新增資料模型、流程狀態與測試範圍另行估價。

交付清單

專案結束時交給客戶的不是「一個網址」，而是一組可以被接手的資產。

對應資料物件：DeliveryAsset｜檔案、帳號、網址、文件

原始檔

  • 網站原始碼 repository
  • 圖片與文案來源檔
  • 修改紀錄可追蹤

部署網址

  • 正式公開網址
  • 自訂網域設定紀錄
  • 部署平台專案連結

帳號權限

  • 程式碼託管帳號歸屬
  • 部署平台帳號歸屬
  • 網域與信箱權限清單

文件

  • 操作說明：怎麼改文案與圖片
  • 維護規則與聯絡窗口
  • 驗收紀錄與已知限制

驗收流程

驗收不是「看起來不錯」，是逐項打勾。每一項都要寫清楚檢查方式與不通過時的修正方式。

對應資料物件：AcceptanceCheck｜檢查項目、結果、修正方式

驗收清單示範（結果欄為靜態示意）
檢查項目結果修正方式
手機版：首頁、表單、按鈕可讀可點通過—
表單：欄位齊全、送出有確認訊息通過—
資料：資料模型與匯出格式確認待修補上匯出欄位說明後重驗
付款：成功、失敗、取消頁完整待修在測試環境跑通後列出正式環境缺口
SEO：title 與 meta description 已設定通過—

維護規則

網站交出去之後才開始活。維護規則要回答四件事：多久更新、誰負責、怎麼收費、出錯怎麼回復。

對應資料物件：MaintenanceRule｜更新頻率、責任人、費用

更新頻率

  • 內容更新：依合約週期執行
  • 安全與相依套件：定期檢查

責任人與費用

  • 維護窗口與回應時間寫進合約
  • 費用依維護範圍另行載明

回復流程

  • 每次修改留下版本紀錄
  • 出錯時回復到上一個可用版本
