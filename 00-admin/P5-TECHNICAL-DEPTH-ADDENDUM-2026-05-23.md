# P5 — 教師端技術深度補充

日期：2026-05-23

## 一、文件定位

本文件不是學生正文，而是提供授課老師評估教材深度的補充說明。

新版《網頁設計》教材會維持學生可讀的語氣，但每一章背後會有更完整的技術骨架。正文不會要求學生一開始就成為工程師，但會讓學生理解網站設計背後的資訊架構、資料模型、狀態流程、部署流程、RWD（Responsive Web Design，響應式網頁設計）、SEO（Search Engine Optimization，搜尋引擎最佳化）、後台與金流等概念。

可以把技術深度分成三層：

| 層次 | 對象 | 目標 |
|---|---|---|
| L1 學生操作層 | 一般學生 | 能用 AI 產出網站、檢查畫面、完成基本部署 |
| L2 授課拆解層 | 授課老師帶學生理解 | 能說明頁面結構、資料欄位、狀態、流程與驗收 |
| L3 教師延伸層 | 博士級或技術背景教師 | 可延伸討論版本控制、部署 pipeline、資料一致性、金流通知、後台管理與系統限制 |

第 01 講主要示範 L1 + L2 的教材語氣。  
第 03 講主要示範 L2 + L3 的部署與版本控制深度。  
第 11 講後續會示範功能型網站的 L2 + L3 技術拆解。

## 二、第 03 講可延伸的技術深度

第 03 講主題是「攝影師作品集」，表面上是作品展示網站，但它適合作為版本控制與部署的第一個技術樣章。

### 1. Git 與版本控制

正文會用白話解釋 repository、commit、branch，但授課老師可以進一步延伸：

| 概念 | 可教學重點 |
|---|---|
| working tree | 學生本機正在修改的檔案狀態 |
| commit history | 每一次有說明的存檔紀錄，可追溯修改原因 |
| branch | 不同版本線，可用於實驗新版設計 |
| diff | 比較前後修改，理解「改了什麼」 |
| rollback | 出錯時回到前一版，不必手動找備份 |

教學重點不是要求學生背 Git 指令，而是理解：正式網站不能只靠「另存新檔」管理版本。

### 2. GitHub 與 repository

GitHub 在本教材中不是單純雲端硬碟，而是網站專案的保存與協作入口。

可延伸討論：

- repository 為什麼要有清楚命名。
- `README.md` 為什麼能記錄網站用途、更新方式與部署網址。
- 為什麼圖片路徑、大小寫與資料夾結構會影響部署結果。
- 為什麼 commit message 對教材、團隊合作與日後維護都有價值。

### 3. Vercel deployment pipeline

第 03 講正文會讓學生理解：

```text
GitHub 更新 → Vercel 偵測 → 建立 deployment → production 網站更新
```

教師可進一步拆成：

| 階段 | 技術意義 |
|---|---|
| Git provider connection | Vercel 取得 repository 的讀取權限 |
| build step | 將原始檔轉成可部署網站；靜態 HTML 可無 build |
| preview deployment | 每次變更可先產生測試網址 |
| production deployment | 正式對外公開版本 |
| rollback | 若新版出錯，可回到前一次 deployment |

這能讓學生知道網站上線不是「上傳一次就結束」，而是持續更新的流程。

### 4. DNS 與自訂網域

正文會以自訂網域作為品牌專業度延伸。教師端可以補充：

| 名詞 | 說明 |
|---|---|
| domain | 使用者輸入的網址，例如 `kuophotography.com` |
| registrar | 網域註冊商，負責網域購買與續約 |
| nameserver | 管理 DNS 記錄的服務 |
| A record | 將網域指向 IP 位址 |
| CNAME | 將子網域指向另一個網域 |
| DNS propagation | 設定更新需要時間傳播 |

可討論問題：

- 為什麼 `www.example.com` 和 `example.com` 可能要分開設定？
- 為什麼 DNS 設好後不一定立刻生效？
- 為什麼自訂網域會影響品牌信任？

### 5. RWD（Responsive Web Design，響應式網頁設計）與圖片效能

攝影師作品集特別適合教 RWD，因為照片在不同裝置上很容易出問題。

教師可延伸：

- 桌機三欄、平板兩欄、手機單欄的版面策略。
- `@media` 查詢如何切換欄數。
- 圖片寬度、比例、裁切與載入速度的取捨。
- `alt` 文字同時影響無障礙與搜尋理解。
- 大圖未壓縮會拖慢作品集載入，影響轉換。

### 6. SEO（Search Engine Optimization，搜尋引擎最佳化）與結構化內容

第 03 講可將 SEO 控制在基礎但正確的層次：

- `<title>`：頁面在搜尋結果與分頁標籤顯示的名稱。
- `meta description`：搜尋結果摘要的候選內容。
- 語意化 HTML：`header`、`main`、`section`、`footer` 讓內容結構更清楚。
- 圖片 `alt`：協助搜尋與無障礙閱讀。
- 服務地區與服務類型：讓 Google 知道「台中婚紗攝影」這類關鍵字的關聯。

## 三、第 11 講可延伸的技術深度

第 11 講「民宿訂房網站」是功能型網站樣章。它適合展示網站從「展示頁」進入「交易系統」後會遇到的資料、狀態與一致性問題。

### 1. 為什麼訂房網站不能只是表單

Google Form 或一般聯絡表單只能收資料，不能保證房況。

真正的訂房網站至少要處理：

- 查詢某日期是否還有房。
- 同一間房同一天不能被兩個人訂走。
- 付款前房間要暫時保留。
- 付款失敗後房間要釋放。
- 取消後房況與退款要同步。
- 房東要能在後台封鎖維修日或調整價格。

### 2. 房況資料模型

房況不是一個文字欄位，而是一組以「房間 + 日期」為核心的資料。

最小概念：

| 資料 | 說明 |
|---|---|
| RoomType | 房型，例如雙人房、四人房 |
| Room | 實際房間，例如 201、202 |
| Availability | 某房間在某一天的狀態 |
| Order | 訂單，記錄入住日期、退房日期、旅客資料 |
| Payment | 付款資料，記錄金流狀態 |
| Guest | 旅客資料，可先嵌在訂單內 |

教師可引導學生理解：民宿網站不是只有「日期選擇器」，背後是日期範圍與房間狀態的對應。

### 3. 房況狀態流程

房間某一天的狀態可能包含：

```text
可訂 → 保留中 → 已預訂 → 入住中 → 清潔中 → 可訂
```

也可能被房東手動改成：

```text
封鎖 / 維修
```

這裡可以用交通燈比喻：

- 綠燈：可訂。
- 黃燈：保留中，等待付款。
- 紅燈：已預訂或封鎖。

教師可延伸討論：為什麼「保留中」很重要？如果旅客正在付款，房間卻同時開放給其他人，就會發生超賣。

### 4. 訂單狀態流程

訂單狀態和房況狀態不同，但必須互相對應。

```text
待付款 → 已確認 → 已完成
   │         │
   │         └→ 已取消 → 已退款
   └→ 已取消
```

教學重點：

- 訂單不是送出表單就完成。
- 訂單要等付款成功後才應該正式確認。
- 取消與退款也需要被記錄。
- 訂單狀態改變時，房況狀態也要跟著改。

### 5. 防超賣與資料一致性

民宿訂房最重要的系統問題是防超賣。

可用白話說明：

同一間房、同一個日期，只能有一筆有效訂單占用。若兩位旅客同時下訂，系統必須保證只有一位能成功。

教師端可延伸到：

- 唯一性限制：同一 `(room_id, date)` 不可重複被有效訂單占用。
- 鎖定機制：付款期間先進入保留中。
- 到期釋放：保留中超過時間未付款，要自動回到可訂。
- 交易一致性：付款成功與房況更新不能只完成其中一半。

### 6. 金流 redirect 與 webhook

學生常會以為「看到付款成功頁」就代表付款成功。實務上不應該這樣設計。

較正確的觀念是：

```text
旅客付款後回到成功頁：給旅客看的使用者體驗
金流 webhook 通知系統：系統判定付款成功的主要依據
```

原因：

- 使用者可能關掉付款頁。
- 網路中斷可能讓成功頁沒有回來。
- 有些付款方式會延遲確認。
- 系統必須驗證金流方回傳內容與簽章。

教師可用這題討論：

為什麼付款成功頁不能當成唯一信源？

### 7. 後台管理

功能型網站需要後台，不只是前台畫面。

民宿後台至少包含：

| 後台模組 | 功能 |
|---|---|
| 房況管理 | 開放、封鎖、維修、清潔 |
| 訂單管理 | 查看訂單、確認旅客、處理取消 |
| 價格管理 | 平日、假日、旺季、特殊日 |
| 黑名單日期 | 封鎖颱風天、整修日、私人使用日 |

這一段可以讓學生理解：網站設計不只設計給客人，也要設計給經營者。

## 四、教師可延伸討論題

以下題目可供授課老師在課堂中延伸：

1. 為什麼攝影師只經營 Instagram，不等於擁有自己的網站？
2. 為什麼作品集網站需要版本控制？
3. 為什麼圖片在本機看得到，上線後可能消失？
4. 為什麼 Vercel 可以在 GitHub 更新後自動部署？
5. 為什麼自訂網域會影響品牌信任？
6. 為什麼訂房網站不能只用 Google Form？
7. 為什麼訂房系統需要「保留中」狀態？
8. 為什麼付款成功頁不能當成付款成功的唯一依據？
9. 為什麼功能型網站需要後台？
10. 為什麼資料模型會影響網頁設計？

## 五、後續修稿原則

後續 18 講不會每章都塞同樣深度的工程內容，而是依業態分配：

| 章節類型 | 技術重點 |
|---|---|
| 形象型網站 | 資訊架構、視覺語言、RWD（Responsive Web Design，響應式網頁設計）、SEO（Search Engine Optimization，搜尋引擎最佳化）、部署 |
| 預約型網站 | 表單、時段、容量、取消、通知 |
| 會員型網站 | 登入、會員資料、點數、交易紀錄 |
| 訂房型網站 | 日期、房況、狀態、付款、後台 |
| 媒合型網站 | 雙邊使用者、搜尋、篩選、推薦、信任 |
| 金流型網站 | 付款頁、成功/失敗頁、通知、退款、發票 |

這樣可以避免教材變成純故事，也避免變成純工程手冊。學生讀得到，老師教得開，技術背景較強的授課老師也能看見可延伸的深度。
