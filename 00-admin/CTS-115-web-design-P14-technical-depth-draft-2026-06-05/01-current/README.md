## 一、正式稿規則

### 檔名

- 格式：`NN-主題.md`（兩位數講次 + 半形連字號 + 主題；NN 範圍 01–18）
- 不帶 `第XX講` 前綴、不帶 `.v2` 字尾、不帶 `(上)/(下)` / `-draft` / `-final` / `-backup`
- 章節模板檔保留 `00-` 前綴：`00-chapter-template.md`

### 內容

- 章節骨架統一見 `00-chapter-template.md`，鄒老師已確認的六段結構
- 修稿走 MANIFEST「狀態欄」機制，不在檔名上加版本字尾
- 本資料夾只放正式主稿；草稿、實驗稿、備份請勿落地於此

### 範圍

- 本資料夾共 **18 個正式講次檔案** + `00-chapter-template.md` + `MANIFEST.md` + `README.md`
- 多 1 個或少 1 個都算工程層錯誤，需修

## 三、章節狀態規則（MANIFEST 狀態欄）

| 狀態 | 條件 | 下一步 |
|---|---|---|
| 整理完成 | 歷史狀態；v2 主稿就位、檔名/位置正確 | 不再作為目前章節狀態 |
| 待補技術內容 | 歷史狀態；主稿存在但缺技術段落 | 不再作為目前章節狀態 |
| 綠燈樣章 | 老師已認可方向，可作為全書基準 | 保持方向，只做必要一致化 |
| 技術深度完成 | 已補入資料模型、狀態流程、提示詞與驗收清單 | 進教材化一致化 |
| 教材化一致化完成 | 已統一六段式骨架、設計重點、名詞註解與敏感領域語氣 | 可送老師或排版前審稿 |
| 待老師確認 | 結構或主題仍需鄒老師拍板 | 等老師回覆 |
| 已定稿 | 通過所有審稿、可送排版 | 進 `../04-build/` 流程 |

升級路徑（典型）：
`整理完成` → `技術深度完成` → `教材化一致化完成` → `待老師確認` → `已定稿`

可跳階，但不可繞過：每次升級需在 MANIFEST 備註欄寫一行說明。

## 四、樣板章規則

`01-咖啡廳網站.md` 是 P3 架構樣板章。後續 02–18 講修稿時，先以它作為語氣、深度與段落密度的參考，不直接複製內容。

`03-攝影師作品集.md` 是 P4A 技術深度樣板章。後續涉及部署、版本控制、資料模型、狀態流程或金流等技術主題的章節，需參考它的寫法：先用業態問題導入，再把技術拆成教師可講、學生可操作的流程。

樣板章規則：

- 頂層 `##` 只使用鄒老師確認的六段結構
- 每講開頭先寫「本講任務」與「本講成果」，讓教師、導播與學生都知道本講要完成什麼
- 「破冰案例」保留故事，但篇幅控制在能快速帶出業態問題
- 「業態需求」說明網站要替該行業解決的真問題
- 「網頁設計重點」固定拆成網站風格、建立信任、引導行動
- 「功能與資料模型」用表格拆元件、欄位、狀態與流程
- 「AI 協作實作」提供可複製提示詞、修改句與檢查方法
- 「上線與延伸」收斂成本講驗收標準、部署流程與下一步功能

舊稿回收只做必要補強，不整段搬運舊敘事。回收後要把語氣調回樣板章的教材語氣。
