# 第 11 講：民宿訂房網站

本講任務：用 AI 協作完成一個民宿訂房網站原型，讓旅客能看見房型、查詢日期、理解價格與取消政策，並完成一條「從想住到送出訂房」的流程。

本講完成後，你會得到四個成果：

- 一份民宿訂房網站需求摘要。
- 一個可展示的訂房網站原型。
- 一份房型、房況、訂單、付款與後台的資料模型草稿。
- 一份功能型網站驗收清單，能檢查是否有超賣、假付款、假評論或流程斷點。

## 1. 破冰案例

週五晚上八點半，一對夫妻把車停在休息站，開始找下週末要住的地方。

他們不是第一次用訂房平台。Booking.com、Airbnb、Agoda 都很熟。可是這次他們想找的不是連鎖飯店，也不是標準化的住宿。他們搜尋的是「宜蘭 礁溪 手作民宿」。

搜尋結果第三個不是大型訂房平台，而是一家民宿自己的官網。

首頁第一張照片不是空房間，而是早晨的光從竹百葉窗斜切進來，桌上有一碗剛盛好的稀飯，旁邊放著幾碟醃漬小菜。照片沒有很華麗，但很有生活感。

副駕駛座上的她看了三秒，說：「這家感覺對。」

她繼續往下滑，看見房間、早餐、附近散步路線，也看見主人寫的一小段話：為什麼搬到礁溪，為什麼只做三間房，為什麼早餐不用中央廚房。

到這裡，信任其實已經開始建立。

可是她點到訂房區時，看到一句話：

```text
如欲訂房，請先匯款至以下帳號，再傳簡訊告知入住日期與人數，我們確認後回覆。
```

她停了一下，把手機放回腿上，又打開 Airbnb。

這家民宿做對了一件事，也做錯了一件事。

做對的是，它讓旅客感覺自己不是在買一張床，而是在進入一種生活。那張早餐照片、主人故事、在地路線，都是大型平台很難複製的東西。

做錯的是，它沒有把這份信任接到一個現代的訂房流程上。

旅客不是不願意支持獨立民宿。旅客怕的是：房間到底有沒有保留？匯款後會不會沒確認？日期會不會被別人先訂走？如果臨時取消，錢怎麼退？

民宿網站的難度就在這裡：它既要像一個有溫度的家，也要像一個可靠的系統。

## 2. 業態需求

民宿訂房網站不是把房間照片放上去而已。它要同時解決三層問題：旅客的不安、房東的收入、系統的準確性。

第一，旅客要確認這裡是不是真的適合他。

訂民宿和訂飯店不同。飯店有品牌標準，民宿靠的是獨特性。獨特性會吸引人，也會讓人不安。旅客會擔心照片是不是修太多、隔音好不好、半夜出問題有沒有人處理、導航會不會帶到山路裡。

所以民宿網站要比一般形象網站提供更多證據：

- 真實房間照片，不只放最美的角度。
- 房型設備與限制，例如是否有電梯、是否可帶寵物、是否有停車。
- 清楚的入住、退房、早餐與取消規則。
- 主人或管家的聯絡方式。
- 最後一哩交通指引，例如路口照片、停車入口、夜間導航提醒。

第二，房東要降低對 OTA 平台的依賴。

OTA 是 Online Travel Agency，指 Booking.com、Airbnb、Agoda 這類線上旅遊訂房平台。它們能帶來流量，也會抽成。民宿若每張訂單都依賴平台，就會遇到幾個問題：

| 問題 | 對民宿的影響 |
|---|---|
| 平台抽成 | 每張訂單約有一部分房費流向平台 |
| 評價綁架 | 評論與排名留在平台，民宿自己帶不走 |
| 定價壓力 | 平台促銷、排名與折扣機制會影響價格 |
| 客戶資料隔離 | 房東難以經營回頭客 |

官網直訂不是要完全取代平台，而是讓平台從「唯一入口」變成「曝光管道」。旅客第一次可能從平台認識民宿，但第二次、第三次回來時，最好能找到官網。

第三，系統要避免訂房錯誤。

咖啡廳網站做錯一個按鈕，通常只是少一通電話。民宿訂房網站做錯，可能會變成同一天同一間房被兩組旅客訂走。

所以本講開始，我們正式進入「功能型網站」。

功能型網站不只是展示資訊，而是要處理資料、狀態與流程。民宿訂房至少要知道：

- 哪些房型可以賣。
- 每個房型有幾間實際房間。
- 某一天某間房是否可訂。
- 旅客選的日期是否跨過已訂日。
- 付款前房間是否要暫時保留。
- 付款成功後房況如何變成已預訂。
- 取消後房況與退款如何同步。
- 房東如何在後台鎖定維修日或調整旺季價格。

本講採用的基本業態假設如下：

| 項目 | 本講設定 |
|---|---|
| 店型 | 小型獨立民宿 |
| 主要客群 | 週末短住旅客、夫妻、朋友小旅行、家庭客 |
| 網站型態 | 功能型訂房網站原型 |
| 核心行動 | 查詢房況、選擇日期、送出訂房、完成付款或付款模擬 |
| 技術目標 | 用 AI 產出可展示的訂房流程，並拆出資料模型與狀態流程 |

這一講不要求學生真的串接金流，也不要求學生做出正式可營運系統。第一版的目標是做出「看得懂、跑得通、檢查得出問題」的功能原型。

## 3. 網頁設計重點

### 3.1 網站風格

民宿網站的風格，不能只追求漂亮。它要讓旅客感覺「我知道我會住進什麼地方」。

大型平台的照片常常追求標準化：明亮、乾淨、廣角、床鋪整齊。民宿官網可以做另一件事：把人的生活感放進去。

適合民宿網站的照片不只有房間，也包含：

- 早晨餐桌。
- 入口與招牌。
- 窗外景色。
- 浴室與清潔細節。
- 停車位置。
- 附近散步路線。
- 主人準備早餐或整理空間的片段。

但生活感不等於模糊，也不等於過度包裝。網站可以把空間拍得好看，但不能把三坪房間拍成六坪，也不能把沒有景觀的房間拍得像面山。照片一旦讓旅客抵達後感到落差，網站前面建立的信任會一次破掉。

不同民宿類型可以有不同視覺語言：

| 民宿類型 | 視覺語言 | 首頁優先展示 |
|---|---|---|
| 溫泉民宿 | 放鬆、隱私、安定、療癒 | 浴池、房內光線、泡湯規則 |
| 親子民宿 | 安全、明亮、便利、可預期 | 遊戲區、床型、停車、備品 |
| 山林民宿 | 自然、安靜、慢生活 | 景觀、步道、早餐、交通提醒 |
| 設計民宿 | 品味、材質、空間感 | 房內細節、家具、品牌故事 |

民宿網站的第一張照片很重要。使用者對網頁的第一印象可能在極短時間內形成。對訂房場景來說，首圖不是裝飾，而是整個訂房漏斗的開關。

第一張圖最好回答：

1. 這裡是什麼氣氛？
2. 我是不是想把週末放在這裡？
3. 這張照片是否真實到讓我願意繼續看？

### 3.2 建立信任

民宿的信任模組要直接回應旅客的恐懼。

旅客不是只怕價格太高。他更怕的是到現場才發現和想像不同。

基本信任模組如下：

| 信任模組 | 應該包含 | 解決什麼不安 |
|---|---|---|
| 房型照片 | 主圖、副圖、浴室、窗景、入口 | 避免照騙 |
| 設備清單 | 床型、衛浴、空調、Wi-Fi、停車 | 確認基本需求 |
| 入住規則 | check-in、check-out、早餐、加人、寵物 | 降低溝通成本 |
| 取消政策 | 幾天前可退、退多少、如何申請 | 降低損失焦慮 |
| 真實評論 | 可含小缺點，不只全五星 | 提供第三方證據 |
| 主人介紹 | 為何經營、如何接待、聯絡方式 | 讓人知道背後有人 |
| 交通指引 | 地址、導航連結、停車入口、夜間提醒 | 避免最後一哩焦慮 |
| 安全資訊 | 門鎖、消防、緊急聯絡方式 | 回應安全與隱私 |

取消政策尤其重要。太硬的取消政策會讓旅客在按下訂房前停住，因為他心裡會想：「萬一孩子生病？萬一下雨？萬一工作臨時變動？」

可以用階梯式取消政策來平衡旅客安全感與房東成本：

| 距離入住日 | 建議退款比例 | 設計理由 |
|---|---|---|
| 30 天以上 | 100% | 行程還早，給旅客信心 |
| 14 到 30 天 | 75% | 房東已有部分機會成本 |
| 7 到 14 天 | 50% | 重新出租時間變短 |
| 3 天內 | 0% 或僅保留延期 | 房東很難再賣出房間 |

這不是唯一答案，但它讓規則清楚。清楚比看似彈性但需要私訊詢問更能建立信任。

### 3.3 引導行動

民宿網站的轉換路徑比形象網站長。旅客不是看到照片就訂，他會經過一連串確認。

基本路徑可以設計成：

```text
看到首頁 → 看房型 → 查日期 → 看價格與規則 → 填資料 → 付款或送出訂房 → 收到確認
```

每一步都要回答一個問題：

| 步驟 | 旅客心裡的問題 | 網站要提供的答案 |
|---|---|---|
| 首頁 | 這裡是不是我想住的地方？ | 主張、首圖、生活感 |
| 房型 | 這間房適合幾個人？ | 人數、床型、設備、照片 |
| 日期 | 我想住的日期有沒有房？ | 日期選擇器與房況顯示 |
| 價格 | 總共多少錢？有沒有額外費用？ | 房價、加人費、清潔費、折扣 |
| 規則 | 不合適或臨時取消怎麼辦？ | 取消政策、入住規則 |
| 付款 | 我付了錢後有沒有被確認？ | 付款狀態與訂房確認 |

直訂優惠可以放在首頁與訂房區，但要說得精準。例如：

```text
官網直訂，比平台價格省 8%，並附入住前在地散步地圖。
```

這句比「官網最優惠」更好，因為它同時說明價格利益與民宿自己的價值。

## 4. 功能與資料模型

民宿訂房網站是本課程第一個完整的功能型網站範例。它不能只靠一個聯絡表單完成，因為表單只會收資料，不會保證房況。

### 4.1 為什麼不能只做表單

如果網站只有一個「我要訂房」表單，旅客送出後，房東還要人工確認：

- 這個日期有沒有房？
- 同一時間有沒有另一個人也送出？
- 旅客人數是否超過房型限制？
- 價格是否包含週末或旺季加價？
- 付款前要不要先保留房間？

這種流程可以在非常小的民宿暫時使用，但它不是一個真正的訂房系統。

真正的訂房系統至少要處理三件事：

1. 日期與房況。
2. 訂單與付款。
3. 後台與管理。

### 4.2 頁面與功能模組

一個民宿訂房網站可以拆成下列模組：

| 模組 | 功能 | 必要內容 |
|---|---|---|
| Hero 首頁 | 建立第一印象 | 民宿名稱、生活主張、首圖、直訂優惠 |
| 空間故事 | 建立人感 | 主人介紹、早餐、在地連結、經營理念 |
| 房型列表 | 讓旅客比較 | 房型、價格區間、人數、照片、設施 |
| 房型詳情 | 讓旅客判斷 | 主圖、副圖、床型、衛浴、規則、可入住人數 |
| 日期選擇器 | 查詢房況 | 入住日、退房日、已訂日、價格 |
| 訂房表單 | 收集資料 | 姓名、電話、Email、人數、備註 |
| 付款流程 | 完成確認 | 訂單摘要、付款狀態、成功與失敗頁 |
| 在地指南 | 增加民宿價值 | 景點、餐廳、散步路線、雨天備案 |
| 交通指引 | 降低抵達焦慮 | 導航、停車、夜間路標 |
| 後台 | 讓房東管理 | 房況、訂單、價格、黑名單日期 |

### 4.3 資料模型

先把資料拆出來，AI 才不會把所有內容寫成一張漂亮但不能管理的頁面。

最小資料模型如下：

| 資料物件 | 說明 | 重要欄位 |
|---|---|---|
| `RoomType` 房型 | 對外顯示的房型，例如雙人房、四人房 | 名稱、人數、坪數、設備、照片、平日價、週末價 |
| `Room` 房間 | 實際可出租的房間，例如 201、202 | 房間編號、所屬房型、是否上架 |
| `Availability` 房況 | 某間房在某一天的狀態 | 房間、日期、狀態、價格、保留到期時間 |
| `Order` 訂單 | 旅客的一次訂房 | 入住日、退房日、人數、旅客資料、總金額、訂單狀態 |
| `Payment` 付款 | 訂單的付款紀錄 | 金流商、交易號、金額、付款狀態、通知時間 |
| `Guest` 旅客 | 回頭客或會員資料，第一版可先嵌在訂單中 | 姓名、Email、電話、入住次數 |

這裡最容易被初學者忽略的是 `RoomType` 和 `Room` 的差別。

「檜木雙人房」是一種房型。如果民宿有三間檜木雙人房，它們對旅客看起來可能相似，但系統裡必須是三個房間。否則同一天只要賣出一間，系統就可能以為整個房型都沒了，或反過來以為還有房而造成超賣。

### 4.4 房況狀態

房況不是一句「有房 / 沒房」就能表示。某一天某一間房，可能處在不同狀態。

可以先用這個狀態流程理解：

```text
可訂 → 保留中 → 已預訂 → 入住中 → 清潔中 → 可訂
  │        │
  │        └→ 付款逾時 → 可訂
  └→ 封鎖 / 維修
```

各狀態的意思如下：

| 狀態 | 白話說明 | 典型觸發 |
|---|---|---|
| 可訂 | 旅客可以選這一天 | 沒有訂單、沒有封鎖 |
| 保留中 | 旅客正在付款，先暫時鎖住 | 訂房表單送出 |
| 已預訂 | 付款成功或房東確認 | 金流通知成功 |
| 入住中 | 旅客正在住 | 入住日到達 |
| 清潔中 | 旅客退房後等待整理 | 退房日到達 |
| 封鎖 / 維修 | 房東手動不開放 | 自住、維修、包棟、節日調整 |

「保留中」是訂房系統很重要的一步。

如果旅客正在付款，房間還顯示可訂，另一個人也可能同時下單。這就是超賣。解法是先把房間鎖住一段時間，例如 15 分鐘。如果 15 分鐘內付款成功，就變成已預訂；如果逾時，就釋放回可訂。

### 4.5 訂單與付款狀態

訂單狀態和房況狀態不同，但必須互相對應。

可以先用這個流程：

```text
待付款 → 已確認 → 已完成
   │         │
   │         └→ 已取消 → 已退款
   └→ 已取消
```

付款流程也要分成兩種畫面：

| 畫面 | 功能 | 注意事項 |
|---|---|---|
| 付款返回頁 | 旅客付款後被導回網站看到的畫面 | 只能當使用者體驗提示 |
| 金流通知 webhook | 金流公司主動通知網站付款結果 | 應作為系統確認付款的主信源 |

Webhook 可以先理解成「金流公司打電話回來告訴網站：這筆真的付款成功了」。正式系統不能只相信旅客看到的成功頁，因為成功頁可能被關掉、重整、或傳回不完整。

第一版課堂原型可以不用真的串接金流，但文字與流程要正確：

- 可以做「付款模擬」按鈕。
- 可以顯示付款成功與付款失敗頁。
- 不要宣稱已經完成真實付款串接。
- 不要收真實信用卡資料。

### 4.6 後台管理

民宿網站如果只有前台，房東很快會卡住。因為房價、房況、維修日、取消退款都會變。

後台至少要有四個模組：

| 後台模組 | 房東要做什麼 |
|---|---|
| 房況管理 | 在日曆上鎖日期、開日期、標記維修或私人使用 |
| 訂單管理 | 查看待付款、已確認、已取消訂單，匯出月結 |
| 價格管理 | 設定平日、週末、旺季、連住折扣 |
| 黑名單日期 | 春節、自住、設備維修等日期不可訂 |

動態定價可以先用簡單四層：

| 價格層 | 範例 |
|---|---|
| 平日價 | 週一到週四基本價 |
| 週末價 | 週五、週六加價 |
| 旺季價 | 連假、寒暑假、跨年、特殊活動 |
| 長住折扣 | 三晚以上或七晚以上折扣 |

設計後台時要記得：房東通常不是工程師。後台要像日曆，不要像資料庫。能用點選和表單完成的，就不要讓房東寫公式。

## 5. AI 協作實作

這一講的 AI 協作目標，不是一次做出正式可收款的訂房系統，而是做出一個可展示、可討論、可驗收的功能原型。

建議分四輪跟 AI 合作：

1. 先產出網站原型。
2. 再補資料模型。
3. 再補互動狀態。
4. 最後做驗收與除錯。

### 5.1 第一輪：產出訂房網站原型

可以使用這段提示詞：

```text
請幫我製作一個民宿訂房網站原型。

網站名稱：山邊晨光民宿
地點：宜蘭礁溪
主要客群：台北週末短住旅客、夫妻、小家庭
風格：溫暖、安靜、有手作感
色調：米白、木褐、深苔綠

請製作單頁 HTML，包含以下區塊：
1. 首頁主視覺：一句話說清楚民宿生活主張，並放兩個按鈕：「查看房型」「查詢日期」
2. 空間故事：主人為什麼在這裡開民宿，早餐食材從哪裡來
3. 房型列表：至少兩種房型，每種要有人數、床型、平日價、週末價、主要設備
4. 日期查詢：入住日、退房日、人數選擇器，先用前端模擬房況即可
5. 訂房摘要：顯示選擇的房型、晚數、總價、取消政策
6. 訂房表單：姓名、Email、電話、備註
7. 付款模擬：付款成功與付款失敗兩種狀態，不要收真實信用卡資料
8. 在地指南：三個推薦地點
9. 交通指引：開車、搭車、停車與夜間抵達提醒
10. 常見問答：入住、退房、早餐、寵物、取消政策

技術要求：
1. 請直接輸出完整 HTML，包含 CSS 與少量 JavaScript。
2. 手機版要好讀，按鈕高度至少 44px。
3. 日期不可選到過去。
4. 若選到已預訂日期，要顯示不可訂。
5. 價格要依晚數計算。
6. 所有圖片先用可替換的 placeholder，不要假裝是真實民宿照片。
7. 不要產生假評論、假媒體報導或假證照。
8. 程式碼中請用註解標出哪些地方未來要接資料庫、金流 webhook 與後台。
```

第一輪產出的網站如果看起來漂亮，但流程跑不通，不算完成。民宿網站不是海報，它要能讓人選日期、看價格、填資料、看到下一步。

### 5.2 第二輪：要求 AI 補資料模型

第一版畫面出來後，接著問 AI：

```text
請根據剛才的民宿訂房網站，幫我整理資料模型。

請用表格列出：
1. RoomType 房型
2. Room 實際房間
3. Availability 每房每日房況
4. Order 訂單
5. Payment 付款
6. Guest 旅客

每個資料物件請列出欄位、欄位說明、範例值。
請特別說明 RoomType 和 Room 的差別，以及 Availability 為什麼要用「每房每日一筆」來設計。
```

這一步是讓學生理解：網頁上的卡片、表單、日曆，其實都對應到資料。只要資料沒拆清楚，AI 很容易做出一個不能維護的漂亮頁面。

### 5.3 第三輪：要求 AI 補狀態流程

接著補狀態：

```text
請幫我補上民宿訂房的狀態流程。

請包含：
1. 房況狀態：可訂、保留中、已預訂、入住中、清潔中、封鎖、維修
2. 訂單狀態：待付款、已確認、已取消、已退款、已完成
3. 付款狀態：未付款、付款中、付款成功、付款失敗、已退款

請用白話說明每一個狀態如何轉換。
也請說明為什麼付款成功頁不能取代 webhook。
```

這一輪的重點是讓 AI 不只幫你做畫面，而是幫你把「流程邏輯」說清楚。

### 5.4 第四輪：修改與除錯

AI 第一次產出的民宿訂房網站，常見問題有五種：

| 常見問題 | 為什麼危險 | 修改方式 |
|---|---|---|
| 只做聯絡表單 | 看起來能訂，其實不能查房況 | 要求加入日期查詢與房況狀態 |
| 所有日期都能選 | 會造成已訂日期仍可下單 | 加入已訂日與不可選邏輯 |
| 總價不會變 | 晚數、週末、加人費都沒算 | 要求顯示計價公式 |
| 付款按鈕假裝真串接 | 可能誤導使用者 | 改成「付款模擬」，標明未串接 |
| 放假評論或假證照 | 破壞信任，也可能違法 | 全部刪除，改成「待填入真實評論」 |

可以用這段提示詞修正：

```text
請檢查剛才的民宿訂房網站是否有以下問題：
1. 是否只是表單，沒有房況查詢？
2. 是否所有日期都能選？
3. 是否沒有計算晚數與總價？
4. 是否假裝已串接真實金流？
5. 是否出現假評論、假媒體報導或假證照？

請直接修改 HTML，保留原本視覺風格，但補上缺少的互動與警示文字。
```

### 5.5 驗收清單

完成後，用這份清單檢查：

| 檢查項目 | 通過標準 |
|---|---|
| 首頁定位 | 5 秒內看得懂民宿在哪裡、適合誰、主打什麼 |
| 房型資訊 | 每個房型有人數、床型、價格、設備與限制 |
| 日期查詢 | 過去日期不可選，已訂日期不可選 |
| 計價邏輯 | 晚數、週末價、加人費或折扣至少有一項可看見 |
| 取消政策 | 不需要私訊才能知道退費規則 |
| 付款流程 | 清楚標明是付款模擬或正式金流 |
| 狀態流程 | 看得出保留中、已預訂、取消、退款的差別 |
| 後台概念 | 能說明房東如何鎖日期、改價格、查訂單 |
| 內容誠實 | 沒有假評論、假證照、假媒體報導 |
| 手機版 | 日期、按鈕、表單在手機上可讀可點 |

## 6. 上線與延伸

第一版民宿訂房網站可以先用靜態網站方式上線，例如 Vercel 或 Netlify。這一版適合展示品牌、房型、在地指南與訂房流程概念。

但正式營運前，要分清楚哪些是原型，哪些是正式系統。

### 6.1 第一版可以上線的內容

第一版可以安全上線的內容包括：

- 民宿介紹。
- 房型介紹。
- 真實照片。
- 價格區間。
- 入住規則。
- 取消政策。
- 交通指引。
- 聯絡方式。
- 表單詢問。

如果還沒有資料庫與金流，不要把網站寫成「即時訂房已完成」。可以改成：

```text
送出後，我們會在 2 小時內確認房況並回覆。
```

這句雖然不如即時訂房方便，但它誠實。

### 6.2 正式營運需要的系統

要變成真正的訂房系統，後續需要加入：

| 延伸功能 | 需要處理的事情 |
|---|---|
| 資料庫 | 儲存房型、房間、房況、訂單、付款 |
| 會員或訂單查詢 | 讓旅客查自己的訂單，不一定要完整會員系統 |
| 金流串接 | 綠界、藍新、Stripe 或其他付款服務 |
| Webhook | 以金流通知確認付款結果 |
| Email / LINE 通知 | 訂房成功、入住提醒、取消通知 |
| 房東後台 | 日曆鎖房、改價、查訂單、處理退款 |
| 資安與個資 | 電話、Email、付款資料不可隨便曝光 |

這些功能不適合全部塞進第一版。第一版先把流程說清楚，第二版再逐步接資料庫與付款。

### 6.3 可以延伸的商業功能

民宿的收入不只來自房費。網站後續可以加入：

- 在地導覽預約。
- 早餐加購。
- 接駁服務。
- 租車或活動推薦。
- 回頭客折扣碼。
- 入住後評價邀請。

但每一個延伸功能都要回到同一個原則：不要讓旅客覺得被推銷，而是讓旅客覺得被照顧。

### 6.4 重要名詞

| 名詞 | 白話說明 |
|---|---|
| OTA | 線上旅遊訂房平台，例如 Booking.com、Airbnb、Agoda |
| 直訂 | 旅客不經平台，直接在民宿官網或官方管道訂房 |
| 房況 | 某間房在某一天是否可訂，以及目前狀態 |
| 狀態機 | 把可訂、保留中、已預訂、取消等狀態和轉換規則列清楚 |
| 併發衝突 | 兩個旅客幾乎同時搶同一間房，系統必須避免重複賣出 |
| Webhook | 外部服務主動通知網站事件結果，例如金流公司通知付款成功 |
| 動態定價 | 依平日、週末、旺季、連住等條件調整價格 |
| 黑名單日期 | 房東手動鎖住不可訂的日期，例如維修、自住、特殊活動 |

### 6.5 自我評量

1. 為什麼民宿訂房網站不能只做一個聯絡表單？
2. `RoomType` 和 `Room` 有什麼不同？
3. 為什麼要有「保留中」狀態？
4. 付款成功頁和 webhook 的差別是什麼？
5. 取消政策為什麼會影響轉換率？
6. 如果同一天同一間房被兩個人訂到，可能是哪個流程沒有設計好？
7. 民宿官網直訂和 OTA 平台應該是什麼關係？
8. 哪些內容可以包裝？哪些內容不能偽裝？
