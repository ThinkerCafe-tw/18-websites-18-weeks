# 第 03 講：攝影師作品集

本講任務：用 AI（Artificial Intelligence，人工智慧）協作完成一個攝影師作品集網站，並把網站從本機 `index.html` 推進到 GitHub（程式碼與專案託管平台）與 Vercel（前端部署平台），讓它成為可以公開分享、可以持續更新、可以綁定自訂網域的正式作品集。

本講完成後，你會得到四個成果：

- 一個攝影師作品集網站。
- 一個 GitHub repository，保存網站檔案與修改紀錄。
- 一個 Vercel deployment，取得公開網址。
- 一份手機版、SEO（Search Engine Optimization，搜尋引擎最佳化）、部署與自訂網域檢查清單。

## 1. 破冰案例

小柯是台中一位婚紗攝影師。入行七年，作品穩定，修圖風格也有辨識度。他最常聽到朋友說：「你的照片明明比很多人好，為什麼客人沒有比較多？」

他一直以為問題是 Instagram 經營得不夠好，所以每週固定發文、做 Reels、更新限時動態。粉絲慢慢累積到三千多人，但詢問量還是不穩。

有一次婚禮空檔，一位準新娘跟他聊天，說自己當初差點找另一位攝影師。原因不是對方拍得比較好，而是她在 Google 搜尋「台中婚紗攝影」時，先看到對方的網站。

那個網站很簡單：首頁有代表作、作品分成婚紗、訂婚、婚禮紀錄三類，方案價格寫得清楚，交件時間也有說明。她不用滑幾百張社群貼文，也不用私訊等回覆，就能在五分鐘內判斷這個攝影師是不是適合自己。

小柯的 Instagram 很漂亮，但像一條一直往下流的河。作品會被新貼文蓋掉，分類不清楚，價格找不到，客人也不知道哪幾張才是他的代表作。

網站則像一間整理好的工作室。客人走進來，第一眼看到的是你最想讓他看的作品，接著看到你的拍攝風格、服務流程、方案價格、聯絡方式，最後按下詢問。

這就是本講要解決的問題：攝影師不是只需要「作品展示」，而是需要一套可以被搜尋、可以被信任、可以被更新、可以被部署的數位店面。

前兩講已經讓你知道 AI 可以產出一個網頁。本講開始，網站不能只停在電腦裡。你要學會把它放到一個穩定的位置，留下版本紀錄，讓每一次修改都能追蹤，讓每一次更新都能自動上線。

## 2. 業態需求

攝影師作品集網站要解決的核心問題，不只是「把照片放上網」。它至少要解決五件事。

第一，讓客人快速看懂你的風格。

攝影師的社群貼文常常是時間流：今天一張婚紗、明天一張活動、後天一張家人日常。客人第一次認識你時，並不想考古。他需要在短時間內知道你最擅長什麼、風格是否穩定、你能不能拍出他想要的狀態。

第二，讓作品可以被分類。

婚紗、人像、商業攝影、活動紀錄、品牌形象照，背後的客人、決策流程與預算完全不同。全部混在一起，會讓客人難以判斷。作品集網站要把不同作品類型分開，讓客人直接進入自己的情境。

第三，讓搜尋引擎看得懂。

Instagram 貼文比較像平台內內容，搜尋引擎不一定能完整理解你的服務範圍。自有網站可以用標題、描述、區塊結構與文字說明，清楚告訴 Google：你是誰、在哪裡、提供什麼攝影服務、適合哪一種客人。

第四，讓信任資訊集中。

攝影師的信任不是只靠照片。客人還會看：

- 拍攝流程是否清楚。
- 方案與費用是否透明。
- 交件時間是否明確。
- 是否有真實案例與客戶回饋。
- 是否能看到攝影師本人與工作方式。
- 聯絡後會怎麼進行下一步。

第五，讓網站可以持續維護。

作品集最怕「建好就死掉」。攝影師每隔一段時間都會有新作品、新方案、新價格、新檔期。如果每次更新都要重新找人上傳，網站很快就會過期。

因此本講的技術主題是「版本控制與線上部署」。

| 需求 | 對應網站能力 | 對應技術 |
|---|---|---|
| 展示代表作 | 作品分類、精選照片、案例頁 | HTML（HyperText Markup Language，超文字標記語言）結構、圖片路徑、CSS（Cascading Style Sheets，階層式樣式表）版面 |
| 建立信任 | 流程、方案、交件時間、案例說明 | 內容架構、表單、CTA（Call to Action，行動呼籲） |
| 被搜尋 | 標題、描述、服務地區、文字內容 | SEO（Search Engine Optimization，搜尋引擎最佳化）基本設定 |
| 手機好讀 | 照片單欄、按鈕可點、文字可讀 | RWD（Responsive Web Design，響應式網頁設計）、media query |
| 持續更新 | 每次修改可追蹤、可回復 | Git（版本控制工具）、commit、repository |
| 公開上線 | 取得網址、可分享、可自訂網域 | Vercel、deployment、DNS（Domain Name System，網域名稱系統） |

這一講的重點不是把 GitHub 和 Vercel講成工程師工具，而是讓學生理解：一個正式網站要有保存、更新、發布與檢查流程。沒有這條流程，AI 產出的網頁只是檔案；有了這條流程，它才開始像一個可以經營的網站。

## 3. 網頁設計重點

### 3.1 網站風格

攝影師作品集網站的第一原則是：讓照片當主角。

設計不能比照片搶戲。過多動畫、複雜背景、過度裝飾的字體，都會降低作品本身的力量。好的作品集網站通常很克制：大圖、留白、清楚分類、穩定的文字層級，讓訪客把注意力放在照片與案例故事上。

不同攝影類型需要不同視覺語言：

| 攝影類型 | 視覺語言 | 首頁優先展示 |
|---|---|---|
| 婚紗 / 婚禮 | 溫度、情緒、信任、細節 | 代表作大圖、拍攝流程、交件承諾 |
| 商業攝影 | 專業、效率、品牌感、可交付 | 合作品牌、作品分類、報價入口 |
| 人像寫真 | 個性、氛圍、引導能力 | 風格系列、拍攝體驗、妝髮造型 |
| 活動紀錄 | 穩定、完整、現場掌握 | 活動案例、場景多樣性、交件速度 |

作品集頁面常見的錯誤是把照片一次放太多。訪客不是評審，不會看完兩百張。第一版可以先放 12 到 24 張代表作，每一類挑 4 到 8 張即可。網站要替客人做選擇，不要把選擇壓回客人身上。

攝影師網站的首頁第一屏應該同時回答三件事：

1. 你拍什麼類型？
2. 你的風格是什麼？
3. 我可以怎麼聯絡或看方案？

例如：

```text
台中婚紗與婚禮紀錄攝影，替你留下自然、不過度修飾的真實情緒。
```

這句比「用光影記錄生命中最美好的瞬間」更有用，因為它有地點、服務類型與風格判斷。

### 3.2 建立信任

攝影師作品集的信任模組要把「照片好看」往前推一步，回答客人真正擔心的問題。

| 信任模組 | 應該包含 | 作用 |
|---|---|---|
| 精選作品 | 作品類別、拍攝場景、代表作 | 讓客人快速判斷風格 |
| 案例說明 | 拍攝需求、場地條件、成品方向 | 證明你能處理不同情境 |
| 拍攝流程 | 詢問、溝通、拍攝、選片、交件 | 降低第一次委託的不安 |
| 方案與價格 | 基本方案、加選規則、交通費、交件時間 | 避免費用黑洞 |
| 攝影師介紹 | 經驗、工作方式、溝通風格 | 讓客人知道要跟誰合作 |
| 交件承諾 | 精修張數、原檔規則、交付時間 | 建立預期 |
| 聯絡表單 | 拍攝類型、日期、地點、需求說明 | 讓詢問更具體 |

這裡有一個重要原則：不要用假作品、假客戶、假評價。

AI 產生攝影師網站時，很容易自動補上「服務過上百組新人」「獲得多項國際獎項」「客戶一致好評」這類文字。如果這些不是事實，要刪掉。攝影師的作品集網站最怕信任破口，一旦訪客覺得文字在包裝，照片也會一起變得不可信。

攝影師也要小心圖片權利。網站上放的照片必須是自己有權使用的作品，並且避開不適合公開的人像、活動或商業素材。若客戶要求不公開，就不要放進作品集。作品集要建立信任，不是消耗信任。

### 3.3 引導行動

攝影師作品集的轉換路徑通常不是「立刻付款」，而是「讓客人願意詢問」。

基本路徑可以設計成：

```text
首頁代表作 → 作品分類 → 拍攝流程 → 方案與價格 → 詢問表單
```

每一段都要有清楚的下一步：

| 區塊 | 訪客心裡的問題 | 下一步 |
|---|---|---|
| 首頁 | 這是不是我要的風格？ | 看作品 |
| 作品集 | 他拍得穩不穩？ | 看流程或方案 |
| 案例說明 | 他懂不懂我的情境？ | 看服務內容 |
| 方案 | 我負擔得起嗎？有沒有隱藏費用？ | 詢問檔期 |
| 表單 | 我該提供什麼資訊？ | 送出需求 |

表單欄位不宜太少。只留姓名與電話，攝影師還要來回追問；欄位太多，又會讓客人中途放棄。第一版建議包含：

- 姓名。
- Email 或電話。
- 拍攝類型。
- 預計日期。
- 拍攝地點。
- 預算區間。
- 想說明的需求。

手機版的行動按鈕尤其重要。訪客可能在 LINE 群組裡收到作品集連結，打開看完後想直接詢問。這時「詢問檔期」「查看方案」「回到作品集」按鈕都要夠明顯、夠大、夠容易點。

## 4. 功能與資料模型

攝影師作品集網站看起來是展示型網站，但底層仍然有清楚的資料模型。先把資料拆出來，AI 生成的網站才不會只是一張漂亮首頁。

建議頁面區塊如下：

| 區塊 | 功能 | 必要內容 |
|---|---|---|
| Hero 首頁 | 建立第一印象 | 攝影師名稱、服務類型、主張、代表照片、CTA |
| Portfolio 作品集 | 展示能力 | 類別、照片、標題、拍攝場景、替代文字 |
| Case Study 案例 | 建立情境信任 | 客戶需求、拍攝條件、成果說明 |
| Services 方案 | 說明服務範圍 | 方案名稱、內容、價格區間、加選規則 |
| Process 流程 | 降低不安 | 詢問、溝通、拍攝、修圖、交件 |
| About 攝影師介紹 | 建立人感 | 自我介紹、工作方式、照片 |
| Inquiry 詢問表單 | 收集需求 | 姓名、聯絡方式、日期、類型、地點、留言 |
| Footer 頁尾 | 補齊資訊 | Email、社群、服務地區、版權聲明 |

可以把資料拆成幾個物件：

| 資料物件 | 欄位 | 範例 |
|---|---|---|
| `PhotographerProfile` | 名稱、地區、服務類型、品牌主張 | 小柯攝影、台中、婚紗 / 婚禮 |
| `PortfolioCategory` | 分類名稱、排序、說明 | 婚紗、婚禮紀錄、形象照 |
| `PortfolioItem` | 標題、分類、圖片路徑、地點、年份、alt 文字 | 海邊婚紗、wedding-01.jpg |
| `ServicePackage` | 方案名稱、內容、價格、交件時間、加選規則 | 婚禮紀錄 8 小時 |
| `InquiryField` | 欄位名稱、類型、是否必填、說明 | 拍攝日期、date、必填 |
| `DeploymentProject` | repo 名稱、production branch、部署網址、網域 | photo-portfolio、main、vercel.app |

本講開始要加入一個新的工程概念：網站不是只有畫面，還有檔案與發布流程。

最小檔案結構可以是：

```text
photo-portfolio/
  index.html
  images/
    hero.jpg
    wedding-01.jpg
    portrait-01.jpg
    event-01.jpg
  README.md
```

`index.html` 是首頁。`images/` 放圖片。`README.md` 用來記錄這個網站是什麼、怎麼更新、上線網址在哪裡。這個資料夾放到 GitHub 後，就成為一個 repository。

幾個名詞要先釐清：

| 名詞 | 白話意思 | 在本講的角色 |
|---|---|---|
| Repository | 一個被 GitHub 保存與追蹤的專案資料夾 | 保存作品集網站 |
| Commit | 一次有說明的存檔紀錄 | 記錄「更新首頁照片」「修改方案價格」 |
| Branch | 版本分支 | 第一版可只使用 `main` |
| Deployment | 某一次上線結果 | Vercel 依照 repository 建出的網站 |
| Production | 正式公開版本 | 客人會看到的網站 |
| Domain | 網站地址 | `kuophotography.com` |
| DNS | 網域指向設定 | 把自訂網域指到 Vercel |

典型更新流程如下：

```text
修改 index.html 或圖片
→ commit 到 GitHub
→ Vercel 偵測到 main branch 更新
→ 自動建立新 deployment
→ 正式網址顯示新版本
```

這條線就是本講的技術核心。學生不需要一開始就精通 Git 指令，但必須知道每個環節在做什麼。否則網站一旦上線，後續每次改照片、改價格、改文案，都會變成混亂的人工搬運。

## 5. AI 協作實作

本講的 AI 協作分成兩層。第一層是讓 AI 產出作品集網站；第二層是讓 AI 幫你檢查網站是否適合部署。

先使用這份提示詞產出第一版：

```text
你是網頁設計老師與前端工程師。
請幫我製作一個攝影師作品集網站，並讓它適合上傳到 GitHub 和 Vercel 部署。

【攝影師資料】
名稱：小柯攝影工作室
地區：台中
服務類型：婚紗、婚禮紀錄、個人形象照
品牌主張：自然、不過度修飾，重視情緒與現場關係
主要客群：準新人、小型品牌主、需要形象照的專業工作者

【網站目標】
1. 讓第一次來的客人快速看懂攝影風格
2. 讓作品依照類型清楚分類
3. 說明拍攝流程、方案內容、交件時間與加選規則
4. 引導客人填寫詢問表單
5. 讓網站可部署為單頁靜態網站

【必要區塊】
1. 首頁主視覺：攝影師名稱、服務類型、一句話主張、兩個按鈕「看作品」「詢問檔期」
2. 作品分類：婚紗、婚禮紀錄、形象照，每類至少 4 張 placeholder
3. 案例說明：每一類用 2 到 3 句說明拍攝情境
4. 服務方案：方案名稱、內容、價格區間、交件時間、加選規則
5. 拍攝流程：詢問、溝通、拍攝、精修、交件
6. 關於我：攝影師介紹與工作方式
7. 詢問表單：姓名、聯絡方式、拍攝類型、日期、地點、預算、留言
8. 頁尾：Email、Instagram、服務地區、版權聲明

【視覺風格】
照片優先、留白、專業、安靜、不要過度裝飾。
背景可以用白色、深灰或低飽和米色。
按鈕要清楚，但不要太花。

【技術要求】
1. 請輸出完整單一 HTML 檔案，可存成 index.html。
2. CSS 寫在同一個檔案內。
3. 使用語意化 HTML，例如 header、main、section、footer。
4. 圖片先用 placeholder，並用註解標出日後要替換的圖片檔名。
5. 每張圖片都要有 alt 文字。
6. 使用響應式設計，桌機作品集可三欄，手機改為單欄。
7. 文字在手機上至少要容易閱讀，按鈕要容易點。
8. 表單不需要真的送出到後端，但要有清楚欄位與送出按鈕。
9. 不要編造得獎紀錄、客戶評價或合作品牌。
10. 請在 HTML title 與 meta description 放入適合 SEO 的文字。

請直接輸出完整程式碼，不要省略 CSS，也不要只給說明。
```

AI 產出後，先做四種檢查。

第一，檢查內容是否真實。刪掉 AI 自動編出來的評價、品牌合作、獎項與客戶名稱。

第二，檢查 HTML 結構。至少要能看到 `header`、`main`、多個 `section`、`footer`，作品集不要全部塞在沒有意義的單一大區塊裡。

第三，檢查 CSS 是否包含手機版規則。可以搜尋 `@media`，看看 AI 是否有針對小螢幕調整欄位、圖片與按鈕。如果沒有，就請 AI 補上：

```text
請補上手機版 RWD 規則。桌機作品集維持三欄，平板兩欄，手機單欄。表單欄位在手機上全部改成單欄，按鈕高度至少 44px。
```

第四，檢查圖片路徑。若 HTML 寫的是：

```html
<img src="images/wedding-01.jpg" alt="台中戶外婚紗作品">
```

那你的 GitHub repository 裡就必須真的有：

```text
images/wedding-01.jpg
```

圖片路徑錯誤，是作品集部署後最常見的失敗原因之一。網站本機開起來正常，不代表上線後正常。大小寫、資料夾名稱、檔案副檔名都要一致。

接著建立 GitHub repository。

建議 repository 名稱使用英文與連字號，例如：

```text
photo-portfolio
kuo-photography
wedding-portfolio
```

第一次上傳至少包含：

```text
index.html
images/
README.md
```

第一次 commit message 可以寫：

```text
Add initial photography portfolio site
```

之後每次修改都要讓 commit message 說明你改了什麼，例如：

```text
Update wedding portfolio images
Adjust mobile layout for inquiry form
Revise service package pricing
```

commit message 不只是給工程師看的。三個月後你要回想「我什麼時候改過價格」，commit 紀錄就是答案。

再把 repository 連到 Vercel。

依照 Vercel 官方流程，從 dashboard 建立新 project，選擇 Git provider，匯入剛建立的 GitHub repository。對這種單頁靜態網站來說，通常不需要複雜 build 指令；Vercel 會把可部署檔案發布成公開網址。

部署完成後，你會得到一個像這樣的網址：

```text
https://photo-portfolio.vercel.app
```

請立刻做三件事：

1. 用電腦打開，看首頁與作品集是否正常。
2. 用手機打開，看圖片、按鈕、表單是否正常。
3. 回 GitHub 改一小段文字，commit 後觀察 Vercel 是否自動重新部署。

這第三步很重要。它讓學生看見 GitHub 與 Vercel 的角色分工：GitHub 保存新版本，Vercel 讀到新版本後自動產生新部署。

如果要設定自訂網域，流程是：

```text
購買網域
→ 在 Vercel 專案加入 domain
→ 依照 Vercel 顯示的 DNS 記錄，到網域商後台設定
→ 等待 DNS 生效
→ 用正式網址檢查網站
```

常見錯誤與排查方式：

| 問題 | 常見原因 | 檢查方式 |
|---|---|---|
| 網站 404 | repository 沒有 `index.html` 或部署根目錄錯誤 | 確認檔案位置 |
| 圖片不見 | 圖片路徑錯、大小寫不同、檔案沒上傳 | 對照 HTML `src` 與 `images/` |
| 手機版跑版 | CSS 沒有 RWD 或欄寬固定 | 搜尋 `@media`，縮小瀏覽器測試 |
| 改完沒更新 | commit 沒成功或 Vercel 還在部署 | 看 GitHub commit 與 Vercel deployments |
| 自訂網域失效 | DNS 尚未生效或記錄填錯 | 回 Vercel domain 頁面看驗證狀態 |

這一講的實作不是只教「按哪個按鈕」。它要讓學生理解一條完整網站生命線：

```text
AI 生成 → 本機檢查 → GitHub 保存 → Vercel 部署 → 手機驗收 → 持續更新
```

## 6. 上線與延伸

上線前請依照下面的檢查清單逐項確認。

內容檢查：

- 首頁是否清楚說明攝影師姓名、地區與服務類型。
- 作品是否有分類，而不是全部混在一起。
- 方案價格或價格區間是否清楚。
- 交件時間、精修張數、加選規則是否清楚。
- 是否刪除 AI 編造的評價、獎項與合作品牌。

技術檢查：

- GitHub repository 是否包含 `index.html`。
- 圖片是否放在正確資料夾，路徑與大小寫一致。
- Vercel 網址是否可以公開打開。
- 修改 GitHub 後，Vercel 是否會自動產生新 deployment。
- 自訂網域若已設定，是否能正常連到網站。

手機版檢查：

- 作品集在手機上是否單欄呈現。
- 文字不用放大也能閱讀。
- 詢問按鈕與表單欄位是否容易點。
- 首頁主視覺是否沒有裁掉重要人物或臉部。
- 表單送出按鈕是否清楚可見。

SEO（Search Engine Optimization，搜尋引擎最佳化）與分享檢查：

- `<title>` 是否包含攝影師名稱、服務類型與地區。
- `meta description` 是否說明服務內容。
- 圖片 `alt` 是否描述作品，而不是空白。
- IG、名片、Email 簽名檔是否放上正式網址。
- Google 商家檔案或作品平台是否能連回網站。

本講可以延伸的功能包括：

- 建立獨立案例頁，讓每一組作品有完整故事。
- 加入簡單 CMS，讓攝影師不用改 HTML 也能更新作品。
- 串接表單服務，讓詢問內容寄到 Email 或 Google Sheet。
- 壓縮圖片與使用 WebP，提升載入速度。
- 加入 Open Graph 圖片，讓 LINE 或 Facebook 分享時顯示漂亮預覽。
- 用自訂網域建立更正式的品牌入口。

給學生的實作作業：

1. 建立一個攝影師作品集 `index.html`。
2. 至少放入三個作品分類，每類至少四張 placeholder 或授權圖片。
3. 建立 GitHub repository 並完成第一次 commit。
4. 用 Vercel 部署上線，取得公開網址。
5. 用手機打開網址，截圖記錄首頁、作品集、詢問表單三個畫面。
6. 修改一段首頁文案，commit 後確認 Vercel 自動更新。
