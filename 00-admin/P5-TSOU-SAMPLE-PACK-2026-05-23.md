## 二、樣章一：第 01 講「咖啡廳網站」

### 1. 本章定位

第 01 講是「架構與語氣樣章」，主要任務不是展示最深技術，而是重新定調全書：

- AI 協作不是「不用理解網頁設計」。
- 學生不用先成為工程師，但要理解網站結構、資訊架構、RWD、部署與驗收的基本概念。
- 每章的產業情境要接回具體網站決策，而不是停在故事或商業洞察。

### 2. 本章採用的六段式結構

| 段落 | 寫作重點 |
|---|---|
| 破冰案例 | 用自由工作者找咖啡廳的情境，快速帶出客人真正擔心的問題 |
| 業態需求 | 說明咖啡廳網站要解決找得到、判斷快、建立信任、引導行動 |
| 網頁設計重點 | 拆成網站風格、建立信任、引導行動 |
| 功能與資料模型 | 拆出菜單、空間設備、營業資訊、聯絡入口等資料物件 |
| AI 協作實作 | 提供可複製 prompt、修改句、手機版檢查方式 |
| 上線與延伸 | 收斂成本機檢查、手機版驗收、GitHub/Vercel 上線與後續延伸 |

### 3. 給老師看的重點

這一章可以用來判斷：

- 六段式結構是否清楚。
- 產業故事是否已經壓縮到合理篇幅。
- 「網頁設計重點」是否能清楚說明視覺語言、信任模組、轉換路徑。
- 「功能與資料模型」是否足以讓授課老師把網站元件講清楚。
- AI prompt 是否比原本「丟一句話產生網站」更適合作為教學材料。

### 4. 目前狀態

第 01 講已完成樣章，狀態為「待老師確認」。

## 三、樣章二：第 03 講「攝影師作品集」

### 1. 本章定位

第 03 講是「技術深度樣章」，用來回答老師先前指出的問題：章節副標題是「版本控制與線上部署」，內文就必須真正教出版本控制與部署，而不是只在最後簡短帶過。

這章的修稿目標是讓學生理解一個正式網站的生命線：

```text
AI 生成 → 本機檢查 → GitHub 保存 → Vercel 部署 → 手機驗收 → 持續更新
```

### 2. 已補強的技術內容

| 技術主題 | 教材中的處理方式 |
|---|---|
| GitHub | 說明 repository 是保存網站檔案與修改紀錄的地方 |
| commit | 說明每次修改都要留下有意義的版本紀錄 |
| Vercel | 說明 deployment 與 production 網站的關係 |
| 自動部署 | 說明 GitHub 更新後，Vercel 如何自動建立新版網站 |
| 圖片路徑 | 說明 `images/wedding-01.jpg` 這類路徑必須和檔案一致 |
| RWD | 說明桌機三欄、平板兩欄、手機單欄的檢查方式 |
| SEO | 說明 title、meta description、alt 文字對搜尋與分享的作用 |
| 自訂網域 / DNS | 說明從購買網域到 Vercel domain 設定的流程 |
| 錯誤排查 | 補上 404、圖片不見、手機跑版、部署未更新、DNS 未生效等常見問題 |

### 3. 給老師看的重點

這一章可以用來判斷：

- 技術深度是否比目前版本更接近《網頁設計》課程。
- GitHub / Vercel / DNS 這類概念是否寫得夠清楚，但不會太工程化。
- 是否能讓授課老師拿來講「網站如何從檔案變成可公開訪問的網址」。
- 是否能讓學生理解網站上線後還需要更新、驗收與排錯。

### 4. 目前狀態

第 03 講已完成技術深度樣章，狀態為「待老師確認」。

## 四、功能型樣章方向：第 11 講「民宿訂房網站」

### 1. 為什麼先做技術拆解

第 11 講不是單純形象網站，而是功能型網站。它需要處理房型、日期、房況、訂單、付款、取消、後台管理等問題。如果直接寫正文，很容易變成故事很完整，但技術骨架不清楚。

因此目前先完成技術拆解備料，後續正式修稿時再轉成教材語氣。

### 2. 後續會補進正文的技術方向

| 面向 | 後續教材會說明 |
|---|---|
| 頁面模組 | 首頁、房型列表、房型詳情、日期選擇、訂房流程、付款結果、訂單查詢 |
| 資料模型 | 房型、房間、日期房況、訂單、付款、旅客資料 |
| 房況狀態 | 可訂、保留中、已預訂、封鎖、清潔、維修 |
| 訂單狀態 | 待付款、已確認、已取消、已退款、已完成 |
| 金流流程 | 付款頁、付款成功、付款失敗、金流通知、退款 |
| 後台管理 | 房況管理、訂單管理、價格管理、黑名單日期 |
| 防超賣 | 同一間房同一天只能被一筆有效訂單占用 |
| 取消規則 | 依入住日前天數設計不同退款比例 |

### 3. 給老師看的重點

這一章後續會用來示範「功能型網站」的教法：

- 不只講網站畫面，也講背後資料與狀態。
- 不要求學生立刻寫出完整訂房系統，但要知道訂房網站不是一張表單。
- 把業態問題轉成頁面、資料欄位、狀態流程與後台需求。
- 讓老師可以帶學生理解「網站功能」和「商業流程」如何對應。

### 4. 目前狀態

第 11 講正文尚未重寫。目前先完成技術拆解摘要，避免在老師確認方向前直接改動全文。
