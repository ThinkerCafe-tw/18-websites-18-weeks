## 主文

### 4-1 從展示到功能：網站開發的分水嶺

在前三講中，我們製作的都是「展示型網站」：展示資訊、照片、文字。但從這一講開始，我們要進入「功能型網站」的領域 — 讓網站不只是「看」，更能「用」。

**一、展示型 vs 功能型網站**

（一）展示型網站的特徵

前三講製作的咖啡廳、健身房、攝影師網站都屬於這一類：

1. **主要目的**：傳遞資訊
2. **互動方式**：瀏覽、閱讀、聯絡
3. **技術需求**：HTML + CSS + 基本 JavaScript
4. **資料處理**：靜態內容，不需要後端資料庫

**常見案例**：
- 公司形象網站
- 產品介紹頁面
- 個人部落格
- 作品集網站

（二）功能型網站的特徵

從這一講開始的診所預約網站屬於這一類：

1. **主要目的**：完成特定任務（預約、購買、登記）
2. **互動方式**：填寫表單、選擇選項、提交資料
3. **技術需求**：前端 + 後端 + 資料庫
4. **資料處理**：動態內容，需要儲存和讀取資料

**常見案例**：
- 預約系統（診所、美容院、餐廳）
- 電商網站
- 會員系統
- 報名平台

（三）複雜度的躍升

功能型網站的開發難度顯著提高：

| 維度 | 展示型 | 功能型 |
|------|--------|--------|
| 開發時間 | 數小時 | 數天到數週 |
| 技術棧 | 前端 | 前後端 + 資料庫 |
| 測試複雜度 | 低 | 高（需考慮各種情境）|
| 維護成本 | 低 | 高（需處理使用者資料）|
| 接案價格 | 5千-2萬 | 3萬-10萬+ |

**但好消息是**：有了 AI 協作，這個複雜度大幅降低。原本需要一個月的預約系統，現在可能只需要 2-3 天就能完成基本版本。

### 4-2 診所預約系統的商業需求分析

**二、為什麼診所需要線上預約系統？**

（一）傳統預約方式的痛點

**診所端的困擾**：
1. **電話佔線**：櫃檯人員一直在接電話，無法專心服務現場病患
2. **人工記錄易出錯**：手寫預約本容易塗改、遺漏、重複預約
3. **時段管理混亂**：不知道哪些時段還有空位
4. **提醒困難**：要一個一個打電話提醒病患

**病患端的困擾**：
1. **電話難打通**：上班時間打電話預約，診所電話常忙線
2. **營業時間限制**：只能在診所上班時間打電話，不方便
3. **資訊不透明**：不知道哪些時段還能約
4. **容易忘記**：預約後沒有提醒，容易錯過看診時間

（二）線上預約系統的價值

**對診所的好處**：
- ✅ 減少電話量，櫃檯人員更有效率
- ✅ 自動管理時段，避免重複預約
- ✅ 累積病患資料，建立 CRM 系統
- ✅ 減少爽約率（系統自動提醒）
- ✅ 提升專業形象

**對病患的好處**：
- ✅ 24 小時都能預約
- ✅ 清楚看到可預約時段
- ✅ 立即收到確認通知
- ✅ 自動提醒看診時間
- ✅ 方便變更或取消預約

**三、預約系統的核心功能拆解**

（一）必備功能清單

一個基本的診所預約系統需要包含：

**1. 科別與醫師選擇**
- 顯示診所提供的科別（如：內科、牙科、皮膚科）
- 每個科別有哪些醫師
- 醫師的簡介、專長、照片

**2. 日期與時段選擇**
- 顯示日曆，讓病患選擇日期
- 該日期的可預約時段（上午、下午、晚上）
- 標示「已滿」的時段（不可選）
- 標示「休診」日（週日或國定假日）

**3. 病患資料填寫**
- 姓名、電話、Email
- 就診原因（簡述症狀）
- 首次就診 / 複診

**4. 預約確認**
- 顯示預約摘要讓病患檢查
- 送出後顯示成功訊息
- 發送確認 Email 或簡訊

**5. 預約管理（後台）**
- 診所人員可以查看所有預約
- 可以手動新增、修改、取消預約
- 查看每日預約狀況
- 匯出報表

（二）進階功能（選配）

如果預算充足，可以加入：

- 線上掛號費支付
- 病歷查詢
- 藥品配送
- 視訊看診
- 滿意度調查
- 會員積分系統

但對於初階預約系統，我們先聚焦在**必備功能**即可。

### 4-3 設計使用者流程：從進入到完成預約

**四、病患端的使用者旅程（User Journey）**

（一）完整的預約流程

讓我們詳細拆解一個病患如何完成線上預約：

**Step 1：進入網站**
```
病患打開診所網站 → 看到「立即預約」按鈕 → 點擊
```

**Step 2：選擇科別與醫師**
```
顯示科別列表（內科、牙科、皮膚科）
↓ 點擊「內科」
顯示內科醫師列表（王醫師、李醫師、陳醫師）
↓ 點擊「王醫師」
```

**Step 3：選擇日期**
```
顯示日曆（當月 + 下個月）
↓ 點擊「2025/11/20（三）」
檢查該日期是否有診（如果休診，顯示提示）
```

**Step 4：選擇時段**
```
顯示該日的時段：
[ ] 上午 09:00-12:00 (已滿)
[✓] 下午 14:00-17:00 (尚有名額)
[✓] 晚上 18:00-21:00 (尚有名額)
↓ 選擇「下午時段」
```

**Step 5：填寫基本資料**
```
姓名：______
電話：______
Email：______
症狀簡述：______
□ 首次就診  ☑ 複診
```

**Step 6：確認預約**
```
顯示預約摘要：
─────────────
科別：內科
醫師：王醫師
日期：2025/11/20（三）
時段：下午 14:00-17:00
姓名：陳小明
電話：0912345678
─────────────
[確認預約] [返回修改]
```

**Step 7：完成預約**
```
✓ 預約成功！
您的預約號碼是：A123456

我們已發送確認簡訊到您的手機。
看診當天請提前 10 分鐘到診。

[查看預約紀錄] [返回首頁]
```

（二）每個步驟的設計重點

**選擇科別頁面**：
- 用卡片式設計，每個科別一張卡片
- 卡片包含：科別名稱、圖示、簡短說明
- 點擊後才展開該科別的醫師

**日曆介面**：
- 今天用特殊顏色標示
- 過去的日期不可選（灰色）
- 休診日標示「休診」
- 已約滿的日期標示「額滿」

**時段選擇**：
- 用大按鈕（至少 60px 高度），方便手機點擊
- 已滿的時段顯示為灰色且不可選
- 顯示「尚有 3 個名額」這類資訊

**表單設計**：
- 電話欄位自動切換到數字鍵盤（手機上）
- Email 欄位自動切換到 Email 鍵盤
- 必填欄位標示紅色星號（*）
- 即時驗證（輸入格式錯誤立即提示）

### 4-4 與 AI 協作打造預約系統

**五、從粗糙到完美：迭代優化的藝術**

很多初學者會想一次就做出完美的系統，但實際上，與 AI 協作最有效的方式是**逐步迭代**。讓我們看看如何用四輪對話，從陽春版進化到專業系統。

（一）第一版：快速建立原型（5 分鐘）

**你的需求（很粗糙）**：
```
你：我想做一個診所預約網站
```

**AI 的回應**：
> Claude 會給你一個超級陽春的版本 — 可能只有一個簡單的表單，讓人填寫姓名、電話、想預約的時間。

**結果評估**：
- ✅ 有了基本架構
- ❌ 沒有日曆選擇
- ❌ 沒有時段管理
- ❌ 介面很陽春

**學到什麼**：第一版不用完美，先讓 AI 理解大方向

（二）第二版：補充核心功能（10 分鐘）

**你的需求（更具體）**：
```
你：需要加上這些功能：

1. 科別選擇：內科、牙科、皮膚科
2. 醫師選擇：每個科別顯示 2-3 位醫師
3. 日曆介面：顯示當月和下個月
4. 時段選擇：上午（9-12）、下午（14-17）、晚上（18-21）
5. 病患資料表單：姓名、電話、Email、症狀簡述
6. 確認頁面：顯示預約摘要
```

**AI 的回應**：
> Claude 會根據你的清單，加上這些功能模組

**結果評估**：
- ✅ 功能完整了
- ✅ 流程清楚了
- ❌ 但細節不夠好（按鈕太小、沒有狀態標示）
- ❌ 使用體驗還不夠好

**學到什麼**：功能清單要明確，一次說清楚需要什麼

（三）第三版：優化使用體驗（10 分鐘）

**你的需求（注重細節）**：
```
你：再做這些調整：

1. 日曆部分：
   - 週日標示「休診」且不可選
   - 過去的日期顯示灰色且不可選
   - 今天用特殊顏色標示（如橘色邊框）

2. 時段部分：
   - 按鈕要大一點（至少 60px 高），方便手機點擊
   - 已滿的時段顯示「已額滿」且變成灰色
   - 顯示「尚有 3 個名額」這類提示

3. 時段管理邏輯：
   - 每個時段最多 10 人
   - 超過 10 人就自動顯示「已額滿」
```

**AI 的回應**：
> Claude 會優化介面細節和互動邏輯

**重點**：注意你**不需要告訴 AI 如何寫程式**，只要描述你想要的結果：
- ❌ 不要說：「用 count() 函數計算預約人數...」
- ✅ 要說：「每個時段最多 10 人，超過就顯示已滿」

**結果評估**：
- ✅ 使用體驗提升
- ✅ 手機上好操作
- ✅ 資訊清楚明確

（四）第四版：完善最後細節（5 分鐘）

**你的需求（打磨細節）**：
```
你：最後調整：

1. 表單驗證：
   - 電話要確認是 10 碼數字
   - Email 要有 @ 符號
   - 如果格式錯誤，在欄位下方顯示紅色提示文字

2. 預約成功後：
   - 顯示「預約成功！」的確認頁面
   - 顯示預約編號（例如：A123456）
   - 顯示完整的預約資訊讓病患確認

3. 手機版優化：
   - 日曆改成左右滑動，不要上下捲動
   - 表單的輸入框在手機上點擊時，鍵盤不會遮住內容
```

**AI 的回應**：
> Claude 會處理這些最後的優化

**完成**：🎉 專業級預約系統誕生了！

**六、關鍵對話技巧：如何「說人話」而非「說程式」**

（一）描述需求，而非技術實現

很多人會不自覺地用技術語言跟 AI 溝通，但其實不需要：

| ❌ 技術化說法 | ✅ 需求化說法 |
|------------|-------------|
| "需要一個資料庫 table 存放預約記錄" | "需要儲存所有預約資料" |
| "用 count() 計算預約人數" | "每個時段最多 10 人" |
| "加上 validation function" | "電話要 10 碼，Email 要有 @" |
| "用 state management 處理" | "已滿的時段要自動變灰色" |

**核心概念**：AI 會處理「如何做」，你只需要說清楚「做什麼」

（二）用具體的例子說明

**模糊的說法**：
> "時段按鈕要好按一點"

**具體的說法**：
> "時段按鈕要大一點，至少 60px 高度，方便手機上用手指點擊"

**為什麼要具體？**
- AI 需要明確的標準才能執行
- 避免來回修改浪費時間

（三）分批次溝通，不要一次塞太多

**不好的做法**（一次說太多）：
```
你：我要做一個預約網站，要有科別選擇、醫師選擇、日曆、時段、
    表單驗證、手機版適配、已滿標示、休診標示、確認頁面、
    預約編號、Email 通知、簡訊提醒、管理後台...
```
→ AI 可能會遺漏某些功能，或做得不夠好

**好的做法**（分階段溝通）：
```
第一輪：建立基本架構（科別、醫師、日曆、時段、表單）
第二輪：優化體驗（按鈕大小、狀態標示、提示訊息）
第三輪：完善細節（表單驗證、確認頁面、手機版）
```
→ 每一輪都能確保品質

**七、預約系統的核心邏輯（給 AI 的指令範本）**

當你要做類似的預約系統時，可以直接參考這個範本：

```
你：我要做一個 [行業] 預約網站，需要以下功能：

【基本資訊】
- 行業：[診所/美容院/健身房...]
- 服務項目：[科別/課程/療程...]
- 服務人員：[醫師/教練/美容師...]

【預約流程】
1. 選擇服務項目
2. 選擇服務人員
3. 選擇日期（顯示 [本月 + 下月] 的日曆）
4. 選擇時段（[上午/下午/晚上]，每時段 [X] 人）
5. 填寫資料（姓名、電話、Email、[其他欄位]）
6. 確認預約

【時段規則】
- 每時段上限 [X] 人，超過顯示「已額滿」
- [週日/特定日] 標示「休診」
- 過去的日期不可選

【視覺風格】
- 配色：[專業/溫暖/活力]
- 主色調：[藍色/綠色/...]
- 感覺：[醫療專業感/溫馨親切感/...]

請使用 React + Tailwind CSS 製作
```

**提示**：把 [ ] 裡的內容替換成你的需求，就能快速建立各行業的預約系統！

### 4-5 診所網站的設計原則

**七、醫療網站的特殊考量**

（一）建立信任感的設計元素

醫療是高度信任導向的服務，網站設計必須傳達：

**1. 專業性**
- 配色：藍色（信任）、綠色（健康）、白色（乾淨）
- 字體：選擇易讀的正體字，避免花俏字型
- 版面：簡潔、清晰，資訊層級分明

**2. 安全性**
- HTTPS 加密（網址前有鎖頭圖示）
- 隱私政策聲明（說明如何保護病患資料）
- 安全標章（如：衛福部認證）

**3. 專業團隊展示**
- 醫師照片（穿白袍的正式照）
- 學經歷介紹（畢業學校、專科證書）
- 專長領域（擅長治療哪些疾病）

（二）必備的法律聲明

醫療網站需要包含：

**免責聲明**：
> "本網站資訊僅供參考，不能取代醫師的診斷與治療。如有身體不適，請親自就醫。"

**隱私權政策**：
> 說明收集哪些個人資料、如何使用、如何保護。

**醫療廣告相關規定**：
- 不能宣稱療效（如「保證根治」）
- 不能使用誇大字眼
- 需標示診所開業執照字號

**八、診所網站的頁面架構**

（一）推薦的完整架構

```
診所網站架構：

├── 首頁
│   ├── 診所簡介（1-2 段文字）
│   ├── 服務項目（卡片式展示）
│   ├── 醫療團隊（醫師頭像 + 名字）
│   └── 立即預約按鈕（醒目）
│
├── 關於我們
│   ├── 診所理念
│   ├── 環境照片（診間、候診區）
│   └── 交通位置
│
├── 醫療團隊
│   ├── 每位醫師的詳細介紹
│   ├── 學經歷
│   ├── 專長領域
│   └── 看診時間
│
├── 服務項目
│   ├── 各科別的詳細說明
│   ├── 常見疾病介紹
│   └── 相關衛教資訊
│
├── 線上預約 ⭐ 核心功能
│   └── （完整的預約流程）
│
├── 看診時間表
│   ├── 週一到週日的門診時間
│   └── 各醫師的看診時段
│
├── 就診須知
│   ├── 初診注意事項
│   ├── 健保相關規定
│   └── 常見問題 FAQ
│
└── 聯絡我們
    ├── 地址、電話
    ├── Google Maps 嵌入
    └── 聯絡表單
```

（二）首頁的資訊優先順序

依照使用者最關心的順序排列：

**九、AI 協作的關鍵心法**

（一）不要怕「要求太多」

很多人會擔心：「這樣一直要求 AI 修改，會不會很麻煩？」

**答案是：不會！**

- AI 不會累、不會不耐煩
- 每一次修改都是在優化
- 分次說明比一次說清楚更有效

（二）用「範例」勝過「文字」

與其說：
> "按鈕要大一點"

不如說：
> "按鈕要 60px 高，像 [上傳參考截圖] 這個大小"

（三）善用「對比」來溝通

```
不要：按鈕太小
要：按鈕從現在的 40px 改成 60px

不要：顏色不對
要：從淺藍色 #93c5fd 改成深藍色 #2563eb
```

（四）記錄「可重複使用的指令」

當你做完一個診所預約系統，下次要做美容院預約時：
- 把成功的指令範本存下來
- 只需要改行業名稱、服務項目
- 10 分鐘就能產出新系統！

**這就是 AI 時代的「經驗累積」**

## 重要名詞

◆ **功能型網站（Functional Website）**：以完成特定任務為主要目的的網站，如預約、購物、報名等，需要處理使用者輸入的資料。

◆ **使用者旅程（User Journey）**：使用者從進入網站到完成目標的完整流程，用於分析和優化使用體驗。

◆ **時段管理（Time Slot Management）**：控制和分配可預約時間段的系統，包括設定上限、防止衝突、標示休息日等。

◆ **預約衝突（Booking Conflict）**：當系統允許多人預約同一時段，超過容納上限時產生的問題。

◆ **表單驗證（Form Validation）**：檢查使用者輸入的資料是否符合格式要求，如電話號碼、Email 格式等。

◆ **localStorage**：瀏覽器提供的本地儲存空間，可以暫存資料在使用者的裝置上，但關閉瀏覽器後資料可能會消失。

◆ **CRM 系統（Customer Relationship Management）**：客戶關係管理系統，用於記錄和管理客戶資訊、互動歷史等。

◆ **原型（Prototype）**：產品的初步版本，用於測試核心功能和概念，不一定包含所有細節功能。
