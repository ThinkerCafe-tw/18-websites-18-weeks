## 主文

### 3-1 從本地到雲端：現代網站部署思維

在前兩講中，我們學會了如何用 AI 協作快速建立網站。但你可能會發現一個問題：這些網站都只存在你的電腦裡，別人看不到。這一講，我們要解決這個問題，讓你的作品真正「上線」，讓全世界都能看到。

**一、為什麼需要版本控制？**

（一）網站開發的現實困境

想像以下場景：

**場景一：災難性的修改**
> 你在修改網站時，不小心刪掉了重要的程式碼。你想復原，但已經按下「儲存」了。你沒有備份，辛苦做的網站就這樣毀了。

**場景二：多版本管理混亂**
> 你的電腦裡有：`網站_最終版.zip`、`網站_最終版2.zip`、`網站_真的最終版.zip`、`網站_客戶要的版本.zip`⋯⋯你已經記不清楚哪個才是最新的了。

**場景三：協作困難**
> 你和設計師朋友一起做網站。你改了首頁，他改了關於頁面。你們透過 LINE 傳檔案，結果互相覆蓋了對方的修改，又要重做一次。

這些問題，都可以透過「版本控制系統」解決。

（二）版本控制的三大核心價值

1. **時光機功能**：隨時回到過去任何一個版本
   - 改壞了？退回上一版
   - 想比較新舊差異？一鍵對比
   - 客戶說「還是改回之前的版本」？沒問題

2. **協作不衝突**：多人同時編輯不會互相覆蓋
   - 自動合併不衝突的修改
   - 衝突時會提醒，讓你手動決定要保留哪一版
   - 清楚看到誰改了什麼

3. **雲端備份**：專案永遠不會遺失
   - 電腦壞掉？GitHub 上還有完整備份
   - 換電腦？下載回來繼續做
   - 想在公司和家裡接續工作？同步無痛

**二、Git vs GitHub：工具與平台的關係**

很多初學者會搞混這兩個名詞，我們先釐清：

（一）Git：版本控制工具

Git 是一個**軟體**，安裝在你的電腦上：
- 負責追蹤檔案的變化
- 記錄每一次修改的歷史
- 管理不同的版本分支

**類比**：Git 就像你的「存檔系統」，每次存檔都會記錄完整的歷史。

（二）GitHub：雲端程式碼託管平台

GitHub 是一個**網站**，用來存放你的專案：
- 提供雲端空間存放程式碼
- 讓多人協作更方便
- 提供社群分享功能

**類比**：GitHub 就像「Google Drive」，只是它專門為程式碼設計，有更強大的版本管理功能。

（三）兩者的合作關係

```
你的電腦（Git 管理版本）
    ↓ push（上傳）
GitHub（雲端存放）
    ↓ 連接
Vercel（自動部署成網站）
```

### 3-2 GitHub 實戰：從零到建立第一個專案

**三、註冊 GitHub 帳號**

（一）訪問 GitHub 官網

1. 打開瀏覽器，前往 `https://github.com`
2. 點擊右上角的「Sign up」按鈕

（二）填寫註冊資訊

1. **Email**：建議使用常用的電子信箱
2. **Username**：這會成為你的公開帳號名稱
   - 建議：簡潔、專業、好記
   - 範例：`johnsmith`、`designerchen`、`tangcruzz`
   - 避免：複雜符號、過長名稱
3. **Password**：設定安全的密碼

（三）完成驗證

GitHub 會要求你：
1. 驗證你是真人（通過簡單的圖片驗證）
2. 到信箱收驗證信，點擊連結啟用帳號

**四、建立你的第一個 Repository（儲存庫）**

（一）什麼是 Repository？

Repository（簡稱 repo）就是一個「專案資料夾」：
- 存放你的網站所有檔案
- 記錄所有修改歷史
- 可以設定為公開或私人

（二）建立新的 Repository

登入 GitHub 後：

1. 點擊右上角的「+」圖示
2. 選擇「New repository」
3. 填寫專案資訊：
   - **Repository name**：專案名稱（如 `photographer-portfolio`）
   - **Description**：簡短描述（如 "攝影師作品集網站"）
   - **Public / Private**：選擇 Public（公開）
   - **Initialize with README**：可以勾選（會自動建立說明檔案）
4. 點擊「Create repository」

**恭喜！你的第一個 GitHub 專案誕生了！**

（三）Repository 的基本結構

建立完成後，你會看到：

```
photographer-portfolio/
├── README.md         # 專案說明檔案
└── .gitignore        # 忽略不需要上傳的檔案（選擇性）
```

現在這個專案還是空的，我們要把網站檔案上傳上去。

**五、上傳網站檔案到 GitHub**

（一）透過網頁介面上傳（最簡單）

如果你的網站檔案還不多（10 個檔案以內），可以直接用網頁上傳：

1. 在 Repository 頁面，點擊「Add file」→「Upload files」
2. 把你的網站檔案（HTML、CSS、圖片等）拖曳進來
3. 在下方「Commit changes」區塊：
   - 填寫簡短說明（如 "初次上傳網站檔案"）
   - 點擊「Commit changes」

（二）基本的檔案結構建議

一個攝影師作品集網站通常包含：

```
photographer-portfolio/
├── index.html              # 首頁
├── gallery.html            # 作品集頁面
├── about.html              # 關於我頁面
├── contact.html            # 聯絡頁面
├── css/
│   └── style.css          # 樣式檔案
├── js/
│   └── script.js          # JavaScript 檔案
└── images/
    ├── photo1.jpg         # 作品照片
    ├── photo2.jpg
    └── ...
```

（三）Commit 訊息的藝術

每次上傳（commit）都應該寫清楚的說明：

**好的 commit 訊息**：
- "新增作品集畫廊功能"
- "修正手機版選單顯示問題"
- "更新攝影師個人簡介"

**不好的 commit 訊息**：
- "修改"
- "更新"
- "aaa"（完全看不懂）

清楚的 commit 訊息讓你未來回顧時，能快速理解每次修改的內容。

### 3-3 Vercel 部署：一鍵上線的魔法

**六、認識 Vercel：現代化的部署平台**

（一）為什麼選擇 Vercel？

市面上有很多網站託管服務，為什麼我們選擇 Vercel？

1. **完全免費**（個人專案）
   - 無限次部署
   - 免費的 HTTPS 加密
   - 全球 CDN 加速

2. **極度簡單**
   - 連接 GitHub，自動部署
   - 推送程式碼，自動更新網站
   - 不需要任何伺服器設定

3. **效能優異**
   - 全球 CDN 節點
   - 自動優化圖片
   - 網站載入速度極快

4. **專業功能**
   - 自動獲得 SSL 憑證（HTTPS）
   - 每次部署都有獨立網址（可預覽）
   - 支援自訂網域

（二）Vercel 的運作原理

```
你修改程式碼 → Git commit → Push 到 GitHub
                                    ↓
                    Vercel 偵測到更新，自動重新部署
                                    ↓
                            新版網站上線！
```

這個流程叫做「**持續部署（Continuous Deployment）**」，是現代網站開發的標準做法。

**七、Vercel 部署實戰流程**

（一）註冊 Vercel 帳號

1. 前往 `https://vercel.com`
2. 點擊「Sign Up」
3. **重要**：選擇「Continue with GitHub」
   - 這會自動連接你的 GitHub 帳號
   - 讓部署流程更順暢

4. 授權 Vercel 存取你的 GitHub（點擊「Authorize」）

（二）匯入 GitHub 專案

登入 Vercel 後：

1. 點擊「New Project」
2. 你會看到你的 GitHub repositories 列表
3. 找到 `photographer-portfolio`，點擊「Import」
4. 配置專案設定：
   - **Project Name**：保持預設或自訂（如 `my-portfolio`）
   - **Framework Preset**：選擇「Other」（因為我們是純 HTML）
   - **Root Directory**：保持預設（`./`）
   - **Build Settings**：不用填（純 HTML 不需要編譯）

5. 點擊「Deploy」

（三）部署中⋯⋯

Vercel 會開始部署你的網站：
- 你會看到即時的部署日誌
- 通常只需要 **30 秒到 1 分鐘**
- 畫面會顯示進度條和狀態

（四）部署成功！

當你看到「Congratulations!」和煙火動畫，代表網站已經上線了！

Vercel 會自動給你一個網址，格式是：
```
https://your-project-name.vercel.app
```

**點擊這個網址，你就能看到你的網站了！**

**八、自動部署的威力**

最神奇的部分來了：

1. 你在本地修改網站檔案
2. Commit 並 push 到 GitHub
3. Vercel **自動偵測到更新**
4. **自動重新部署**
5. 網站自動更新！

整個過程不需要你做任何額外操作。這就是現代 web 開發的「持續部署」魔法。

### 3-4 自訂網域：建立專業品牌形象

**九、為什麼需要自訂網域？**

（一）vercel.app vs 自訂網域

**Vercel 預設網址**：`https://my-portfolio.vercel.app`
- ✅ 免費、快速
- ❌ 不夠專業
- ❌ 難以記憶
- ❌ 品牌識別度低

**自訂網域**：`https://johnphotography.com`
- ✅ 專業形象
- ✅ 容易記憶
- ✅ 建立品牌
- ❌ 需要購買（通常 300-500 元/年）

（二）何時需要自訂網域？

**暫時不需要**：
- 練習作品、學習專案
- 還在測試階段
- 預算有限

**建議購買**：
- 接案用的作品集
- 正式的商業網站
- 想建立個人品牌

**十、購買與設定自訂網域**

（一）購買網域名稱

常見的網域註冊商：
1. **Namecheap**（推薦，便宜且介面友善）
2. **GoDaddy**（知名度高）
3. **Google Domains**（Google 旗下，可惜已停止新註冊）

購買步驟（以 Namecheap 為例）：

1. 前往 `https://www.namecheap.com`
2. 搜尋你想要的網域名稱（如 `johnphotography.com`）
3. 檢查是否可用：
   - 如果可用，加入購物車
   - 如果不可用，嘗試其他名稱或後綴（.net、.co、.tw）
4. 結帳購買（通常第一年有優惠）

**網域命名建議**：
- 簡短好記
- 避免數字和連字號
- 與你的品牌相關
- `.com` 最常見，`.tw` 適合台灣在地品牌

（二）在 Vercel 設定自訂網域

購買網域後，回到 Vercel：

1. 進入你的專案
2. 點擊「Settings」→「Domains」
3. 輸入你購買的網域（如 `johnphotography.com`）
4. 點擊「Add」

Vercel 會顯示需要設定的 DNS 記錄。

（三）設定 DNS（域名系統）

回到 Namecheap（或你的網域註冊商）：

1. 進入「Domain List」→ 選擇你的網域 → 點擊「Manage」
2. 找到「Advanced DNS」頁面
3. 新增 Vercel 提供的記錄：
   - **A Record**: 指向 Vercel 的 IP 位址
   - **CNAME Record**: 將 www 指向你的 Vercel 網址

4. 儲存設定

（四）等待 DNS 生效

DNS 設定需要時間生效：
- 通常 **10 分鐘到 48 小時**
- 大部分情況下 **1-2 小時**就會生效
- 可以用 `https://dnschecker.org` 檢查是否生效

生效後，你就可以用自訂網域訪問你的網站了！

### 3-5 響應式設計測試：確保完美體驗

**十一、為什麼 RWD 測試如此重要？**

（一）多裝置瀏覽的現實

2024 年的網路流量統計：
- 📱 手機：約 **60%**
- 💻 電腦：約 **35%**
- 📱 平板：約 **5%**

如果你的網站只在電腦上看起來正常，你就流失了超過一半的使用者。

（二）常見的 RWD 問題

**問題一：文字太小**
- 電腦上看起來剛好的字體，在手機上可能小到看不清楚

**問題二：按鈕太小**
- 電腦可以精準點擊，手機上手指容易誤觸

**問題三：圖片跑版**
- 電腦上完美對齊的圖片，在手機上可能被截掉或變形

**問題四：橫向滾動**
- 手機出現左右滾動條，使用體驗很差

**十二、多裝置測試的完整流程**

（一）瀏覽器內建測試工具

不需要真的拿出手機，就能測試手機版效果：

**Chrome DevTools 操作步驟**：

1. 在你的網站頁面按 **F12**（或右鍵 → 檢查）
2. 點擊左上角的「手機/平板」圖示（Toggle device toolbar）
3. 選擇不同裝置：
   - iPhone SE（小螢幕手機）
   - iPhone 14 Pro（標準手機）
   - iPad Air（平板）
   - Samsung Galaxy（Android 手機）

4. 檢查每個裝置的顯示效果

（二）關鍵測試項目

**✅ 導航選單**
- 手機版是否變成漢堡選單（☰）？
- 點擊後能否正常展開？
- 選單項目是否夠大，容易點擊？

**✅ 圖片顯示**
- 圖片是否隨螢幕寬度自動縮放？
- 圖片比例是否正確（沒有變形）？
- 照片畫廊在手機上是否變成單欄顯示？

**✅ 文字大小**
- 標題在手機上是否還看得清楚？
- 內文字體大小是否至少 16px？
- 段落間距是否足夠？

**✅ 按鈕與連結**
- 按鈕大小是否至少 44x44 像素（手指好按）？
- 連結之間間距夠大嗎（避免誤觸）？
- Call-to-action 按鈕是否明顯？

**✅ 表單輸入**
- 輸入框在手機上是否夠大？
- 鍵盤彈出時，輸入框是否被遮住？
- 送出按鈕是否容易點擊？

（三）真實裝置測試

開發者工具很方便，但還是建議用真實裝置測試：

1. **用你的手機瀏覽**
   - 在手機瀏覽器輸入網址
   - 實際滑動、點擊，感受真實體驗

2. **請朋友幫忙測試**
   - 不同品牌、型號的手機可能有差異
   - Android 和 iOS 的瀏覽器也有細微不同

3. **注意網路速度**
   - 在手機網路（4G/5G）下測試載入速度
   - 圖片是否太大，載入太慢？

**十三、常見問題修正**

（一）圖片跑版解決方案

如果圖片在手機上跑版，在 CSS 加入：

```css
img {
  max-width: 100%;
  height: auto;
}
```

（二）文字太小解決方案

確保基本字體大小至少 16px：

```css
body {
  font-size: 16px;
}
```

（三）導航選單優化

手機版可以使用「漢堡選單」，在與 AI 溝通時可以說：

> "手機版的導航選單請改成漢堡選單（☰），點擊後從右側滑出。"

### 3-6 攝影師作品集的特殊設計考量

**十四、作品展示的最佳實踐**

（一）畫廊版面選擇

攝影師作品集的核心是照片展示，常見版面：

1. **瀑布流（Masonry Layout）**
   - 不同高度的圖片，像瀑布一樣排列
   - 優點：視覺動感、充分利用空間
   - 適合：風格多樣的作品

2. **網格式（Grid Layout）**
   - 整齊的方形圖片排列
   - 優點：簡潔、專業
   - 適合：想強調秩序感的作品集

3. **全螢幕輪播（Fullscreen Slider）**
   - 一次只展示一張照片
   - 優點：聚焦、震撼力強
   - 適合：想突出單張作品的攝影師

（二）圖片最佳化

攝影師的照片通常檔案很大（5MB 以上），必須最佳化：

1. **壓縮圖片**
   - 使用線上工具如 TinyPNG
   - 品質設定在 80-85% （肉眼幾乎看不出差異）
   - 檔案大小降低到 200-500KB

2. **使用適當的尺寸**
   - 網頁顯示不需要原始的 6000x4000 解析度
   - 一般用 1920x1280 就很足夠
   - 縮圖可以用 600x400

3. **延遲載入（Lazy Loading）**
   - 只載入畫面上看得到的圖片
   - 使用者滾動時才載入後續圖片
   - 可以請 AI 實現這個功能

（三）分類與篩選

如果作品很多，建議加入分類：

- 婚紗攝影
- 商業攝影
- 人像攝影
- 風景攝影

使用者可以點擊分類，只看該類別的作品。

**十五、專業攝影網站的必備元素**

（一）首頁設計

1. **英雄橫幅（Hero Banner）**
   - 一張最震撼的代表作品
   - 大大的攝影師名字
   - 簡短的標語（如 "捕捉每個動人瞬間"）

2. **作品精選**
   - 展示 6-9 張最佳作品
   - 吸引訪客進入完整作品集

3. **服務項目**
   - 婚紗攝影、商業攝影、活動攝影等
   - 每項服務搭配一張代表照片

（二）關於頁面

攝影是很個人化的服務，客戶想知道：
- 攝影師的背景經歷
- 拍攝風格與理念
- 使用的器材（選擇性）
- 得獎記錄或客戶評價

## 重要名詞

◆ **版本控制（Version Control）**：記錄檔案修改歷史的系統，可以追蹤誰在何時修改了什麼內容，並能回到過去任何一個版本。

◆ **Git**：最流行的分散式版本控制系統，由 Linux 之父 Linus Torvalds 創建。

◆ **Repository（儲存庫）**：存放專案所有檔案和版本歷史的容器，簡稱 repo。

◆ **Commit（提交）**：將修改記錄到版本歷史的動作，每次 commit 都會產生一個獨特的版本編號。

◆ **Deploy（部署）**：將網站程式碼從開發環境推送到生產環境，讓使用者可以訪問的過程。

◆ **CDN（Content Delivery Network）**：內容傳遞網路，將網站內容分散到全球各地的伺服器，讓使用者從最近的節點存取，提升載入速度。

◆ **DNS（Domain Name System）**：網域名稱系統，將人類可讀的網址（如 example.com）轉換成電腦可讀的 IP 位址。

◆ **Lazy Loading（延遲載入）**：網頁優化技術，只載入使用者看得到的內容，節省頻寬並加快載入速度。
